# CS

## 입출력장치

1. 까다로운 두가지 이유

- 장치의 종류가 많다.
- 전송률의 차이가 너무 크다. CPU <-> 키보드 마우스차이

## 장치 컨트롤러 : 하드웨어적 중개자

입출력장치는 위 두가지 이유로 컴퓨터에 직접 연결되지 않고 장치 컨트롤러라는 하드웨어를 통해 연결

- 입출력 제어기(I/O Controller), 입출력 모듈(I/O Module)등으로 불림

### 주요 역할

- CPU와 입출력장치 간의 통신 중계 : 규격화가 어려운 장치들 사이에서 일종의 번역가 역할 수행
- 오류 검출 : 연결된 장치에 문제가 없는지
- 데이터 버퍼링 : 전송률이 높은 장치와 낮은 장치 사이의 간극을 메우기 위해서 버퍼라는 임시 저장 공간에 데이터를 모았다가 한꺼번에 내보내거나, 한 번에 많이 받아 조금씩 내보내는 방법, 이를 통해 전송률 차이를 완화시킨다.

### 내부 구조

- 데이터 레지스터 : CPU와 입출력장치 사이에 주고받을 실제 데이터가 담기는 곳, 버퍼 역할을 수행하며 최근 전송량이 많은 장치는 레지스터 대신 RAM을 이용하기도 함
- 상태 레지스터 : 장치가 작업 준비가 되었는지 혹은 오류가 없는지에 대한 상태 정보를 저장
- 제어 레지스터 : 장치가 수행할 내용에 대한 제어 정보와 명령을 저장

## 장치 드라이버

새로운 장치를 컴퓨터에 연결하여 사용하기 위해서는 반드시 장치 드라이버를 설치해야 함

### 장치 드라이버의 정의와 역할

장치 컨트롤러의 동작을 감지하고 제어함으로써 컴퓨터 내부와 정보를 주고받을 수 있게 하는 프로그램
메모리 저장 : 프로그램이기 때문에 실행 과정에서 당연히 메모리(RAM)에 저장
통로의 구분 : 하드웨어적 통로 -> 장치 컨트롤러, 소프트웨어적 통로 -> 장치 드라이버

## 프로그램 입출력

프로그램속 명령어로 입출력 장치를 제어하는 방식, CPU가 명령어를 실행하다가 입출력 명령어를 만나면 장치 컨트롤러와 상호작용하며 작업을 수행

### 작업 과정 상세

'메모리 정보를 하드디스크에 백업한다'는 것은 '하드디스크에 새로운 정보를 쓴다'는 의미와 같다.

1. 쓰기 명령 : CPU는 하드디스크 컨트롤러의 제어 레지스터에 쓰기 명령을 보냄
2. 상태 확인 : 하드디스크 컨트롤러는 하드디스크의 상태를 확인, 준비가 되었다면 상태 레지스터에 준비 완료를 표시
3. 읽기 및 쓰기 : CPU는 상태 레지스터를 주기적으로 읽어보며 준비 여부를 확인, 준비가 됐음을 알게되면 백업할 정보를 데이터 레지스터에 쓴다. 작업이 끝날때까지 1번부터 반복 완료되면 종료

### 장치 컨트롤러의 주소를 아는 방식

1. 매모리 맵 입출력 (Memory-mapped I/O)

메모리에 접근하기 위한 주소 공간과 입출력 장치에 접근하기 위한 주소 공간을 하나로 간주하는 방법

예시) 1024개의 주소가 있을때 메모리주소로 512, 나머지 512는 장치 컨트롤러 레지스터 주소로 사용하는 방식
특징) CPU는 메모리 주소나 레지스터 주소나 똑같이 메모리 주소를 대하듯 하면 됨, 따라서 메모리 접근 명령어와 입출력 접근 명령어가 굳이 다를필요가 없다.

2. 고립형 입출력 (Isolated I/O)

메모리를 위한 주소 공간과 입출력 장치를 위한 주소 공간을 분리하는 방법

- 구조 : 제어 버스에 메모리 읽기/쓰기 선 외에 입출력장치 읽기/쓰기 선이 따로 존재
- 특징 : CPU는 입출력 장치에 접근하기 위해 메모리 접근 명령어와는 다른 별도의 입출력 전용 명령어를 사용

메모리 맵 방식은 명령어가 동일하지만 주소 공간 일부를 포기해야 하고, 고립형 방식은 주소 공간을 온전히 활용하지만 전용 명령어가 따로 필요하다는 차이가 있다.

## 인터럽트 기반 입출력 (Interrupt-Based I/O)

인터럽트 기반 입출력은 CPU가 장치 컨트롤러에 입출력 작업을 명령한 뒤, 장치가 일을 수행하는 동안 CPU는 다른 업무를 처리하다가 작업이 완료되었을때 신호를 받는 방식

### 하드웨어 인터럽트의 실제 주체

인터럽트는 입출력 장치 자체가 아닌 장치 컨트롤러가 발생시킴

- CPU는 장치 컨트롤러에 작업을 명령
- 컨트롤러가 입출력을 수행하는 동안 CPU는 다른일
- 작업이 끝나면 컨트롤러가 CPU에 인터럽트 요청 신호를 보냄
- CPU는 하던일을 잠시 백업하고 인터럽트 서비스 루틴(ISR)을 실행한 뒤 다시 돌아옴

### 인터럽트 vs 폴링(Polling)

- 폴링 : CPU가 주기적으로 장치 컨트롤러의 상태 레지스터를 확인하며 장치의 상태를 체크하는 방식, CPU의 부담이 매우 큼
- 인터럽트 : 신호를 받을 때까지 CPU가 온전히 자기 일에 집중할 수 있어 폴링보다 훨씬 효율적

### 다중 인터럽트 처리 방법

여러 장치에서 인터럽트가 동시 발생하면 처리방식은 다음과 같다.

1. 순차 처리 : 발생한 순서대로 처리, 현재 처리중인 동안에는 다른 인터럽트를 비활성화함
2. 우선순위 처리 : 더 급한 일(우선순위가 높은 인터럽트)을 먼저 처리, 현재 작업보다 순위가 높으면 잠시 멈추고 높은 순위부터 해결, NMI : 우선순위가 가장 높아 무시할 수 없는 가장 중요한 인터럽트

### PIC(프로그래머블 인터럽트 컨트롤러)

여러 장치 컨트롤러를 관리하며 우선순위를 판별해주는 하드웨어

1. PIC가 장치들로부터 인터럽트 요청 신호를 받음
2. 우선순위를 판단해 CPU에 처리해야할 신호를 보냄
3. CPU가 PIC에 확인신호를 보냄
4. PIC는 데이터 버스를 통해 CPU에 인터럽트 벡터를 보냄
5. CPU는 벡터를 통해 주체를 파악하고 해당 서비스 루틴을 실행

## DMA 입출력 (Direct Memory Access I/O)

앞선 방식들은 데이터 이동 시 모두 CPU를 거쳐야한다는 공통점, 이는 대용량 데이터 전송시 CPU에 부담, 이를 해결하기 위한 방식이 DMA

### DMA 정의와 과정

DMA는 이름 그대로 CPU를 거치지 않고 직접 메모리에 접근하여 데이터를 주고받는 방식, 시스템 버스에 연결된 DMA 컨트롤러라는 하드웨어가 필요

1. CPU가 DMA 컨트롤러에 작업 정보(주소, 연산종류, 메모리 위치 등)를 명령
2. DMA 컨트롤러가 CPU대신 장치 컨트롤러와 상호작용하며 메모리에 직접 데이터를 읽고 씀
3. 작업이 끝나면 DMA 컨트롤러가 CPU에 인터럽트를 걸어 종료를 알림

CPU의 관여도
CPU는 오로지 입출력의 시작과 끝에만 관여하면 되기 때문에 작업 부담을 훨씬 줄일 수 있음

### 사이클 스틸링

시스템 버스는 공용 자원이므로 CPU와 DMA컨트롤러가 동시에 쓸 수 없음

- DMA 컨트롤러는 CPU가 버스를 이용하지 않을 때마다 조금씩 이용하거나
- CPU가 일시적으로 버스를 쓰지 않도록 허락을 구하고 집중적으로 이용
- 이를 CPU의 사이클을 뺏는다는 의미에서 사이클 스틸링이라고 함

### 입출력 버스 (I/O Bus)

DMA를 위해 시스템 버스를 너무 자주 사용하면 CPU의 업무가 방해받는다. 이를 해결하기 위해 입출력 버스라는 별도의 통로를 사용

- 구조 : 장치 컨트롤러들을 시스템 버스가 아닌 입출력 버스로 DMA 컨트롤러에 연결
- 장점 : 장치 간 데이터 전송 시 시스템 버스를 이용할 필요가 없으므로 사용 빈도를 줄일 수 있음
- 종류 : PCI 버스, PCI Express(PCIe) 버스 등이 있으며, 현대 컴퓨터의 대부분 입출력 장치는 이 입출력 버스를 통해 CPU와 정보를 주고 받음

## 🎮 팀 프로젝트 — 언로드 설계 \& Primary Asset

### 언로드 설계 방향

메모리를 두 곳에서 동시에 붙잡고 있어 GC가 동작하지 않는 구조가 문제였다.

- **`PhaseHandles`**: 비동기 핸들을 `TSharedPtr`로 잡고 있어 언로드 불가
- **`DataCache`**: `TObjectPtr<UObject>`로 잡고 있어 GC 차단

**해결 방향**

1. DataCache 캐시 초기화
2. PhaseHandles 핸들 해제
3. AssetManager의 `UnloadPrimaryAsset` / `UnloadPrimaryAssets` 함수로 언로드 요청[^1][^2]
4. 게임 종료 → 로비 복귀 시, 로비 관련 데이터를 다시 로드하는 흐름으로 설계

### Primary Asset ID 구조 이해

`FPrimaryAssetId`는 `AssetType + Name` 조합으로 유일성을 보장한다. `AssetType`이 같아도 파일명(Name)이 다르면 서로 다른 에셋으로 취급되므로, 같은 타입의 에셋을 여러 개 등록해도 충돌하지 않는다.[^3]

## 🌳 알고리즘 — BOJ 1539 이진 검색 트리

### 핵심 개념 복기

BST는 왼쪽 자식 < 나 < 오른쪽 자식 규칙으로 구성된다. **중위 순회(In-order Traversal)** 시 항상 오름차순으로 출력된다.

### 문제 핵심 인사이트

**"BST 시뮬레이션 탈을 쓴 lower_bound 문제"**[^4][^5]

새 값 X의 부모 후보는 딱 두 노드다.

- X보다 큰 값 중 가장 작은 것 (`set::lower_bound`)
- X보다 작은 값 중 가장 큰 것 (`prev(it)`)

이 둘 중 **깊이(depth)가 더 깊은 노드** 의 자식으로 X가 붙는다.

### 왜 직접 BST 구현은 시간초과인가?

N이 최대 25만일 때 편향 트리(skewed tree)가 만들어지면 삽입 탐색이 O(N)이 되어 전체 O(N²)으로 터진다. `std::set`의 `lower_bound`는 내부적으로 Red-Black Tree 기반이라 O(log N)을 보장하므로 이 방법으로 교체해야 통과한다.

```cpp
auto it = s.lower_bound(x);   // x 이상인 값 중 최솟값
// *it     -> x보다 큰 값 중 최솟값 (오른쪽 후보 부모)
// *prev(it) -> x보다 작은 값 중 최댓값 (왼쪽 후보 부모)
// 둘 중 depth가 더 깊은 쪽의 자식으로 x를 붙임
```

| 계층 | 이름          | 종류       | 역할                                         |

| 1    | 장치 컨트롤러 | 하드웨어   | CPU ↔ 장치 간 규격 번역, 버퍼링, 오류 검출   |

| 3    | 입출력 방식   | 설계 패턴  | 아래 세 방식으로 진화                        |

### 입출력 방식의 진화 흐름

**① 프로그램 입출력 (Programmed I/O)**

CPU가 직접 상태 레지스터를 반복 확인(폴링)하는 방식. CPU가 쉬지 못하고 계속 체크해야 하므로 낭비가 심하다.

**② 인터럽트 기반 입출력 (Interrupt-driven I/O)**

장치 컨트롤러가 작업 완료 시 CPU에 인터럽트 신호를 발생시킨다. CPU는 그동안 다른 일을 처리하므로 폴링보다 훨씬 효율적이다. 다중 인터럽트는 **PIC(Programmable Interrupt Controller)** 가 우선순위를 판별해 순서를 조정해준다.

**③ DMA 입출력 (Direct Memory Access I/O)**

대용량 전송 시에도 CPU가 데이터 이동에 관여하면 부담이 크다. DMA 컨트롤러가 CPU를 대신해 메모리 ↔ 장치 간 데이터를 직접 전송하고, 완료 시에만 인터럽트를 보낸다. CPU는 시작과 끝에만 관여한다.

### 사이클 스틸링 \& 입출력 버스

DMA가 시스템 버스를 과도하게 점유하면 CPU 업무가 방해받는다. 이를 해결하는 두 가지 방법:[^6][^7]

- **사이클 스틸링**: CPU가 버스를 안 쓰는 유휴 사이클을 틈타 조금씩 전송[^6]
- **입출력 버스(PCI / PCIe)**: 장치 컨트롤러를 시스템 버스가 아닌 별도 입출력 버스로 DMA 컨트롤러에 연결, 시스템 버스 사용 빈도를 대폭 감소

### 오늘의 핵심 관점

> 컴퓨터 입출력의 역사 = **"바쁜 CPU의 시간을 어떻게 덜 빼앗을 것인가"** 의 역사
> 폴링 → 인터럽트 → DMA 순으로, CPU의 개입 지점이 점점 줄어드는 방향으로 발전했다.
