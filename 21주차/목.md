# 팀프로젝트

## AEDMonsterSpawner

```c++
UEDMonsterDataAsset* AEDMonsterSpawner::GetMonsterDataAsset() const
{
	UEDGameDataSubsystem* DataSubsystem = UEDGameDataSubsystem::Get(this);
	if (!IsValid(DataSubsystem))
		return nullptr;

	return DataSubsystem->GetData<UEDMonsterDataAsset>(MonsterDataAssetId);
}
```

## EDMonsterAIController

기존 방식 : FStreamableHandle들고있음
수정 방식 : 비동기 핸들을 EDGameDataSubsystem에서 다 들고 있으므로 굳이 따로 핸들 쓸필요가 없다.

캐싱한 데이터 사용하는 이거 이거를 SpawnSubsystem같은곳에서도 불러서 사용하면 된다.
기존에 에셋전체를 들고있는거에서 경로 드는걸로 바꾸고 GetMonsterDataAsset이 함수를 통해서 GameDataSubsystem같은곳에서도 호출해서 사용하면된다.

## 정리

- **Pre-loading**: `EDGameDataSubsystem`이 게임 시작 시 모든 몬스터 데이터를 미리 메모리에 로드 및 캐싱.
- **Zero-Wait Initialization**: 몬스터 액터는 이미 메모리에 올라온 에셋 주소를 `.Get()`으로 즉시 참조하여 동기적으로 설정.
- 여기서 .Get으로 가져오는거는 동기로드가 아니고 GameDataSubsystem에서 기존에 로딩화면에서 로드했던 모든 정보들을 TObjectPtr로 들고있고 (DataCache) 이것에 대해서 .Get()을 하면 서브시스템에서 이미 에셋을 비동기 로드해서 경로를 들고 있으므로 그 주소를 가리키고 있기 때문에 바로 가져와서 실행가능

## 해야할거

1. 에셋 번들(Asset Bundle)을 활용한 일괄 로딩

- 소프트 레퍼런스의 유연함과 하드 레퍼런스의 즉각성을 동시에 챙기기 위해 Asset Bundle 기능 추가해야함
- 작동 원리: DataAsset 내부 변수에 AssetBundles 태그를 달고, Subsystem에서 해당 태그를 포함하여 로드 요청.
- 결과: 로딩 창에서 데이터 에셋뿐만 아니라 그 안에 담긴 메시, 애니메이션까지 한꺼번에 메모리에 안착시켜 인게임 스폰 시 Hitch(끊김) 발생을 원천 차단함.

2. 로드 전략

- 전투 리소스 (몬스터/캐릭터): 로딩 창에서 Asset Bundle을 통해 비동기 로드 후 DataCache에 상주시켜 즉각 대응함.
- UI/아이콘 리소스: 메모리 절약을 위해 Async Load 방식을 사용하여 인벤토리를 여는 시점 등 필요한 때만 로드함.

# 챌린지반

## 네트워크

### TCP

신뢰성 보장
로그인, 채팅, 상점처럼 신뢰도가 중요할때
연결할때 핸드쉐이크 필요
패킷 손실되면 뒤 패킷들도 블로킹된다.

### UDP

비연결형 신뢰성같은거 없다.
위치, 전투, 음성같은 조금 덜 중요한 정보? 에대해서
패킷 손실되어도 그냥 계속 진행

### 보간

장점 : 실제 데이터 기반이므로 부드럽고 정확, 그냥 구현하기 편함
단점 : 항상 과거를 보여줌

### 외삽

마지막으로 받은 데이터를 기준으로 예측하는거
정확도가 보간보다 낮음
방향 전환시 튄다.

### 최적화 기법

양자화 : 정밀도 줄여서 자료형을 축소, 정확하지 않음
델타 압축 : 변경된 필드만 전송, 어떤 필드가 변경되었는지 비트마스크로 추적
빈도 조절 : 모든 데이터가 매 프레임 동기화 될 필요는 없다.
