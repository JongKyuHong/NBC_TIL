# 인벤토리 최적화

## 작업 배경

롤처럼 화면에 표시되는 인벤토리 퀵바 위젯(`EDInventoryQuickBarWidget`)에서  
에셋을 불러오는 방식이 동기 블로킹(`TryLoad()`)이었고,  
같은 에셋을 슬롯마다 중복으로 조회하는 비효율적인 구조였음.

---

## 문제점

### 1. 동기 블로킹 로드 (`TryLoad()`)

```cpp
// ❌ 기존 코드 - 게임 스레드를 블로킹
UObject* ItemObject = UAssetManager::Get().GetPrimaryAssetObject(ItemId);
if (!ItemObject)
{
    const FSoftObjectPath AssetPath = UAssetManager::Get().GetPrimaryAssetPath(ItemId);
    if (AssetPath.IsValid())
        ItemObject = AssetPath.TryLoad(); // 게임 스레드 블로킹!
}
```

- `TryLoad()`는 I/O가 끝날 때까지 게임 스레드를 멈춤
- 퀵바 슬롯이 N개면 리프레시 1회에 최대 **2N번** 동기 로드 발생

### 2. 같은 에셋 중복 조회

```cpp
// ❌ 기존 - 이름 조회, 등급 조회가 각각 분리되어 같은 에셋을 2번 로드
FText Name = ResolveItemDisplayName(SlotData.Item.ItemId);   // 내부에서 TryLoad
EEDItemRarity Rarity = ResolveItemRarity(SlotData.Item.ItemId); // 또 TryLoad
```

### 3. 인벤토리 변경 시 즉시 전체 갱신

```cpp
// ❌ 기존 - 변경이 생길 때마다 즉시 전체 갱신
void HandleInventoryChanged()
{
    RefreshQuickSlots();     // 매번 실행
    RefreshEquipmentSlots(); // 매번 실행
}
```

- 빠르게 아이템을 여러 개 옮기면 프레임마다 전체 리프레시 폭발

## 해결 방법

### 1. `EDGameDataSubsystem::DataCache` 조회로 교체

```cpp
// ✅ 변경 - DataSubsystem 캐시 우선 조회 (이미 비동기로 올라온 데이터)
const UEDInventoryItemDataAsset* UEDInventoryQuickBarWidget::ResolveItemData(
    const FPrimaryAssetId& ItemId) const
{
    if (!ItemId.IsValid()) return nullptr;

    // 1순위: DataSubsystem 캐시 (TryLoad 없음)
    if (const UEDGameDataSubsystem* DS = UEDGameDataSubsystem::Get(this))
    {
        if (const UEDInventoryItemDataAsset* Cached =
                DS->GetData<UEDInventoryItemDataAsset>(ItemId))
            return Cached;
    }

    // 2순위: AssetManager 인메모리 확인 (TryLoad 없음)
    return UEDAssetManager::Get().GetPrimaryAsset<UEDInventoryItemDataAsset>(ItemId);
}
```

- 기존 `ResolveItemDisplayName` + `ResolveItemRarity` 2개 함수를 `ResolveItemData` 1개로 통합
- 슬롯당 에셋 조회 **2회 → 1회**, 동기 블로킹 **0회**

### 2. Dirty Flag + 프레임 지연 Refresh

```cpp
// ✅ 변경 - 같은 프레임에 몰린 변경을 다음 틱에 한 번만 처리
void UEDInventoryQuickBarWidget::HandleInventoryChanged()
{
    if (bRefreshPending) return; // 이미 예약됨 → 중복 방지
    bRefreshPending = true;

    GetWorld()->GetTimerManager().SetTimerForNextTick(
        FTimerDelegate::CreateUObject(this, &UEDInventoryQuickBarWidget::DoRefresh)
    );
}

void UEDInventoryQuickBarWidget::DoRefresh()
{
    bRefreshPending = false;
    RefreshQuickSlots();
    RefreshEquipmentSlots();
}
```

- 실제 갱신을 딱 한번만 진행하게끔
- 이런 구조를 DirtyFlag라고 한다고 함 (약간 웹에서 pending생각남)

### 3. 비동기 프리로드 + DataSubsystem 완료 대기

```cpp
// ✅ 변경 - NativeConstruct에서 DataSubsystem 상태에 따라 분기
void UEDInventoryQuickBarWidget::NativeConstruct()
{
    // ...
    if (UEDGameDataSubsystem* DS = UEDGameDataSubsystem::Get(this))
    {
        if (DS->IsDataReady())
        {
            // 이미 완료 → 다음 틱에 바로 실행
            GetWorld()->GetTimerManager().SetTimerForNextTick(
                FTimerDelegate::CreateUObject(this, &UEDInventoryQuickBarWidget::PreloadInventoryAssets)
            );
        }
        else
        {
            // 아직 로드 중 → 완료 콜백에서 실행
            DS->OnAllDataLoaded.AddUniqueDynamic(
                this, &UEDInventoryQuickBarWidget::PreloadInventoryAssets);
        }
    }
}
```

## 트러블슈팅 과정

### 문제 1: `ResolveItemData` 로그가 아예 안 찍힘

- **원인**: `NativeConstruct`에서 `PreloadInventoryAssets()` 호출 누락
- **해결**: `SetTimerForNextTick`으로 다음 틱에 호출 예약

### 문제 2: Cache MISS만 발생

- **원인**: `EDGameDataSubsystem`이 `GameInstanceSubsystem`이라 서버/클라이언트 인스턴스가 독립적으로 동작함.  
  `InitializeGameData()`가 서버의 `TryStartGame()`에서만 호출되어 **클라이언트 DataSubsystem은 LobbyReady 상태로 멈춰있었음**
- **해결**: 게임 맵 GameState의 `BeginPlay`에서 `IsDataReady()` 확인 후 미완료 시 `InitializeGameData()` 호출

```cpp
void AEDGameState::BeginPlay()
{
    Super::BeginPlay();
    if (UEDGameDataSubsystem* DS = UEDGameDataSubsystem::Get(this))
    {
        if (!DS->IsDataReady())
            DS->InitializeGameData(); // 클라이언트 측 DataSubsystem 로드 시작
    }
}
```

> `GameState::BeginPlay`는 서버/클라이언트 양쪽에서 호출되므로,  
> 서버는 `IsDataReady()=true`라 스킵되고 클라이언트만 로드를 시작함.

### 문제 3: `AddUObject`로 Dynamic 델리게이트 바인딩 실패

- **원인**: `DECLARE_DYNAMIC_MULTICAST_DELEGATE`는 리플렉션 기반이라 `AddUObject` 불가
- **해결**: 바인딩 함수에 `UFUNCTION()` 추가 + `AddUniqueDynamic` 사용

---

## 핵심 정리

| 항목             | 기존                      | 변경                                      |
| ---------------- | ------------------------- | ----------------------------------------- |
| 에셋 조회 방식   | `TryLoad()` 동기 블로킹   | `DataSubsystem` 캐시 조회                 |
| 슬롯당 조회 횟수 | 2회 (이름 + 등급 분리)    | 1회 (`ResolveItemData` 통합)              |
| Refresh 빈도     | 변경마다 즉시 전체 갱신   | 다음 틱에 1회 (Dirty Flag)                |
| 초기화 타이밍    | 즉시 실행 (슬롯 비어있음) | 다음 틱 또는 DataSubsystem 완료 콜백 대기 |

---

## 배운 것

- `UGameInstanceSubsystem`은 서버/클라이언트가 **완전히 독립된 인스턴스**이므로, 서버에서 호출한 함수가 클라이언트에 자동으로 영향을 주지 않는다.
- `DECLARE_DYNAMIC_MULTICAST_DELEGATE`로 선언된 델리게이트에 바인딩하려면 반드시 `UFUNCTION()`이 붙어있어야 하고 `AddDynamic` / `AddUniqueDynamic`을 사용해야 한다.
- `SetTimerForNextTick`은 현재 프레임의 모든 `BeginPlay`가 끝난 이후 실행되므로, 초기화 타이밍 문제 해결에 유용하다.

# 베이직 반

## TArray / TMap GC 주의사항

`TArray`, `TMap`을 `UPROPERTY` 없이 사용하면 **GC(Garbage Collector)가 추적하지 못함**.
UObject 포인터를 담는 컨테이너는 반드시 `UPROPERTY`를 붙여야 안전함.

```cpp
// GC 추적 불가
TArray<UMyObject*> Items;

// GC 추적 가능
UPROPERTY()
TArray<UMyObject*> Items;
```

## TSharedPtr과 UPROPERTY

`TSharedPtr`은 `UPROPERTY`를 붙이면 안됨, UObject 소유권 관리는 `UPROPERTY` + 일반 포인터로 하고, Non-UObject 리소스 관리에만 `TSharedPtr`을 사용할 것.

## TOptional

값이 없는 상태를 표현하는 언리얼 자료구조. `unset` 상태를 명시적으로 가질 수 있어 안전한 초기화에 유용함.

```cpp
TOptional<int32> MaybeValue;
if (MaybeValue.IsSet())
{
    int32 Val = MaybeValue.GetValue();
}
```

## FArchive와 세이브 시스템

`FArchive`와 `operator<<`를 오버로드하면 세이브/로드 로직을 간결하게 만들 수 있음.

```cpp
friend FArchive& operator<<(FArchive& Ar, FMyStruct& Data)
{
    Ar << Data.SomeField;
    Ar << Data.AnotherField;
    return Ar;
}
```

## CS - 자료구조

### 해시맵(HashMap)과 해시 충돌(Hash Collision)

해시맵은 키를 해시 함수로 "으깨서(hash)" 인덱스로 변환해 저장함.
직접 탐색(O(n)) 대신 해시값으로 O(1) 접근이 가능해서 빠름.

### 기타 등등

우리가 작성하는 Unreal 코드는 기본적으로 **게임 스레드(Game Thread)** 에서 실행된다.
때문에 동기 로딩 같은 블로킹 작업은 프레임 드랍 또는 게임 멈춤을 유발하므로 주의해야 함.

## 포트폴리오 가치 있는 기능 아이디어

- **JSON Export / Import** : 게임 데이터를 JSON으로 추출하고 다시 불러오는 기능 (포트폴리오 가치 높음)
- **CSV Export** : Unreal Header Tool에서 CSV로 추출 → 엑셀 또는 VSCode에서 수정 → 다시 Import
- **Binary Export / Import** : 게임 데이터를 `.bin` 형태로 내보내고 불러오는 기능
