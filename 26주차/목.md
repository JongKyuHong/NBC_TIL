### FName

TSet, TMap, TArray를 지정할때 타입이나 키를 int32가 아닌 FName으로 많이 쓰는게 궁금했다.
실제로 언리얼에서는 FName을 많이 선호한다고 한다. 우선 int32는 코드에서만 의미가 있고 사람이 봤을때 어떤 값인지 알 수 없다.

```c++
EquippedPartIds = { 14, 27, 3 }  // 14가 뭔지 모름

EquippedPartIds = { "Part_Helmet_01", "Part_Chest_01" }
```

또, UE에서 FName을 선호하는 이유중 하나는 DataAsset, Blueprint의 이름과 직접 매핑이 된다. "Part_Helmet_01"이라는 FName이 있으면 에디터에서 같은 이름의 에셋을 바로 찾을 수 있다.
성능또한 FName은 내부적으로 전역 해시 테이블에 저장돼서 비교가 int32만큼 빠르다.

### Magnitude, SetByCaller

Magnitude란 이펙트가 Attribute에 적용할 수신 값의 크기, SetByCaller는 이를 런타임에 외부에서 주입하는 방식 중 하나

#### Modifier Magnitude란?

GameplayEffect에는 Modifier라는 개념이 있는데, Modifier는 Attribute를 어떻게 변경할지를 정의하는데 -> Attribute : Health, Operation : Add, Magnitude : -30.0(얼마나 변경할 것인지)

여기서 Magnitude를 어디서 가져올지에 따라 여러 타입이 존재

- ScalableFloat : 에디터에서 고정값 또는 Curve Table로 설정
- AttributeBased : 다른 Attribute값을 기반으로 계산
- CustomCalculationClass (MMC) : C++클래스를 만들어 직접 계산 로직 작성
- SetByCaller : Ability나 코드에서 런타임에 직접 주입

#### SetByCaller

GameplayEffect 에셋 자체에는 값을 적어두지 않고, 이 이펙트를 실제로 쓰는 쪽(Ability나 코드)에서 값을 결정하는 방식
E에 SetByCaller로 지정한 태그의 값을 Spec에 주입하지 않으면, 런타임 에러 + 0 반환이 됩니다. 값을 반드시 세팅해야됨
