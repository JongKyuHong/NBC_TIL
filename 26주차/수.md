## 개념 모음

### AddDynamic vs CreateUObject

항상 혼재해 왔던 개념

1. AddDynamic은 DECLARED*DYNAMIC*\* 델리게이트에 사용된다
   - 헤더에 반드시 UFUNCTION()이 있어야 함
   - Blueprint에서도 바인딩 가능
   - 리플렉션 시스템을 통해 동작 -> 약간 느림
   - 주로 컴포넌트 이벤트(OnComponentBeginOverlap등에 사용)

2. CreateUObject는 DECLARED*DELEGATE*_ / DECLARED*MULTICAST_DELEGATE*_ (non - dynamic에 사용)
   - UFUNCTION() 없어도 됨
   - Blueprint 바인딩 불가
   - 리플렉션 없이 직접 함수 포인터 등록 -> 빠름
   - 주로 C++ 내부 콜백에 사용

Dynamic이 무엇인가?

UE의 리플렉션 시스템을 통해 동작한다는 의미
non-Dynamic은 컴파일 타임에 함수 포인터를 직접 연결, Delegate -> 함수 주소 직접 저장
Dynamic은 함수 이름 문자열로 런타임에 찾음, Delegate -> "OnOverlapBegin" 문자열 저장 -> 런타임에 UFUCNTION 테이블에서 검색

그래서 UFUNCTION에 있어야 검색이 되므로 Dynamic에서는 UFUNCTION이 필수

### GetLifetimeReplicatedProps

컴포넌트가 네트워크 복제될 때 어떤 멤버변수를 복제할지 등록하는 함수

```c++
DOREPLIFETIME_CONDITION(UNSPlayerProgressComponent, UnlockedNPCIds, COND_OwnerOnly);
```

UnlockedNPCIds를 복제하되 오너에게만 보여라, 즉 이 컴포넌트를 소유한 플레이어 본인에게만 값이 전달되고 다른 컴포넌트에서는 안보임

### 컴포넌트 생명주기

액터 컴포넌트기준

1. 생성자에서 기본값 설정
2. 액터에 붙음
3. 복제 가능 여부가 설정
4. 네트워크 시스템이 GetLifetimeReplicatedProps를 읽음
5. 서버값이 클라이언트로 동기화됨
