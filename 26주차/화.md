# 세이브 시스템 설계와 비동기 처리 정리

오늘은 세이브 시스템 아키텍처와 비동기 처리 방식, 그리고 언리얼 엔진 인터페이스 관련 개념을 정리했다. 추가로 우선순위 큐를 활용한 알고리즘 문제 풀이도 함께 복습했다.

## 세이브 시스템 설계

### Repository 패턴 도입

데이터 저장 방식을 추상화하기 위해 `INSSaveRepository` 인터페이스를 정의하고, 로컬 저장과 클라우드 저장이 각각 이를 구현하는 구조로 설계했다.

Repository 패턴의 핵심은 데이터가 어디에 저장되는지와 상관없이 동일한 인터페이스로 접근할 수 있게 만드는 것이다.

- 비즈니스 로직은 Repository 인터페이스에만 의존한다.
- 실제 저장 위치가 로컬 파일인지, 서버 API인지, 데이터베이스인지 알 필요가 없다.
- 핵심 로직은 데이터를 항상 메모리 안에 있는 것처럼 다룰 수 있다.

### 프로젝트에서의 적용

`UNSSaveGameSubsystem`은 `INSSaveRepository`의 존재만 알고 있으며, 실제 저장 방식은 전혀 신경 쓰지 않는다.

```cpp
Repo->SaveBytesAsync(...)
```

이처럼 호출부는 동일하고, 내부 구현만 로컬 저장 또는 클라우드 저장으로 달라질 수 있다. 즉, Subsystem은 저장소 구현체와 분리되어 있다.

## NSGameSaveSubsystem

### 왜 GameInstanceSubsystem인가?

세이브 시스템은 특정 레벨이나 씬에 종속되면 안 되고, 게임 전역 어디서든 접근할 수 있어야 한다. 또한 씬 전환과 관계없이 살아 있어야 하기 때문에 `GameInstanceSubsystem`을 선택했다.

정리하면 다음 이유 때문이다.

- 전역 접근이 가능하다.
- 씬과 무관하게 유지된다.
- 세이브 데이터는 게임 전체 상태와 더 가깝다.

## 동기에서 비동기로 전환

처음에는 로컬 저장만 고려해서 동기 방식으로 처리할지 고민했다. 하지만 이후 클라우드 저장 등 확장 가능성을 생각했을 때, 처음부터 비동기 구조로 맞춰두는 편이 훨씬 안전하다고 판단했다.

이 결정의 장점은 다음과 같다.

- 나중에 저장소 구현이 바뀌어도 상위 로직 수정이 적다.
- 느린 저장 작업이 게임 스레드를 막지 않는다.
- 시스템 전체 구조를 초기에 일관되게 가져갈 수 있다.

## Async와 AsyncTask

### `Async(EAsyncExecution::ThreadPool, Lambda)`

```cpp
Async(EAsyncExecution::ThreadPool, []()
{
    // 백그라운드 작업
});
```

이 방식은 언리얼이 관리하는 스레드 풀에 작업을 넘겨서 게임 스레드와 병렬로 실행하게 해준다. 자바스크립트의 `Promise`처럼 “이 작업은 뒤에서 처리해줘”라고 맡기는 느낌으로 이해할 수 있다.

추가로 정리하면 다음과 같다.

- `MoveTemp`를 사용하면 큰 데이터를 복사하지 않고 이동시킬 수 있다.
- `TArray<uint8>` 같은 바이트 배열은 복사 비용이 있을 수 있으므로 이동이 유리하다.
- 람다는 일회성 작업에 적합해서 별도 함수로 분리하지 않아도 된다.

### `AsyncTask(ENamedThreads::GameThread, Lambda)`

```cpp
AsyncTask(ENamedThreads::GameThread, []()
{
    // 게임 스레드에서 처리
});
```

백그라운드 스레드에서 작업이 끝난 뒤, 다시 게임 스레드로 돌아와 후처리를 해야 할 때 사용한다. 언리얼에서는 다음 대상들이 기본적으로 게임 스레드에서만 안전하게 접근 가능하다.

- `UObject`
- 델리게이트 브로드캐스트
- UI 업데이트

즉, 저장 자체는 백그라운드에서 처리하고, 완료 통지나 UObject 접근은 다시 게임 스레드에서 처리해야 한다.

## 언리얼 엔진 개념 정리

### FPaths

`FPaths`는 프로젝트 내 다양한 경로에 접근할 수 있도록 도와주는 static 유틸리티 클래스다.

예를 들면 다음과 같은 경로를 쉽게 얻을 수 있다.

- 프로젝트 경로
- Saved 경로
- Config 경로

파일 저장 시스템을 구현할 때 경로를 하드코딩하지 않고 안전하게 가져오기 위해 자주 사용하게 된다.

### `TArray<uint8>& Data`

파일 저장에서는 결국 데이터를 바이너리 형태로 다루게 된다. `uint8`은 1바이트 크기의 부호 없는 정수 타입이고, `TArray<uint8>`은 이 바이트들이 연속적으로 들어 있는 배열이다.

즉, 세이브 데이터는 최종적으로 이런 바이트 배열 형태로 직렬화되어 파일에 저장된다.

### FFileHelper

`FFileHelper`는 언리얼이 제공하는 파일 입출력 유틸리티 클래스다. 파일을 읽거나 저장할 때 반복적으로 사용되는 처리를 간단하게 감싸고 있어서 빠르게 구현할 수 있다.

세이브 시스템에서는 바이트 배열을 파일로 저장하거나, 파일에서 다시 읽어올 때 유용하다.

### TScriptInterface

언리얼의 인터페이스는 일반 C++ 인터페이스와 다르게 두 클래스가 함께 생성된다.

```cpp
UINTERFACE()
class UNSSaveRepository : public UInterface
{
    GENERATED_BODY()
};

class INSSaveRepository
{
    GENERATED_BODY()
};
```

- `U` 접두사 클래스는 리플렉션 시스템에 등록하기 위한 타입 정보용 껍데기다.
- `I` 접두사 클래스는 실제 함수 선언을 담당한다.

### TScriptInterface가 필요한 이유

다음처럼 인터페이스 포인터를 바로 `UPROPERTY`로 들고 있을 수는 없다.

```cpp
UPROPERTY()
INSSaveRepository* Repository;
```

이유는 `INSSaveRepository`가 `UObject`가 아니기 때문이다. `UPROPERTY`는 기본적으로 `UObject` 기반 객체를 추적해 GC가 관리할 수 있어야 하는데, 인터페이스 포인터만으로는 그 정보가 부족하다.

그래서 다음처럼 `TScriptInterface`를 사용한다.

```cpp
UPROPERTY()
TScriptInterface<INSSaveRepository> Repository;
```

이 타입은 내부적으로 실제 `UObject` 포인터와 인터페이스 포인터를 함께 들고 있어서 다음 장점이 있다.

- GC 추적 가능
- 인터페이스 호출 가능
- 언리얼 리플렉션 시스템과 호환 가능

즉, 언리얼에서 인터페이스 구현체를 안전하게 보관하려면 `TScriptInterface`가 사실상 정석에 가깝다.

### `UINTERFACE(MinimalAPI)`

`UINTERFACE`는 인터페이스 타입 정보를 언리얼 리플렉션 시스템에 등록하는 역할을 한다. 여기서 `MinimalAPI`를 붙이면 최소한의 정보만 export하도록 하여 불필요한 심볼 노출을 줄일 수 있다.

`U` 클래스는 실제 로직보다는 타입 정보 등록용 껍데기 역할에 가깝기 때문에, `MinimalAPI`를 사용하는 것이 자연스러운 선택이 될 수 있다.

## 알고리즘 복습

### 1. 더 맵게

이 문제는 가장 작은 두 값을 반복해서 꺼내 조합해야 하므로 힙 구조가 적합하다. C++에서는 힙을 직접 구현하지 않고 `priority_queue`를 사용해도 충분히 해결할 수 있다.

```cpp
priority_queue<int, vector<int>, greater<int>> pq;
```

여기서 핵심은 다음과 같다.

- 기본 `priority_queue`는 최대 힙이다.
- `greater<int>`를 사용하면 최소 힙처럼 동작하게 만들 수 있다.
- 따라서 가장 작은 원소를 빠르게 꺼낼 수 있다.

풀이 흐름은 다음과 같다.

1. 모든 스코빌 지수를 우선순위 큐에 넣는다.
2. 가장 작은 두 값을 꺼낸다.
3. 새 값을 계산해서 다시 넣는다.
4. 가장 작은 값이 `K` 이상이 될 때까지 반복한다.

이 문제를 통해 “정렬 상태를 계속 유지해야 하는 상황”에서는 우선순위 큐가 매우 강력하다는 점을 다시 확인했다.

### 2. 디스크 컨트롤러

이 문제는 작업 스케줄링 문제로 볼 수 있다. 현재 시점에서 실행 가능한 작업 중 소요 시간이 가장 짧은 작업을 우선 선택해야 평균 시간을 줄일 수 있다.

그래서 다음 두 단계가 중요하다.

- 먼저 요청 시각 기준으로 전체 작업을 정렬한다.
- 현재 시각까지 들어온 작업들만 우선순위 큐에 넣고, 그중 소요 시간이 가장 짧은 작업을 선택한다.

### 비교 함수 구조체

```cpp
struct Compare {
    bool operator()(const pair<int, int>& a, const pair<int, int>& b) {
        if (a.second != b.second) return a.second > b.second;
        return a.first > b.first;
    }
};
```

이 비교 함수는 다음 우선순위를 만든다.

1. 소요 시간이 짧은 작업 우선
2. 소요 시간이 같으면 요청 시각이 빠른 작업 우선

처음 보면 구조체와 연산자 오버로딩 문법이 낯설 수 있지만, C++에서 사용자 정의 정렬 조건을 만들 때 매우 자주 쓰는 방식이다. 우선순위 큐뿐 아니라 정렬 기준을 세밀하게 제어해야 하는 상황에서도 반복해서 만나게 된다.

## 느낀 점

이번 정리를 하면서 단순히 파일 저장 기능을 만드는 것이 아니라, 저장 방식 자체를 추상화해서 상위 시스템과 분리하는 것이 훨씬 중요하다는 점을 다시 느꼈다.

특히 다음 부분이 인상적이었다.

- 비동기 처리는 “나중에 붙이는 기능”이 아니라 초기에 구조로 설계하는 편이 좋다.
- 언리얼 인터페이스는 일반 C++ 감각으로 접근하면 헷갈릴 수 있으므로 `UINTERFACE`, `IInterface`, `TScriptInterface` 역할을 분리해서 이해해야 한다.
- 우선순위 큐는 자동 정렬이 필요한 문제에서 매우 강력한 도구다.

앞으로는 세이브 시스템에 실제 직렬화 방식, 버전 관리, 실패 처리 정책까지 붙이면 더 완성도 있는 구조가 될 것 같다.
