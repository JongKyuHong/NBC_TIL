# 언리얼 네트워크 기초 정리: Replication, RepNotify, Relevancy 한 번에 보기

언리얼 멀티플레이를 공부하다 보면 `Replication`, `RepNotify`, `Relevancy`, `NetUpdateFrequency` 같은 개념이 계속 등장한다.  
이번 글에서는 처음 공부할 때 헷갈렸던 지점을 중심으로, 나중에 다시 봐도 이해될 수 있게 간단히 정리해보려 한다.

## 1. Property Replication

Property Replication은 서버가 가진 액터의 속성값을 클라이언트로 동기화하는 기능이다.  
멀티플레이에서는 중요한 상태값을 보통 서버가 권한을 가지고 관리하고, 그 결과를 클라이언트가 받아서 화면에 반영하게 된다.

---

## 2. NetLoadOnClient

`NetLoadOnClient`는 맵에 배치된 액터가 클라이언트 측에서도 로드될 수 있도록 하는 옵션으로 이해하면 편하다.  
처음에는 “클라이언트가 직접 스폰한다”는 느낌으로 받아들일 수 있지만, 정확히는 **레벨에 배치된 액터를 클라이언트 월드에서도 로드할지**와 관련된 성격으로 보는 편이 더 정확하다.

---

## 3. RepNotify

속성이 클라이언트에 복제되었을 때 후처리가 필요하다면 `RepNotify`를 사용한다.  
C++에서는 보통 아래처럼 `UPROPERTY(ReplicatedUsing = OnRep_변수명)` 형태로 선언하고, 대응하는 함수를 `UFUNCTION()`으로 정의한다.

```cpp
UPROPERTY(ReplicatedUsing = OnRep_ServerRotationYaw)
float ServerRotationYaw;

UFUNCTION()
void OnRep_ServerRotationYaw();
```

RepNotify에서 자주 헷갈리는 점은, **C++의 OnRep 함수는 클라이언트에서 복제 결과를 반영할 때 자동 호출된다는 점**이다.  
그래서 같은 로직이 서버에서도 필요하다면 서버 쪽에서는 별도로 직접 호출하거나, 공통 함수를 분리해서 함께 실행하는 방식으로 작성해야 한다.

또 하나 헷갈리는 부분은 블루프린트 RepNotify와 C++ OnRep의 체감 차이이다.  
실무에서는 이 둘이 완전히 똑같이 동작한다고 생각하면 자주 꼬이기 때문에, “C++은 복제 시점의 콜백”, “블루프린트는 스크립트 관점의 알림”처럼 구분해서 이해하는 편이 안전하다.

### C++ OnRep와 블루프린트 RepNotify 차이

- C++의 OnRep 함수는 기본적으로 클라이언트에서 복제 결과를 반영할 때 호출된다.
- 서버에서도 같은 처리가 필요하다면 명시적으로 직접 호출하거나 공통 함수를 따로 분리해야 한다.
- 블루프린트의 RepNotify는 체감상 C++과 동일하게 보면 헷갈리는 경우가 많다.
- 따라서 둘을 완전히 같은 개념으로 외우기보다, 사용 맥락과 호출 타이밍 차이를 같이 기억하는 편이 좋다.

---

## 4. NetUpdateFrequency

`NetUpdateFrequency`는 액터가 초당 몇 번 정도 복제 업데이트 대상으로 고려될지를 정하는 최대 빈도이다.  
기본값은 보통 100으로 많이 보게 되며, 이론상 서버는 `1 / 100`초 간격으로 복제를 시도할 수 있다.

다만 이 값은 **최대치**에 가깝고, 실제 전송 빈도는 서버 성능이나 상황에 따라 달라질 수 있다.  
그래서 처음부터 이 값을 지나치게 만지기보다는, 우선 복제 구조를 올바르게 만드는 것이 더 중요하다고 느꼈다.

---

## 5. 네트워크 부하를 줄이는 방법

예를 들어 어떤 액터의 위치를 서버와 클라이언트가 맞춰야 한다고 해보자.

### 방법 1. 서버 위치를 자주 복제하기

가장 단순한 방법은 서버 위치를 자주 복제해서 클라이언트에 계속 반영하는 것이다.  
구현은 직관적이지만, 액터 수가 많아지면 네트워크 부하가 커질 수 있다.

### 방법 2. 예측 기반으로 동기화하기

조금 더 발전된 방법은 초기 위치와 속도 같은 최소한의 정보만 보내고, 클라이언트가 예측해서 보정하도록 만드는 것이다.  
즉, 무조건 자주 보내는 것만이 답은 아니고, **무엇을 보내고 무엇을 클라이언트가 계산할지**를 설계하는 것이 중요하다.

---

## 6. 언리얼 네트워크의 기본 단위

언리얼 네트워크를 이해할 때는 아래 단위를 같이 기억해두면 좋다.

### 넷 커넥션

서버와 클라이언트가 데이터를 주고받는 통로이다.  
서버는 접속한 클라이언트 수만큼 넷 커넥션을 가지고, 클라이언트는 일반적으로 서버와 통신하기 위한 하나의 넷 커넥션을 가진다.

### 채널

넷 커넥션 안에서 데이터 흐름을 관리하는 단위이다.  
넷 커넥션이 여러 채널을 관리하면서 액터 복제나 RPC 같은 데이터를 나눠서 처리한다.

### 패킷

네트워크 전송의 기본 단위이다.

### 번치

언리얼 내부에서 사용하는 특별한 패킷 단위로 이해하면 편하다.

## 7. BeginPlay는 어떻게 시작될까

처음 공부할 때 가장 헷갈렸던 부분 중 하나는 `AActor::BeginPlay()` 호출 흐름이었다.  
특히 클라이언트에는 `GameMode`가 복제되지 않는데도 왜 `BeginPlay()`가 정상적으로 동작하는지가 의문이었다.

핵심은 **GameMode는 서버 전용이지만, 게임 시작 상태는 GameState를 통해 클라이언트에도 전파된다는 점**으로 이해하면 비교적 흐름이 잘 잡힌다.  
즉, GameMode가 직접 클라이언트에 존재하지 않더라도, 게임 시작에 필요한 상태 전파는 다른 네트워크 경로를 통해 이어진다.

---

## 8. Relevancy

언리얼은 모든 액터를 모든 클라이언트에 무조건 복제하지 않는다.  
대신 어떤 클라이언트에게 “이 액터가 지금 중요한가”를 판단해서 필요한 경우에만 복제한다.

### 액터 클래스의 연관성 판단 기준

1. **Owner**  
   해당 액터를 소유하고 있는 액터이다.  
   보통 해당 액터와 Owner는 연관성이 있다고 판단한다.

2. **Instigator**  
   해당 액터에게 영향을 끼친 Pawn이다.  
   해당 액터와 Instigator는 연관성이 있다고 판단한다.

3. **AlwaysRelevant**  
   해당 액터가 모든 클라이언트에 항상 연관이 있게끔 설정하는 옵션이다.  
   예를 들면 보스 몬스터처럼 모든 플레이어에게 중요한 액터가 여기에 해당할 수 있다.

4. **NetUseOwnerRelevancy**  
   오너 액터의 연관성을 그대로 따를 때 사용한다.  
   예를 들어 캐릭터에 달린 무기는 캐릭터가 안 보이는데 무기만 보여서는 어색하므로, 오너 기준으로 판단하는 것이 자연스럽다.

5. **OnlyRelevantToOwner**  
   오너 액터에게만 연관성을 가진다.  
   예를 들어 특정 플레이어만 봐야 하는 길라잡이 액터 같은 경우에 적합하다.

6. **NetCullDistance**  
   뷰어와의 거리에 따라 연관성 여부를 결정한다.  
   가까우면 복제하고, 멀어지면 복제를 줄이거나 하지 않는 식으로 생각하면 된다.

### Pawn 클래스에서 자주 보는 기준

- **Viewer**  
  클라이언트 커넥션이 소유하고 있는 플레이어 컨트롤러를 가리킨다.

- **ViewTarget**  
  플레이어 컨트롤러가 현재 바라보고 있거나 빙의한 대상이다.

멀티플레이에서는 “누가 보고 있는가”, “지금 무엇을 기준으로 월드를 바라보는가”가 연관성 판단에 영향을 주기 때문에 이 개념들이 중요하다.
