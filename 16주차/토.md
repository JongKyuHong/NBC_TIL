# ENGINE_API

- ENGINE_API는 이 함수/클래스를 모듈 DLL밖에서도 쓸 수 있도록 공개한다는 표시
  - export같은 개념

언리얼은 코드를 모듈 단위로 쪼개서 각각 DLL파일로 컴파일한다. 그런데 DLL은 기본적으로 내부 심볼을 외부에서 접근하지 못하도록 막혀있어서 다른 모듈에서 쓰고싶은 함수나 클래스는 명시적으로 내보내기표시를 해주어야 한다. 그게 `ENGINE_API`가 하는 역할
명명규칙이 모듈이름\_API이고 프로젝트이름\_API도 이런느낌

# .lib와 dll차이

.lib는 컴파일 타임에 코드를 실행파일안에 박아넣고, .dll은 런타임에 따로 불러오는 방식이다.

## 정적 라이브러리(lib)

빌드할때 링커가 .lib파일 안의 코드를 .exe안에 통째로 복사해넣는다. 그래서 .exe 하나만 있으면 어디서든 실행되지만 실행 파일 크기가 커지고 같은 코드를 쓰는 프로그램이 여러 개 있으면 메모리에도 여러 번 올라간다.
라이브러리 코드가 바뀌면 **다시 빌드해야 반영**된다.

## 동적 라이브러리(.dll)

.dll은 실행파일안에 코드를 넣지 않고, 이름만 기억해두었다가 **프로그램이 실제로 실행될 때 OS가 메모리에 로드**한다.
여러 프로그램이 같은 .dll을 공유해서 쓸 수 있어서 메모리 효율이 좋고, .dll만 교체하면 재빌드 없이 버그 수정이나 업데이트가 가능하다. 대신 실행 환경에 해당 .dll파일이 반드시 존재해야 한다.없으면 프로그램 바로 죽음

### 왜 .dll쓸때 .lib는 같이 있는가?

.dll배포시에도 .lib가 생기는데 이 .lib는 정적 라이브러리가 아니라 Import Library, 즉 .dll안에 어떤 함수들이 있는지 링커한테 알려주는 안내서 역할이다. 실제 코드는 .dll에 있고, .lib는 빌드 타임에만 쓰이고 배포할 때는 필요 없다.
언리얼이 모듈마다 .dll로 쪼개는 이유도 이것 때문이다. 엔진 전체를 거대한 .exe파일에 박아넣는 대신 모듈별로 .dll을 분리하면 특정 모듈만 교체하거나 에디터/게임빌드를 따로 관리하기 훨씬 편하다.
