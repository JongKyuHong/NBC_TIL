# 멀티플레이 게임 개발

## NetRole

### Authority, Proxy

- 서버에 스폰된 액터가 가진 NetRole 속성 값은 언제나 Authority
- 게임에 중대한 영향을 끼치는 로직은 NetRole이 Authority일때 수행해야한다. (서버이므로)
- NetRole이 Authority인 액터가 클라이언트로 복제되었을때, 클라이언트에 복제된 액터의 NetRole 속성값은 Proxy이다.

### NetRole 종류

- None
  - 보통의 경우 리플리케이션 되지 않는 액터
  - LocalRole은 Authority고 RemoteRole이 None이라면 서버에서 스폰되고, 클라이언트로 리플리케이션 되지않는 액터라고 보면 된다.

- Authority
  - 게임에 중대한 영향을 끼칠 수 있는 권한을 가진 액터
  - 서버에서 스폰된 액터가 LocalRole으로 Authority를 가질 수 있다.
  - ex) GameMode

- Autonomous Proxy
  - Authority 액터의 복제본
  - 서버로부터 데이터를 수신 받아서 동기화도 되면서, 서버로 송신도 가능
  - ex) PlayerController

- Simulated Proxy
  - Authority 액터의 복제본
  - 서버로부터 수신받아서 동기화 당하기만 한다.
  - ex) PlayerCharacter

# 언리얼의 스마트 포인터

## 대전제

- 언리얼 UObject는 GC(가비지 컬렉션)으로 살아있음/죽음이 관리
- TSharedPtr/TUniquePtr같은 C++ RAII 스마트 포인터는 기본적으로 비 UObject용이다.
  - UObject에 TSharedPtr로 소유권을 걸면 GC?엔진 수명 모델과 충돌한다 (권장되지 않음)

## UObject 계열 포인터

### TObjectPtr<T>

- 강한 참조 형태
- 보통 UPROPERTY와 같이 씀
  - 없으면 GC/직렬화/에디터 노출 같은 리플렉션 기능이 안 걸릴 수 있다.
- 언제쓰나?
  - Actor/Component/Asset 등 UObject를 멤버로 보관해야 할때

### TWeakObjectPtr<T>

- UObject에 대한 약한 참조
  - 객체가 Destroy/GC로 사라지면 자동으로 무효화(null처럼 됨)
- 특징
  - 수명 연장을 하지 않는다.
  - 접근 시 IsValid() 체크가 기본
- 언제 쓰나?
  - 참조는 필요하지만 소유/수명 연장은 원치않는 관계
  - 순환 참조 방지

- ->를 재정의 해놓음
- .으로는 T 오브젝트 포인터 객체 자체에 접근가능
- ->로는 오브젝트 포인터에 접근 가능

# 개념 공부

## 언리얼 접두사

- 언리얼은 접두사로 어떤 역할을 하는지 유추 가능
- F : 일반 C++ 구조체
- U : UObject 상속
- A : AActor상속
- I : 인터페이스
- E : 열거형
- T : 템플릿 클래스
- B : bool 변수

- 여기서 F에 집중해보자
- FDelegateHandle : 프로젝트에서 사용했던 델리게이트 핸들러를 저장하는 변수, 함수의 고유ID를 저장한다고 보면됨 (종료하기 위해)
- 즉 기본 C++구조체나 클래스라서 언리얼 GC의 대상이 아니다.
- 언리얼 GC가 메모리를 관리하지만(U, A들은) F는 일반 C++방식으로 스택/힙에서 관리
