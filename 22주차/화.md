# [TIL] Unreal Engine 에셋 매니저 및 데이터 서브시스템 최적화, 프로파일링

오늘 한일은 각 컴포넌트, UI 위젯등에 커스텀 에셋매니저를 적용시키고 기존 캐싱구조를 사용하여 동기로드를 사용하는 곳의 hitch를 방지하고 메모리 효율을 극대화하는데 중점적으로 작업했습니다.

개념공부는 초반에 데이터에셋을 익스포팅/임포팅 할 수 있는 플러그인을 만들어볼까? 고민하여 관련된 자료들을 학습했었고 왜 구현하지 않았는지에 대해서 적었습니다. 또, 위젯이랑 패널차이, 최적화에 대한 프로파일링에 대해 공부했습니다.

---

# 작업

## 핵심 개념 및 아키텍처

- UEDAssetManager (Custom Asset Manager): 엔진 싱글톤으로서 프로젝트 전역의 에셋 로드 상태를 관리
- GetPrimaryAssetObject를 통해 이미 메모리에 올라온 에셋을 즉시 반환하고, 없을 경우에만 동기 로드를 수행하도록 설계
- UEDGameDataSubsystem: 게임의 각 단계(Lobby, UI, Monster 등)에 맞춰 에셋을 비동기로 미리 로드하고, DataCache(TMap)에 하드 레퍼런스로 저장하여 GC(가비지 컬렉션)로부터 에셋을 보호
- 중앙 집중식 로드 흐름: 에셋 ID -> 캐시 확인 -> (없을 시) (비)동기 로드 순의 파이프라인을 구축

## 주요 변경 사항 및 리팩토링

1. TryLoad() 제거 및 최적화
   기존 코드의 FSoftObjectPath::TryLoad()는 에셋이 메모리에 있더라도 매번 디스크 조회를 시도하여 성능 저하를 유발했습니다. 이를 커스텀 매니저의 캐시 참조 로직으로 대체

- EDInventoryComponent, EDInventoryPanelWidget, EDItemCraftingWidget, EDCraftTreeWidget, EDInventoryBlueprintLibrary 등등
- 효과: 동일 에셋에 대한 중복 로드 비용을 0에 가깝게 줄임

2. UI 및 라이브러리 레이어 통합
   UI 위젯과 Blueprint Library에서도 동일한 에셋 로직을 공유하도록 수정

- EDInventoryPanelWidget: 아이템 등급(Rarity) 및 이름 조회 시 매니저 캐시를 우선 참조하도록 수정.
- EDInventoryBlueprintLibrary: ResolveItemData_BP 함수를 통해 모든 블루프린트 노드에서 최적화된 데이터를 사용하도록 개편.

## 프로그래밍 팁 및 주의사항

- 템플릿 함수와 UFUNCTION: C++ 템플릿 함수는 UFUNCTION 매크로를 사용할 수 없다. 블루프린트 노출이 필요할 경우 일반 함수로 래핑하여 사용해야 함
- Context와 싱글톤: 서브시스템은 World 정보가 필요하지만, AssetManager는 GEngine에 직접 붙어 있어 어디서든(static/namespace) 안전하게 호출할 수 있다는 점을 활용했다.

# 개념 공부

## UE5 DataTable CSV/JSON 익스포트 & 임포트

### DataTable (공식 지원)

- 에디터에서 DataTable 에셋 **우클릭 → "CSV로 익스포트" / "JSON으로 익스포트"**
- 임포트: `.csv/.json` 파일을 콘텐츠 브라우저에 **드래그 앤 드롭** → 행 유형(Row Type) 구조체 선택
- 재임포트: 기존 DataTable **우클릭 → Reimport**
- CSV 첫 번째 열은 반드시 `RowName`, 이후 컬럼명이 구조체의 `UPROPERTY` 변수명과 **정확히 일치**해야 함

---

## Data Asset / Primary Data Asset JSON 직렬화

### 왜 에디터 버튼이 없는가?

- DataTable은 행/열 구조 → CSV/JSON 파이프라인 내장
- Data Asset은 `UObject` 기반 → 표 형식이 아니므로 공식 지원 없음
- 직접 C++ 코드로 구현해야 함

---

## Data Asset CSV/JSON 직렬화 실사용 사례

### 공식 문서 부재의 의미

- Epic 공식 문서는 **DataTable의 CSV/JSON만** 공식 지원
- Data Asset의 CSV/JSON 직렬화는 설계 단계부터 의도되지 않은 기능
- 다시말해 권장되는 구조가 아님 데이터에셋에서 데이터 테이블을 참조하는 구조로 가야됨

---

## UMG 위젯과 패널의 차이

### 위젯(Widget)

- UMG에서 화면에 표시되는 **모든 UI 요소**의 총칭
- 패널 위젯, 시각적 위젯(Button, Text, Image), 유저 위젯(User Widget) 등 포함

### 패널(Panel)

- 위젯의 하위 분류로 **자식 위젯을 담는 컨테이너 역할**
- 모든 패널은 위젯이지만, 모든 위젯이 패널은 아님

## UE5 인벤토리 로드 최적화 프로파일링 도구

### 분석 관점 2가지

| 관점             | 병목 원인                        | 사용 도구                       |
| ---------------- | -------------------------------- | ------------------------------- |
| 에셋 로드 타이밍 | 아이템 아이콘/메시 로드 시점     | Unreal Insights (Asset Loading) |
| UI 렌더링 비용   | 위젯 수, Invalidation, Draw Call | `stat slate`, Widget Reflector  |

### 주요 도구

#### Unreal Insights

- `-trace=asset,loading,cpu,frame` 커맨드라인 추가 후 실행
- `FAsyncLoadingThread` 트랙에서 로드 이벤트 확인
- 인벤토리 열 때 스파이크 → **동기 로드 발생** (문제)
- 스파이크 없음 → 비동기 사전 로드 정상 동작

#### `stat slate` (콘솔 명령)

- `Num Batches` — Draw Batch 수 (인벤토리 슬롯 많을수록 증가)
- `Slate RT CPU Time` — UI 렌더 스레드 비용
- `Pre-pass Layout` — 레이아웃 재계산 비용

#### Widget Reflector

- `Tools → Widget Reflector`
- 웹 개발자 도구 Elements 탭과 유사
- 클릭한 위젯의 계층 구조, 레이아웃 비용, 페인트 횟수 실시간 확인

#### `stat unit` (콘솔 명령)

- `Game` 수치 스파이크 → 로직(로드/생성) 문제
- `Draw` 수치 스파이크 → 위젯 렌더링 문제
- 가장 먼저 실행해서 병목 위치를 특정

### 프로파일링 순서

1. `stat unit` → 어느 스레드 병목인지 확인
2. **Unreal Insights** → 동기 로드(`LoadObject`) 호출 여부 확인
3. `stat slate` → Num Batches 확인
4. **Widget Reflector** → 위젯 계층 레이아웃 재계산 범위 확인
5. 수정 후 재프로파일링으로 수치 비교

## LoadPrimaryAssetsAsync, ChangeBundleStateForPrimaryAssets 차이

|              | LoadPrimaryAssetsAsync                                                     | ChangeBundleStateForPrimaryAssets                                 |
| ------------ | -------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| 역할         | Primary Asset(DA) 자체를 메모리에 올림                                     | 이미 로드된 DA의 TSoft 필드(Secondary Asset)를 추가로 로드/언로드 |
| 대상         | DA 오브젝트 자체                                                           | DA 안의 TSoftObjectPtr 필드들                                     |
| 전제 조건    | 없음 (처음 로드 가능)                                                      | DA가 이미 메모리에 있어야 함                                      |
| 번들 지정 시 | DA 로드 + 번들 로드 시도하지만 처음 로드되는 경우 TSoft 필드 로드가 불안정 | DA는 건드리지 않고 TSoft 필드만 정확히 로드                       |
