# 도입 이유

하드 레퍼런스로 인한 문제 : 모든 몬스터/아이템 데이터가 게임 시작 시 일괄 로드되어, 로비 진입 시에도 불필요한 게임씬 데이터가 메모리를 점유

# 설계

## 커스텀 에셋 매니저 : EDAssetManager

- `UAssetManager` 상속 : 엔진 내장 `FStreamableManager`를 직접 활용, 중복 관리 방지

## 데이터 관리 서브 시스템 : EDGameDataSubsystem

- `UGameInstanceSubsystem` 사용 씬 전환에도 데이터 캐시 유지, 수명 주기가 게임 인스턴스와 일치
- `DataCache` + `PhaseHandles` 이중 구조 `TObjectPtr`로 GC 방지, `FStreamableHandle`로 언로드 타이밍 명시적 제어
- `DataCache` 로 어디서든 캐싱된 데이터를 사용가능
- 단계적 순차 로딩 (Item → Monster) 의존 관계 있는 데이터의 로드 순서 보장, 콜백 체이닝으로 오버헤드 없이 구현

## 데이터 서브시스템

### **상황**

게임이 로비와 게임 씬으로 분리되어 있고, 필요한 데이터(몬스터, 아이템, 게임 UI < - > 로비 데이터, 로비 UI)가 다릅니다. 기존 하드 레퍼런스 방식은 씬과 무관하게 모든 데이터가 게임 시작 시 일괄 로드되어, 로비에서도 게임씬 데이터가 메모리를 점유하는 문제가 있었습니다. 또한 씬 전환 시 데이터 캐시를 유지할 수 있는 수명 주기를 가진 객체가 필요했습니다.

### **선택지**

1. GameMode에서 관리
   - 고려는 하였으나 씬 전환시 소멸됨
   - 게임모드에서 데이터 관리를 하는것은 적절한 설계가 아니라고 판단
2. 전역 싱글톤
   - UE의 GC 시스템 밖에 존재하여 메모리관리가 안될것이라고 판단
3. UGameInstanceSubsystem
   - 게임 인스턴스와 수명이 같고, GC내에서 안전하게 동작.

### **선택한 방법과 이유**

`UGameInstanceSubsystem`을 상속한 `UEDGameDataSubsystem`을 구현했습니다.

`UGameInstance`는 앱 실행부터 종료까지 단 하나만 존재하며 씬 전환에도 소멸되지 않습니다. 따라서 서브시스템도 동일한 수명을 가지므로, 로비 → 게임씬 → 로비 전환 과정에서 캐시(`DataCache`)와 스트리밍 핸들(`PhaseHandles`)을 안정적으로 유지할 수 있습니다.

또한 `UGameInstanceSubsystem`은 UE의 GC 시스템 안에서 동작하기 때문에, `TObjectPtr`로 에셋 포인터를 보관하면 GC가 에셋을 수거하지 않도록 명시적으로 제어할 수 있습니다. 전역 싱글톤과 달리 `Deinitialize()`가 자동 호출되므로, PIE 종료 시 `UnloadAllData()`를 통해 핸들과 캐시를 확실히 정리할 수 있습니다.

### **결과**

- 게임, 로비에 따라 알맞은 데이터를 로드, 언로드 할 수있는 데이터 시스템으로 개발
- DataCache를 통해 편리하게 캐싱 데이터에 접근
- 모든 스트리밍 핸들을 중앙 집중적으로 관리하여 의도치 않은 GC방지 및 언로드 타이밍을 명시적으로 제어 가능
- GetData<t>, GetAllDataOfType<T> 등 단일 진입점 API제공으로 어느 클래스던 일관성있는 방식으로 데이터 접근 가능
