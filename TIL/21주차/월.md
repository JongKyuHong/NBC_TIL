# 서버

## Stateful / Stateless

### Session-Oriented (Stateful)

실시간 온라인 게임서버의 대표주자 Stateful 서버
모든 클라이언트는 서버에 접속하여 연결을 유지한 상태로 플레이를 진행, 게임 로직, 데이터는 서버가 모두 게임서버의 데이터로 로직을 처리

UE클라이언트가 FSocket(UDP/TCP)로 서버와 항상 붙어있음
매 프레임 입력 -> 서버 판정 -> 결과를 전 플레이어에게 브로드캐스트

몬스터 HP, 버프, 날씨, 플레이어 좌표 등 현재 상태 전체가 서버 메모리에 상주
로그인 시 한꺼번에 로드, 로그아웃/체크포인트 때만 DB로 플러시

핵 방지가 필수인 게임에서 서버권한이 절대적

### Request-Oriented (Stateless)

실시간 컨텐츠, 실시간 상호작용 불필요한 게임에 적합한 서버

- 퍼즐, 수집형, 방치형 게임
- 실시간 플레이가 없는 게임
- 혼자 놀고 결과만 공유 게임

필요할때만 HTTP/HTTPS 호출

- 스테이지 클리어, 가챠, 랭킹 조회
- Unreal 클라이언트는 FHttpModule 비동기 POST/GET

서버는 기억상실증

- 요청마다 DB->로직->DB 패턴
- 서버 인스턴스가 1대든 1000대든 결과는 동일 -> 로드 밸런서 뒤 수평확장 쉽다.

웹 스택 활용

- Node.js / Python / Go + MySql 등

로그인 / 로비 / 결제 / 랭킹 / 통계 : Stateless -> HTTP토큰, 트랜잭션 관리, 대규모 R/W, 캐시활용
매칭 / 실시간 전투 : Stateful -> 지연 < 50ms 필수
채팅 : Hybrid(WebSocket + REST) -> 메시지는 실시간 , 로그는 REST

## 동기 비동기 블로킹 논블로킹

동기 : 작업이 끝날때까지 기다리고 다음 줄로 넘어감
비동기 : 호출한 함수가 바로 다음줄을 실행하고, 작업완료는 추후 처리

블로킹 vs 논블로킹 : 함수 호출이 CPU자원을 점유하며 다른 작업을 막는가?

블로킹 : 요청한 작업이 끝날 때까지 스레드가 대기, 다른 작업 못함, read() 함수가 데이터를 받을 때까지 CPU를 점유

논블로킹 : 요청했는데 작업이 아직 준비 안됐으면 바로 리턴하고 다음작업 넘어감, CPU점유하지 않음, read() 호출했는데 읽을 게 없으면 null 리턴하고 넘어감

결제/거래/경매 : Blocking + Sync , 데이터 정합성이 중요, 반드시 성공 여부 확인해야 함
킬 로그 기록 : Non-Blocking + Async , 실패해도 게임에 영향 없음, 비동기 큐로 적재
이펙트 출력 : Non-Blocking + Async, 클라이언트 기반 처리, 서버는 통보만
이벤트 큐 스케줄링 : Blocking or Non-Blocking + Async, 이벤트 예약이나 리워드 지급은 후처리

## 소켓 프로그래밍

1. 소켓 생성 : OS 커널에 전송/수신 버퍼 확보
2. 설정 : 버퍼 크기, 논블로킹 등 옵션
3. 연결 : 3-way handshake로 서버와 세션 형성
4. 송신 : 바이트 스트림을 패킷화해서 내보냄
5. 수신 : 커널 버퍼 -> 유저 버퍼에 복사
6. 종료 : FIN -> TIME_WAIT -> CLOSED

바이트 순서 유지 , 중복 제거

커널이 알아서 재정렬 하지만 애플리케이션 계층은 '메시지' 단위가 필요하다.

흐름제어, 혼잡제어

tcp 내부 알고리즘 우리가 직접 건드릴일 없음
너무 큰 패킷은 애초에 잘라서 보내거나 거절해야됨
