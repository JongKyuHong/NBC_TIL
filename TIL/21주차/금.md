## 오늘 한 작업 요약

PR #68, #69를 통해 몬스터 관련 시스템을 `UEDGameDataSubsystem` 기반으로 리팩토링했다.

## PR #68 - 몬스터 DataAsset 직접 참조 → Subsystem 캐시 기반 조회로 전환

### 변경된 파일

- `EDMonsterAIController.cpp / .h`
- `EDMonsterBase.cpp / .h`
- `EDMonsterSpawner.cpp / .h`
- `EDMonsterDataAsset.h`

### 핵심 변경 내용

#### 1. `EDMonsterBase` - DataAsset 직접 참조 제거

```cpp
// 변경 전
UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Monster|Data")
TObjectPtr<UEDMonsterDataAsset> DataAsset;

// 변경 후
UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "Monster|Data")
FPrimaryAssetId MonsterDataId;
```

- `DataAsset` 포인터를 직접 들고 있지 않고, `FPrimaryAssetId`만 저장
- `GetDataAsset()`은 이제 `UEDGameDataSubsystem`에서 ID로 조회하여 반환

```cpp
UEDMonsterDataAsset* GetDataAsset() const
{
    if (UEDGameDataSubsystem* Subsystem = UEDGameDataSubsystem::Get(this))
    {
        return Subsystem->GetData<UEDMonsterDataAsset>(MonsterDataId);
    }
    return nullptr;
}
```

#### 2. `EDMonsterSpawner` - MonsterDataAsset 포인터 → AssetId로 교체

```cpp
// 변경 전
TObjectPtr<UEDMonsterDataAsset> MonsterDataAsset;

// 변경 후
FPrimaryAssetId MonsterDataAssetId;
```

- `BeginPlay()`에서 `DataSubsystem->IsDataReady()` 확인 후 즉시 실행 or 딜레이 실행 분기
- 데이터가 준비되지 않으면 `OnDataLoadedResponse()` 콜백 바인딩으로 대기

```cpp
if (DataSubsystem->IsDataReady())
{
    OnDataLoadedResponse();
} else
{
    DataSubsystem->OnAllDataLoaded.AddDynamic(this, &AEDMonsterSpawner::OnDataLoadedResponse);
}
```

#### 3. `EDMonsterAIController` - BT 비동기 로드 로직 개선

- BT가 이미 메모리에 로드되어 있으면(`.Get()` 유효) 즉시 `RunBehaviorTree()` 실행
- 로드되지 않은 경우에만 `UEDAssetManager::Get().LoadAssetAsync()`로 비동기 로드
- `OnUnPossess()`에서 `BTLoadHandle` 취소 로직 제거 → `OnBTLoaded()` 콜백 내부에서 처리

#### 4. `EDMonsterDataAsset.h` - AssetBundles 메타 추가

```cpp
UPROPERTY(..., meta = (AssetBundles = "MonsterData"))
TSoftObjectPtr<USkeletalMesh> Mesh;

UPROPERTY(..., meta = (AssetBundles = "MonsterData"))
TSoftClassPtr<UAnimInstance> AnimInstance;

UPROPERTY(..., meta = (AssetBundles = "MonsterData"))
TSoftObjectPtr<UBehaviorTree> BehaviorTree;
```

- Asset Bundle을 명시해야 `LoadPrimaryAssetsAsync()` 호출 시 해당 에셋들이 함께 로드됨
- Bundle 이름이 없으면 Primary Asset 껍데기만 로드되고 내부 Soft 참조는 로드되지 않음

## PR #69 - Subsystem 캐싱 최적화 및 문자열 하드코딩 제거

### 변경된 파일

- `EDMonsterBase.cpp / .h`
- `EDGameDataSubsystem.cpp`

### 핵심 변경 내용

#### 1. `EDMonsterBase` - DataSubsystem 캐싱

```cpp
// .h에 추가
TWeakObjectPtr<UEDGameDataSubsystem> CachedDataSubsystem;

// BeginPlay()에서 캐싱
CachedDataSubsystem = UEDGameDataSubsystem::Get(this);
```

- 매번 `UEDGameDataSubsystem::Get(this)`를 호출하지 않고 `BeginPlay()`에서 한 번만 가져와 캐싱
- `TWeakObjectPtr`을 사용하는 이유: Subsystem이 GC에 의해 소멸될 경우 자동으로 null이 되어 댕글링 포인터 방지

#### 2. `LoadVisuals()` - 데이터 준비 완료 시 즉시 콜백 호출

```cpp
UEDGameDataSubsystem* DataSubsystem = CachedDataSubsystem.Get();
if (DataSubsystem && DataSubsystem->IsDataReady())
{
    OnVisualsLoaded(); // 이미 로드된 경우 비동기 없이 즉시 적용
    return;
}
// 캐시 미스면 아래 비동기 로드 수행
```

#### 3. `EDGameDataSubsystem` - 하드코딩 문자열 → 상수로 교체

```cpp
// 변경 전
PhaseHandles.Add(TEXT("Monster"), Handle);

// 변경 후
PhaseHandles.Add(MonsterAssetType.GetName(), Handle);
```

- `TEXT("Monster")`, `TEXT("Lobby")` 등의 하드코딩 문자열을 `AssetType.GetName()`으로 통일
- 상수 불일치로 인한 버그 가능성 제거

## 배운 점

### `FPrimaryAssetId` 기반 데이터 참조

- UObject 포인터를 직접 들고 있으면 GC 관리, 복제, 직렬화가 복잡해짐
- `FPrimaryAssetId`만 저장하고 Subsystem에서 조회하는 패턴이 훨씬 깔끔하고 안전함
- 데이터가 없을 때 `nullptr`을 반환하므로 유효성 체크가 자연스럽게 이루어짐

### `AssetBundles` 메타데이터의 중요성

- `UPrimaryDataAsset`을 `LoadPrimaryAssetsAsync()`로 로드할 때, 내부의 `TSoftObjectPtr` / `TSoftClassPtr`는 **Bundle 명시 없이는 함께 로드되지 않음**
- `meta = (AssetBundles = "번들이름")`을 지정해야 해당 에셋들이 같이 로드됨

### `TWeakObjectPtr` vs `TObjectPtr`

|           | `TObjectPtr`          | `TWeakObjectPtr`    |
| --------- | --------------------- | ------------------- |
| GC 방지   | O (UPROPERTY와 함께)  | X                   |
| 소멸 감지 | UPROPERTY면 자동 null | `.IsValid()`로 확인 |
| 사용 목적 | 강한 소유             | 참조만, 소유 없이   |

- Subsystem처럼 수명이 나보다 길 가능성이 높지만 보장할 수 없는 경우 → `TWeakObjectPtr` 적합

### `IsDataReady()` 분기 패턴

- 게임 시작 타이밍에 따라 Subsystem 로드가 완료됐을 수도, 아닐 수도 있음
- 항상 `IsDataReady()` 체크 후 완료 상태면 즉시 실행, 아니면 델리게이트 바인딩 패턴을 사용해야 타이밍 버그 없음
