## 2. 작업 후 구조 (After)

```
NSPlayerController
  ├─ AugmentSelectionComponent (CreateDefaultSubobject) ← 새로 부착
  └─ OpenAugmentationSelection()
       ├─ IsRunReady=true  → AugmentSelectionComponent->RequestOffer(Normal)
       └─ IsRunReady=false → DataSubsystem->EnterRun() + OnTestRunDataReady 콜백 대기

NSAugmentSelectionComponent (PlayerController에 부착)
  └─ Server_RerollCard()  ← Index 제거, 3장 전부 새로 추첨

NSAugmentationWidget
  ├─ NativeConstruct()   → GetSelectionComponent() → AddDynamic(OnOfferPresented, OnOfferClosed)
  ├─ NativeDestruct()    → RemoveDynamic + IconLoadHandle 취소
  ├─ HandleOfferPresented() → CreateChoiceCard + 소프트포인터 수집 + RequestAsyncLoad → OnIconsLoaded
  ├─ OnIconsLoaded()     → PopulateOfferCards() → (첫 오퍼면) ShowAugmentation()
  ├─ PopulateOfferCards() → SetAugmentName + SetAugmentDescription + SetAugmentIcon
  ├─ SelectCardByIndex() → ConfirmAugmentSelection(Index)
  ├─ ConfirmAugmentSelection() → CurrentOfferIds 검증 → Server_Choose(Index)
  ├─ RequestRerollAugment() → Server_RerollCard()
  └─ HandleOfferClosed() → HideAugmentation()

NSAugmentCardWidget
  └─ SetAugmentIcon(UTexture2D*) 추가
```

#### 생성자 변경


SelectionComponent는 데이터 보관소인 InventoryComponent(PlayerState)와 달리, 추첨/선택 로직을 담당하는 per-player 매니저 성격. Lyra의 InventoryManagerComponent 관례에 따라 PlayerController에 부착.

#### `Server_RequestStartRun_Implementation` 변경

```cpp
// Before: 조건이 if(HasAuthority())로 감싸져 있었고, EnterRun 호출 없음
// After:
if (!HasAuthority()) return;  // 얼리 리턴 방식으로 변경
// ... GameMode 호출 ...
DataSubsystem->EnterRun();         // 서버 인런 데이터 로드
Client_NotifyRunStarted();         // 클라이언트에도 EnterRun 지시

// Client_NotifyRunStarted_Implementation:
DataSubsystem->EnterRun();  // 클라이언트 인런 데이터 로드
```

`DataSubsystem`은 `GameInstanceSubsystem`이라 서버/클라이언트 각자 인스턴스를 가짐. 서버에서만 호출해도 클라이언트에는 반영 안 되므로 별도 Client RPC 필요.

#### `OpenAugmentationSelection` 변경 (3회 수정 끝에 최종 확정)

```
시도 1: UIManager->ShowAugmentation() 직접 호출
        → UI는 열리지만 HandleOfferPresented 우회 → CurrentOfferIds 비어있음
        → ConfirmAugmentSelection에서 인덱스 검증 실패 → Server_Choose 미호출

시도 2: Server_RequestOffer(Normal) 바로 호출
        → IsRunReady=false일 때 FindPool이 null 반환 → UI 안 열림
        → 로그도 없어서 원인 파악에 시간 소요

최종:
if (IsRunReady)
    AugmentSelectionComponent->RequestOffer(NSGameplayTags::Augment_Pool_Normal)
else
    DataSubsystem->OnRunGameDataReady.AddUniqueDynamic(this, &::OnTestRunDataReady)
    DataSubsystem->EnterRun()

// OnTestRunDataReady:
DataSubsystem->OnRunGameDataReady.RemoveDynamic(this, ...)
AugmentSelectionComponent->RequestOffer(Normal)
```

`EnterRun()`은 비동기이므로 호출 직후 `IsRunReady()`가 false인 구간이 존재. `OnRunGameDataReady` 델리게이트로 완료 시점을 기다리는 방식으로 해결.

```cpp

CreateChoiceCard(OfferIds.Num());

// 소프트포인터 수집: Icon + StackEffectClass + GrantedAbilityClass
TArray<FSoftObjectPath> PathsToLoad;
for (const FPrimaryAssetId& Id : OfferIds)
{
    const UNSAugmentDefinition* Def = Data->GetData<UNSAugmentDefinition>(Id);
    if (!Def->Icon.IsNull())           PathsToLoad.Add(Def->Icon.ToSoftObjectPath());
    if (!Def->StackEffectClass.IsNull()) PathsToLoad.Add(Def->StackEffectClass.ToSoftObjectPath());
    if (!Def->GrantedAbilityClass.IsNull()) PathsToLoad.Add(Def->GrantedAbilityClass.ToSoftObjectPath());

// 이전 로드 취소 (리롤 시 새 요청으로 교체)
if (IconLoadHandle.IsValid()) { IconLoadHandle->CancelHandle(); IconLoadHandle.Reset(); }

    IconLoadHandle = UAssetManager::GetStreamableManager().RequestAsyncLoad(
        PathsToLoad, FStreamableDelegate::CreateUObject(this, &UNSAugmentationWidget::OnIconsLoaded));
else
{
    PopulateOfferCards();
    if (GetVisibility() != ESlateVisibility::Visible) ShowAugmentation();
}
```

- 이전 "데이터 컴포넌트" 시절 로드를 보장하던 로직이 DataSubsystem 전환 시 누락됨
- 오퍼 시점에 함께 로드하면 플레이어가 선택할 때 `.Get()`으로 바로 사용 가능 (리슨서버 기준)

## 4. 핵심 설계 결정 및 고민

### 4-1. UI 통신 방식: GameplayMessageSubsystem vs 델리게이트

**검토 배경:** 재화가 HUD + 상점 + 강화창 N:N 공유 구조 예정 → 메시지 버스 검토.

**결론: 델리게이트 유지.**

`GameplayMessageRouter` 플러그인이 **UE 5.7 기본 설치에 없음** (과거 Lyra 딸림 플러그인). 대신 `AsyncMessageSystem`(Experimental)이 있으나 API 다르고 레퍼런스 부족. 직접 경량 메시지 Subsystem 구현도 고려했으나, 현재 단계에서 실제 N:N이 없으므로 기존 컴포넌트 델리게이트 방식 유지 결정.

**정리된 UI 통신 원칙:**

| 방향                 | 수단                                  |
| -------------------- | ------------------------------------- |
| 게임플레이 → UI 알림 | 컴포넌트 DYNAMIC_MULTICAST 델리게이트 |
| UI → 서버 행위       | Server RPC (항상, 메시지로 대체 불가) |
| 즉시·1:1 호출        | UIManagerSubsystem 직접 호출          |

### 4-2. 자산 로딩 전략 (B 전략)

**DataSubsystem이 하는 일:**

- `EnterRun()` → `UNSAugmentDefinition` DA 파일 자체를 메모리에 올림
- `TSoftObjectPtr<UTexture2D> Icon`, `TSoftClassPtr<UGameplayEffect> StackEffectClass` 등 소프트포인터는 **경로(문자열)만 저장**, 실제 자산은 메모리에 없음

**A 전략 (번들 로드):** `meta=(AssetBundles="InRunUI")` 추가 → EnterRun 시 50개 아이콘 전부 상주 (~12MB 메모리 상시 점유)

**B 전략 (온디맨드 로드, 채택):**

- AssetBundles 메타데이터 없음
- `HandleOfferPresented` 수신 시 확정된 3개의 소프트포인터만 `FStreamableManager::RequestAsyncLoad`
- 로드 완료(1~2프레임) 후 카드 채우고 UI 표시 → 빈 카드 노출 없음
- 아이콘 + GEClass + GAClass 한꺼번에 로드 → GE/GA null 버그도 동시 해결

**리롤 UX:**

- 기존 카드 그대로 유지 → 새 3장 로드 완료 → 한꺼번에 교체 → 빈 카드 없음

### 4-3. GE/GA를 EnterRun에 미리 로드하지 않는 이유

- GE/GA는 **서버에서 선택 시 즉각 적용** 필요 → 로드 지연 허용 불가처럼 보이지만
- 리슨서버 환경: 클라이언트 = 서버 = 같은 프로세스 → 클라이언트에서 로드하면 서버에서도 `.Get()` 유효
- 오퍼 시점에 로드된 GE/GA가 플레이어 선택 전 이미 메모리에 있으므로 `ApplyAugment`의 `.Get()` 정상 작동
- 전용 서버 확장 시: 서버 측 별도 로딩 전략 필요 (현재 미구현, TODO)

### 4-4. 리롤 방식 결정

**Before:** `Server_RerollCard(int32 Index)` — 특정 카드 하나만 교체, 현재 Rarity 유지

**After:** `Server_RerollCard()` — 카드 3개 전부 재추첨, Rarity도 새로 결정 (fresh gamble)

내부 구현: `RollAndPresent()` 전체 재실행 → `Client_PresentOffer` → `HandleOfferPresented` → B 전략 비동기 로딩 자동 적용

> 리롤 시 Rarity도 새로 뽑힘. "같은 등급 유지하며 카드만 교체" 방식이 필요하면 별도 로직으로 변경 필요.

## 5. 발견된 버그와 해결 현황

| 버그                                     | 원인                                                        | 상태                                      |
| ---------------------------------------- | ----------------------------------------------------------- | ----------------------------------------- |
