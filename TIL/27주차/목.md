# 개념들

1. Blueprint와 C++의 성능 차이가 발생하는 이유와 각각 언제 사용하는것이 좋은가??
   C++ : 코어 로직, 연산 집약 시스템
   Blueprint : 기확자 / TA가 수치 조정, 에셋 연결, 애니메이션 조합, UI 배치처럼 변경이 잦고 연산이 가벼운 작업에 적합

   성능 차이의 원인
   - VM(가상 머신) 인터프리터 방식 : Blueprint는 내부적으로 언리얼 자체 VM 위에서 바이트 코드로 실행됨. C++처럼 네이티브 기계어로 직접 실행되지 않고, 매 명령마다 VM을 거치는 오버헤드가 발생함
   - 인라인 최적화 불가 : C++ 컴파일러는 함수를 인라인화(inline)하여 함수 호출 오버헤드를 제거할 수 있지만, 블루프린트 함수는 객체로 유지되어 컴파일러 수준의 인라인 최적화를 받을 수 없다.
   - CPU 캐시 미스 : C++ 코드는 CPU캐시(L1, L2, L3)에 올라갈 가능성이 있지만, Blueprint는 항상 RAM을 통해 접근해야 하므로 캐시 히트율이 낮다.
   - 더많은 명령 처리 : Blueprint함수는 어떤 C++ 함수를 호출할지에 대한 메타정보를 가지고 있어야해서, 동일 로직이라도 더 많은 명령어를 처리함

2. UI 구현시에 어떤 부분까지 블루프린트로 하고 어떤 부분을 코드로 작업하였는가? 그 이유는 무엇인가?
   디자인은 블루프린트, 로직은 C++
   블루프린트로 하는 부분 : 위젯 레이아웃 배치, 애니메이션, 디자이너/기획자 협업 영역
   C++로 하는 부분 : UUserWidget 상속, meta=(BindWidget)으로 위젯 바인딩, 이벤트 드리븐 업데이트, 데이터 연산 및 예외처리

3. Widget과 UMG, CommonUI에 대해서 설명하시오
   Widget : UE5에서 화면에 보이는 모든 UI요소의 기본 단위 (Button, Text, Image, ProgressBar같은 개별 컴포넌트부터, 이것들을 묶는 컨테이너(CanvasPanel, HorizontalBox 등))
   코드에서 위젯의 기반 클래스는 UUserWidget이며 이를 C++에서 상속하여 커스텀 UI 클래스를 만들어서 쓴다.

   UMG : Unreal Motion Graphics, 위젯들을 조합하여 실제 UI화면을 만드는 에디터 툴 + 프레임워크 전체를 말함. Designer뷰, Graph 뷰, 위젯 블루프린트(UMG에서 생성하는 에셋 파일, .uasset 형태로 저장됨) 등등

   CommonUI : UMG위에 얹는 플러그인 형태의 확장 UI 프레임워크. 기본 UMG만으로는 처리하기 어려운 크로스 플랫폼 지원, 입력 라우팅, 레이어 기반 화면 전환 같은 복잡한 요구사항을 해결하기 위해 Epic이 만들었다.

4. Tick 함수란 무엇이며, 게임 성능 최적화를 위해 사용 시 주의해야 할 점
   Tick은 매프레임마다 자동으로 호출되는 업데이트 함수.AActor와 UActorComponent모두 Tick을 가지고 있으며, 파라미터로 받는 DeltaTime은 이전 프레임과 현재 프레임 사이의 경과시간으로, 이를 속도값에 곱해야 FPS가 달라져도 일정한 속도로 동작하는 프레임 독립적 로직을 구현할 수 있다.

   성능 최적화
   1. 불필요한 Tick 비활성화
   2. TickInterval로 호출 주기 조절
   3. 이벤트/델리게이트로 대체
   4. 중복 연산 매니저로 일원화
