# TIL: 데이터 컴포넌트 -> 데이터 서브시스템 리팩토링

## 1. 데이터 컴포넌트 -> 데이터 서브시스템

### 기존 구조

인런 데이터와 아웃런 데이터를 각각 별도의 `ActorComponent`로 관리했다.

```
GameState (InRun)   → UNSRunGameDataComponent   (인런 데이터)
GameState (OutGame) → UNSOutGameDataComponent   (아웃런 데이터)
```

```cpp
// 기존 접근 방식
UNSRunGameDataComponent* RGDC = UNSRunGameDataComponent::Get(this);
// 내부적으로 GameState->FindComponentByClass<UNSRunGameDataComponent>()

if (!RGDC || !RGDC->IsReady()) return;
UNSAugmentDefinition* Def = RGDC->GetData<UNSAugmentDefinition>(DefId);
```

로드 방식도 `RequestAsyncLoad`에 raw `FSoftObjectPath` 배열을 직접 넘기는 방식이었다.

```cpp
// 기존 로드 방식
TArray<FSoftObjectPath> AllPaths;
AM.GetPrimaryAssetDataList(Type, Assets);
for (const FAssetData& AD : Assets)
    AllPaths.Add(AD.GetSoftObjectPath());  // Primary Asset 경로만 수집

LoadHandle = AM.GetStreamableManager().RequestAsyncLoad(AllPaths, Callback);
// → DataAsset .uasset 파일 자체만 로드됨
// → 내부 TSoftClassPtr (GE, GA), TSoftObjectPtr (아이콘) 은 로드되지 않음
```

### 문제점

**생명주기가 레벨에 종속됨**  
`ActorComponent`는 `GameState`와 함께 레벨 전환 시 파괴된다. 레벨 전환 후에도 유지해야 하는 공용 데이터(Common)를 관리할 수 없었다.

###

### 변경 후 구조

두 컴포넌트를 `UGameInstanceSubsystem` 하나로 통합했다.

```
GameInstance → UNSDataSubsystem  (Common + OutGame + Run 통합 관리)
```

`GameInstanceSubsystem`은 앱 실행 중 단 한 번만 생성되고, 레벨 전환과 무관하게 유지된다. 페이즈 기반으로 어떤 데이터가 현재 메모리에 있는지 명확하게 제어한다.

```cpp
// 변경 후 접근 방식
UNSDataSubsystem* Data = UNSDataSubsystem::Get(this);
// 내부적으로 GameInstance->GetSubsystem<UNSDataSubsystem>()

if (!Data || !Data->IsRunReady()) return;
UNSAugmentDefinition* Def = Data->GetData<UNSAugmentDefinition>(DefId);
```

### 데이터 흐름

```
NotStarted
  ↓ LoadCommonData()
CommonReady        (PlayerData - 게임 전체 유지)
  ↓ LoadOutGameData()
OutGameReady       (HubData, PartData)
  ↓ EnterRun()     (OutGame 언로드 → Run 로드)
RunReady           (MonsterData, AugmentData, AugmentPool, PartData)
  ↓ ReturnToOutGame() (Run 언로드 → OutGame 재로드)
OutGameReady       (복귀)
```

## 2. RequestAsyncLoad → LoadPrimaryAssets (Bundle API)

GE/GA를 위한 다른 로직이 또 필요했고, 추후에 DLC나 쿠킹시 이점을 가지기 위해서 BundleAPI로 로드하는식으로
변경하기로 마음먹었다.

### AssetBundles 메타 + LoadPrimaryAssets API

**Step 1 – DataAsset에 AssetBundles 메타 선언**

```cpp
// NSAugmentDefinition.h
UPROPERTY(EditDefaultsOnly, meta = (AssetBundles = "UI"))
TSoftObjectPtr<UTexture2D> Icon;             // UI에서만 필요

UPROPERTY(EditDefaultsOnly, meta = (AssetBundles = "Game"))
TSoftClassPtr<UGameplayEffect> StackEffectClass;   // 서버 게임플레이에 필요

UPROPERTY(EditDefaultsOnly, meta = (AssetBundles = "Game"))
TSoftClassPtr<UGameplayAbility> GrantedAbilityClass;
```

- `AssetBundles` 메타는 "이 소프트 레퍼런스는 이 번들 소속"이라고 Asset Manager에게 알리는 태그다.
- 번들명 자체는 자유 문자열이다. 의미가 생기는 건 Bundle API로 로드하거나, `Always Cook` 규칙과 연동할 때다.
- 기존 방식에서는 Always Cook으로 지정하고 번들명은 중요치 않았다.

**Step 2 – LoadPrimaryAssets API로 전환**

```cpp
// NSDataSubsystem.cpp

// 번들명 상수 (DataAsset의 meta=(AssetBundles="...") 와 반드시 일치)
const TArray<FName> UNSDataSubsystem::LoadBundles = { FName("UI"), FName("Game") };

// 기존 방식 (Primary Asset 파일만 로드)
TArray<FSoftObjectPath> Paths;
AM.GetPrimaryAssetDataList(Type, Assets);
for (auto& AD : Assets) Paths.Add(AD.GetSoftObjectPath());
RequestAsyncLoad(Paths, Callback);

// 변경 후 (Primary Asset + 번들 의존성 한 번에 로드)
TArray<FPrimaryAssetId> Ids;
AM.GetPrimaryAssetIdList(Type, Ids);
AM.LoadPrimaryAssets(Ids, LoadBundles, Callback);
//  → "UI", "Game" 번들 태그된 소프트 레퍼런스도 같은 배치로 로드됨
//  → OnRunAssetsLoaded 호출 시점에 GE/GA 클래스도 메모리에 있음
```

**Step 3 – 2단계 의존성 로드 제거**

기존에는 DataAsset 로드 완료 후 GE/GA를 추가로 로드하는 `StartLoadRunDependents()` 함수가 있었다 (또는 주석 처리 상태였다). Bundle API가 이를 내부에서 자동으로 처리하므로 해당 함수 전체를 제거했다.

```cpp
// 기존 (2단계)
OnRunAssetsLoaded()
  → CacheLoaded()
  → StartLoadRunDependents()   // GE/GA 소프트 레퍼런스 별도 로드
      → OnRunDependentsLoaded() → SetPhase(RunReady)

// 변경 후 (1단계)
OnRunAssetsLoaded()
  → CacheLoaded()
  → SetPhase(RunReady)         // 이미 번들까지 로드 완료된 상태
```

## 요약

| 항목                 | 기존                                | 변경 후                             |
| -------------------- | ----------------------------------- | ----------------------------------- |
| 데이터 관리 주체     | `ActorComponent` (GameState 부착)   | `GameInstanceSubsystem`             |
| 생명주기             | 레벨에 종속 (레벨 전환 시 파괴)     | 앱 실행 내내 유지                   |
| 페이즈 관리          | 컴포넌트별 분산 (Run/OutGame 별도)  | 단일 서브시스템에서 통합 제어       |
| 접근 방식            | `GameState->FindComponentByClass()` | `GameInstance->GetSubsystem<>()`    |
| 로드 API             | `RequestAsyncLoad(SoftPaths)`       | `LoadPrimaryAssets(Ids, Bundles)`   |
| 소프트 레퍼런스 로드 | 미로드 (2단계 별도 필요)            | 번들과 함께 한 배치로 로드          |
| 패키징 안전성        | AssetBundles 메타 없어 누락 위험    | AssetBundles + Always Cook으로 보장 |

## 키워드 정리

- Server Travel : 서버가 주도하여 현재 연결된 모든 클라이언트를 새로운 레벨로 데려가는 방식
- Client Travel : 클라이언트 본인이 주도하여 특정 서버의 IP를 치고 들어가거나 현재 서버에서 빠져나와 로컬맵으로 돌아가는 방식
- Hard Travel : 엔진의 기본 이동 방식, 클라와 서버간의 네트워크 연결을 완전히 끊고 (Tear down), 이전 맵의 메모리를 싹 비운 뒤 새로운 맵을 로드하여 처음부터 다시 접속하는 방법
- Seamless Travel : 네트워크 연결을 끊지 않은 상태에서 뒷단으로 부드럽게 다음 맵을 로드하여 넘어가는 방식

마지막에 들어갈 원소를 찾는부분에서 count를 남발하고 있어서 성능저하가 크다.
두가지개선방안이 있다.

1. 지금 로직 유지하며 더 빠르게

```c++
#include <string>
#include <vector>
#include <algorithm>

using namespace std;

// 전역 배열로 방문(추가) 여부 체크 (0으로 초기화됨)
bool visited[100001] = { false, };

vector<int> solution(string s) {
    vector<int> answer;
    vector<vector<int>> strVec;

    // 1. 유저님의 기존 파싱 로직 (그대로 유지)
    vector<int> tmp;
    string tmpStr = "";
    for (int i = 1; i < s.length()-1; i++){
        if (s[i] == '{') continue;
        else if (s[i] == ','){
            if (tmpStr.length() > 0){
                tmp.push_back(stoi(tmpStr));
                tmpStr = "";
            }
        } else if (s[i] == '}'){
            tmp.push_back(stoi(tmpStr));
            tmpStr = "";
            strVec.push_back(tmp);
            tmp = {};
        } else {
            tmpStr += s[i];
        }
    }

    // 2. 크기 순 정렬
    sort(strVec.begin(), strVec.end(), [](const auto& a, const auto& b){
        return a.size() < b.size(); // return문을 명시해주는 것이 좋습니다.
    });

    // 3. 탐색 최적화: visited 배열 활용
    for (auto& vec : strVec) {
        for (int v : vec) {
            // 이 숫자를 아직 answer에 넣은 적이 없다면?
            if (!visited[v]) {
                visited[v] = true; // 방명록에 체크하고
                answer.push_back(v); // 정답에 추가
                break; // 한 집합당 추가되는 숫자는 무조건 1개이므로 바로 다음 집합으로 넘어감
            }
        }
    }

    return answer;
}
```

- 문자열 파싱은 그대로 하고 visited배열을 통해서 통과한다.
- 문제에 튜플내에서 중복된 숫자가 안나온다고 했는데 그걸 못봐서 비효율적으로 짰었다......
