# 증강 시스템 만들기 TIL

오늘 정리한 내용은 **언리얼 엔진 컴포넌트 리플리케이션**, `TArray` 탐색 방식, 그리고 함께 공부한 **정렬/해시 테이블/그리디 문제 풀이**까지 한 번에 복습할 수 있도록 묶어 정리한 것이다.

## 1. 언리얼 엔진 개념 정리

### SetIsReplicatedByDefault

처음에는 `SetIsReplicatedByDefault(true)`를 단순히 “이 컴포넌트는 기본적으로 리플리케이션된다” 정도로만 이해하고 있었다.
그런데 정확히는 **정적(Static) 컴포넌트**, 즉 **액터 생성자에서 `CreateDefaultSubobject`로 생성해서 붙이는 컴포넌트**에 대해 기본 리플리케이션 여부를 설정할 때 쓰는 함수라는 점을 알게 되었다.

반대로 내가 자주 사용하던 `SetIsReplicated(true)`는 주로 **동적(Dynamic) 컴포넌트**에 사용하는 쪽에 더 가깝다.
즉, 런타임에 생성되었거나 BeginPlay 이후에 다뤄지는 컴포넌트라면 `SetIsReplicated`를 사용하는 방식이 자연스럽다.

#### 정리

- `SetIsReplicatedByDefault(true)`
  - 용도: **정적 컴포넌트**의 기본 리플리케이션 설정
  - 주 사용 위치: **컴포넌트 내부 생성자**
  - 전제: 이 컴포넌트가 액터 생성자에서 `CreateDefaultSubobject`로 생성되어 붙는다.

- `SetIsReplicated(true)`
  - 용도: **동적 컴포넌트** 또는 런타임 시점의 리플리케이션 설정
  - 주 사용 위치: `BeginPlay`, 런타임 생성 직후 등
  - 전제: 이미 생성된 인스턴스에 대해 리플리케이션을 켠다.

#### 내 상황에 적용

내 경우 증강 인벤토리 컴포넌트는 **처음부터 액터에 고정적으로 붙는 구조**이므로,
생성자에서 `SetIsReplicatedByDefault(true)`를 설정하는 방식이 맞다.

즉,

- 처음부터 붙는 컴포넌트라면 → `SetIsReplicatedByDefault`
- 런타임에 생성하거나 후처리로 붙인다면 → `SetIsReplicated`

이렇게 구분해서 이해하면 된다.

### `TArray::FindByKey` vs `FindByPredicate`

이번에 `Owned`가 구조체 배열이라서 `FindByKey`가 바로 동작하지 않는 상황을 겪었다.
이유는 `FindByKey`가 내부적으로 비교 연산을 수행할 수 있어야 하는데, 내가 사용한 구조체에는 그에 필요한 `==` 연산자 정의가 없었기 때문이다.

그래서 `FindByPredicate`를 사용해 **람다에서 직접 비교 조건을 정의하는 방식**으로 바꾸게 되었다.
`FindByPredicate`는 이름 그대로 “이 조건을 만족하는 원소를 찾아라”라는 방식이라서, `bool`을 반환하는 조건식을 넘겨주면 된다.

#### 현재 코드

```cpp
UPROPERTY(ReplicatedUsing=OnRep_Owned)
TArray<FNSAugmentInstance> Owned;
```

#### 기존 코드

```cpp
int32 UNSAugmentInventoryComponent::GetStackCount(const FPrimaryAssetId& DefId) const
{
    const FNSAugmentInstance* Found = Owned.FindByKey(DefId);
    return Found ? Found->Stacks : 0;
}
```

#### 수정 코드

```cpp
int32 UNSAugmentInventoryComponent::GetStackCount(const FPrimaryAssetId& DefId) const
{
    const FNSAugmentInstance* Found = Owned.FindByPredicate(
        [&DefId](const FNSAugmentInstance& Instance)
        {
            return Instance.DefId == DefId;
        });

    return Found ? Found->Stacks : 0;
}
```

#### 핵심 차이

| 방식              | 특징                                  | 적합한 경우                                |
| ----------------- | ------------------------------------- | ------------------------------------------ |
| `FindByKey`       | 키와 원소 비교가 가능한 전제가 필요함 | 단순 타입, 혹은 비교 연산이 잘 정의된 경우 |
| `FindByPredicate` | 직접 조건식을 작성할 수 있음          | 구조체 배열, 복합 조건 비교, 커스텀 탐색   |

#### 배운 점

- 구조체 배열에서는 무조건 `FindByKey`가 되는 것이 아니다.
- 비교 기준이 단순하지 않거나 `==`가 없다면 `FindByPredicate`가 더 안전하다.
- 람다를 활용하면 “무엇을 기준으로 찾을지”를 코드에 명확하게 드러낼 수 있다.

## 2. 알고리즘 개념 정리

### 퀵 정렬

```cpp
void quickSort(vector<int>& arr, int left = 0, int right = -1)
{
    if (right == -1) right = arr.size() - 1;
    if (left >= right) return;

    int mid = std::midpoint(left, right);
    int pivot = arr[mid];

    int i = left;
    int j = right;

    while (i <= j)
    {
        while (arr[i] < pivot) i++;
        while (arr[j] > pivot) j--;

        if (i <= j)
        {
            std::swap(arr[i], arr[j]);
            i++;
            j--;
        }
    }

    if (left < j) quickSort(arr, left, j);
    if (i < right) quickSort(arr, i, right);
}
```

퀵 정렬은 **피벗을 기준으로 작은 값과 큰 값을 나누고**, 그 결과로 분할된 구간을 다시 재귀적으로 정렬하는 방식이다.
핵심은 **정렬하면서 분할이 같이 진행된다**는 점이다.

#### 특징

- 평균 시간 복잡도: `O(N log N)`
- 최악 시간 복잡도: `O(N^2)`
- 추가 메모리 사용이 적다.
- 원본 배열 내부에서 자리 교환을 하므로 in-place 정렬에 가깝다.

### 병합 정렬

```cpp
#include <vector>
#include <algorithm>

vector<int> merge(vector<int>& left, vector<int>& right)
{
    vector<int> result;
    int i = 0, j = 0;

    while (i < left.size() && j < right.size())
    {
        if (left[i] < right[j])
        {
            result.push_back(left[i++]);
        }
        else
        {
            result.push_back(right[j++]);
        }
    }

    while (i < left.size())
    {
        result.push_back(left[i++]);
    }

    while (j < right.size())
    {
        result.push_back(right[j++]);
    }

    return result;
}

vector<int> mergeSort(vector<int>& arr)
{
    if (arr.size() <= 1)
    {
        return arr;
    }

    int mid = arr.size() / 2;
    vector<int> left(arr.begin(), arr.begin() + mid);
    vector<int> right(arr.begin() + mid, arr.end());

    left = mergeSort(left);
    right = mergeSort(right);

    return merge(left, right);
}
```

병합 정렬은 배열을 **끝까지 쪼갠 뒤**, 다시 합치는 과정에서 정렬을 수행한다.
즉, 퀵 정렬과 달리 **먼저 분할하고 나중에 병합하면서 정렬**한다.

#### 특징

- 평균 시간 복잡도: `O(N log N)`
- 최악 시간 복잡도: `O(N log N)`
- 안정적인 성능을 보장한다.
- 대신 원본 크기만큼의 추가 메모리가 필요하다.

---

### 퀵 정렬과 병합 정렬 차이

| 항목             | 퀵 정렬            | 병합 정렬                     |
| ---------------- | ------------------ | ----------------------------- |
| 정렬 시점        | 분할 과정에서 정렬 | 병합 과정에서 정렬            |
| 평균 시간 복잡도 | `O(N log N)`       | `O(N log N)`                  |
| 최악 시간 복잡도 | `O(N^2)`           | `O(N log N)`                  |
| 추가 메모리      | 적음               | 많음                          |
| 실제 체감 성능   | 빠른 경우가 많음   | 안정적이지만 메모리 비용 존재 |

겉으로만 보면 병합 정렬이 더 좋아 보일 수 있다.
하지만 실제 환경에서는 퀵 정렬이 더 빠르게 체감되는 경우가 많다.

그 이유는 퀵 정렬이 **원본 배열 내부에서 연속적인 메모리 접근**을 많이 하기 때문이다.
이렇게 되면 CPU가 캐시에 올려둔 데이터를 다시 사용할 가능성이 커져서 **캐시 히트(cache hit)** 가 잘 발생한다.

반대로 병합 정렬은 임시 배열을 따로 사용하고, 여러 메모리 영역을 넘나들며 작업하게 된다.
이 과정에서 **공간적 지역성(spatial locality)** 이 깨지기 쉬워서 **캐시 미스(cache miss)** 가 더 자주 발생할 수 있다.

즉, 정리하면 다음과 같다.

- 이론적인 최악 성능 안정성은 병합 정렬이 더 좋다.
- 실제 메모리 접근 효율과 캐시 활용 면에서는 퀵 정렬이 강한 경우가 많다.
- 그래서 “실전 속도”만 보면 퀵 정렬이 더 빠른 상황이 자주 나온다.

### 해시 테이블

해시 테이블은 **키를 해시 함수로 변환해 배열 인덱스로 매핑하고**, 그 위치에 값을 저장하는 자료구조이다.
검색, 삽입, 삭제를 매우 빠르게 처리할 수 있기 때문에 캐시, 인덱싱, 딕셔너리 구조 등에 매우 자주 쓰인다.

예시로 Redis는 대표적인 **인메모리 Key-Value 저장소**이며, 단순 문자열 저장뿐 아니라 정렬된 세트, 리스트 등 다양한 자료 구조도 제공한다.
이런 특성 때문에 대용량 트래픽 환경에서 캐시나 빠른 조회 목적으로 자주 사용된다.

#### 해시 함수의 역할

1. 키를 일정한 범위의 숫자로 변환한다.
2. 그 숫자를 테이블 크기에 맞게 조정해 인덱스로 만든다.
3. 해당 인덱스에 데이터를 저장한다.

#### 좋은 해시 함수 조건

1. 결과 값이 일정한 범위를 유지해야 한다.
2. 같은 입력에 대해 항상 같은 출력이 나와야 한다.
3. 충돌이 적게 발생해야 한다.

#### 자주 사용하는 해시 함수 기법

1. **나눗셈법**
   - 키를 특정 수(보통 소수)로 나눈 나머지를 사용한다.
   - 단순하고 구현이 쉽지만, 테이블 크기 선택이 중요하다.

2. **곱셈법**
   - 상수를 곱한 뒤 소수 부분 등을 활용해 인덱스를 만든다.
   - 비교적 분포가 고르고 소수 선택 부담이 적다.

3. **문자열 해싱**
   - 문자열을 숫자 조합으로 바꾸고 다항식 방식 등으로 해싱한다.
   - 문자열 키를 다루는 해시 구조에서 자주 등장한다.

#### 충돌 처리 기법

1. **체이닝(Chaining)**
   - 같은 인덱스에 여러 원소가 들어오면 연결 리스트나 다른 컨테이너로 묶는다.
   - 충돌이 많아질수록 해당 버킷 탐색 비용이 커진다.

2. **개방 주소법(Open Addressing)**
   - 충돌이 발생하면 테이블 내부의 다른 빈 칸을 찾아 저장한다.
   - 대표 방식:
     - 선형 탐사(Linear Probing)
     - 이중 해싱(Double Hashing)

#### 핵심 포인트

- 해시 테이블 성능은 결국 **해시 함수 품질**과 **충돌 처리 전략**에 크게 좌우된다.
- 평균적으로는 매우 빠르지만, 충돌이 많아지면 성능이 급격히 떨어질 수 있다.
- 그래서 “배열처럼 빠르게 접근하면서도 키 기반 검색이 가능한 구조”로 이해하면 좋다.

## 3. 코드카타 기록

문제: [단속카메라](https://school.programmers.co.kr/learn/courses/30/lessons/42884)

```cpp
#include <string>
#include <vector>
#include <algorithm>
#include <iostream>

using namespace std;

int solution(vector<vector<int>> routes)
{
    int answer = 0;

    sort(routes.begin(), routes.end(), [](const auto& a, const auto& b)
    {
        if (a[0] != b[0]) return a[0] > b[0];
        return a[1] > b[1];
    });

    while (!routes.empty())
    {
        int start = routes.back()[0];
        int end = routes.back()[1];
        routes.pop_back();

        while (!routes.empty())
        {
            if (routes.back()[0] <= end)
            {
                end = min(end, routes.back()[1]);
                routes.pop_back();
            }
            else
            {
                break;
            }
        }

        answer++;
    }

    return answer;
}
```

### 풀이 아이디어

이 문제의 핵심은 **모든 차량이 최소 한 번은 카메라를 만나야 하고**, 동시에 **카메라 수는 최소**여야 한다는 점이다.
즉, 겹치는 구간을 최대한 하나의 카메라로 커버하는 **그리디 접근**이 필요하다.

현재 코드는 다음 흐름으로 동작한다.

1. 차량 경로를 시작 지점 기준으로 정렬한다.
2. 뒤에서부터 하나의 구간을 꺼내 현재 감시 가능한 구간의 끝 `end`를 잡는다.
3. 다음 차량의 시작점이 `end` 이하라면 같은 카메라로 커버 가능하므로 계속 묶는다.
4. 이때 단순히 포함만 확인하는 것이 아니라, **겹치는 구간의 공통 범위**를 유지하기 위해 `end = min(end, routes.back()[1])`로 갱신한다.
5. 더 이상 겹치지 않으면 카메라 하나를 확정하고 `answer++` 한다.

### 이번에 막혔던 지점

가장 오래 걸린 부분은 **while 안에서 `end`를 갱신해야 한다는 점**을 바로 떠올리지 못했다는 것이다.
이 갱신이 없으면 “현재까지 겹친 차량들이 실제로 공통으로 지나가는 지점”을 유지하지 못해서 잘못된 판단을 하게 된다.

즉, 이 문제는 단순히 구간이 겹치느냐만 보는 것이 아니라,
**겹칠 때마다 공통 교집합 구간을 계속 좁혀 가야 한다**는 점이 핵심이었다.

### 느낀 점

- 테스트 케이스를 직접 만드는 연습이 더 필요하다.
- 특히 구간 문제는 “겹치는 것처럼 보이지만 실제 공통 구간은 없는 경우”를 잘 만들어봐야 한다.
- 그리디는 구현보다도 **무엇을 기준으로 최적 선택을 유지할지**를 떠올리는 것이 더 어렵다.

예를 들면 아래 같은 케이스를 직접 만들어보면 도움이 된다.

```cpp
routes = {
    {-20, -10},
    {-15, -5},
    {-14, -13},
    {-5, -3}
};
```

이런 식으로 **큰 구간 + 아주 짧은 구간 + 경계에서 맞닿는 구간**을 섞으면,
내가 현재 유지하고 있는 기준값(`end`)이 정말 맞는지 검증하기 좋다.

## 4. 오늘의 핵심 한 줄

- 정적 컴포넌트는 `SetIsReplicatedByDefault`, 동적 컴포넌트는 `SetIsReplicated`로 구분해서 생각하자.
- 구조체 배열 탐색에서 비교 연산이 애매하면 `FindByPredicate`가 훨씬 유연하다.
- 퀵 정렬은 캐시 효율 덕분에 실전에서 강하고, 병합 정렬은 최악 성능이 안정적이다.
- 해시 테이블은 해시 함수와 충돌 처리 전략이 성능의 핵심이다.
- 그리디 문제는 “현재 선택을 왜 최적이라고 볼 수 있는지”를 끝까지 유지하는 기준이 중요하다.
