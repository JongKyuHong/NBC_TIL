# 재화 만들기

## 기획

- 공통재화, 스킬 재화는 영구적임 (아웃런에서도 존재)
- 임시재화는 인런에서만 저장되며 아웃런으로 나갈경우 사라짐(인런에서 소비)
- 서버에서 재화를 스폰시켜주면 각 클라이언트마다 각자 독립적인 상태로 존재
- 예를들어, A컴퓨터에서는 재화를 다먹고, B컴퓨터에서는 재화를 안먹은 경우 A컴퓨터에서는 재화가 안보이고 B컴퓨터에서는 재화가 보임

## 설계 의도

- 드롭파츠랑 조금 다르게 설계하고자 함
- 재화가 얼마나 생길지도 명확하지 않고, 파츠는 모든 클라에 동일하게 존재한다면 재화는 각 클라마다 드롭되는 갯수는 같겠지만 상태가 다르다 (먹었던지 안먹었던지)
- 그래서 재화 액터가 맵에 엄청나게 뿌려지더라도 최대한 성능저하가 없는 방식을 고려했다.

## 설계

서버에서 드랍에 대한 정보만 레지스트리에 등록해서 가지고 있고, 복제액터는 하나도 만들지 않음
클라에서는 비주얼 액터만 스폰함(복제하지 않음, 여기서 복제 비용도 아낄 수 있다) -> 각자 자기 화면에만 보이는 재화 액터를 표현
서버는 클라에서 재화를 획득했을때 검증만 해줌

정리하자면, 서버 복제 코인 액터 0개, 스폰비용은 멀티캐스트 1방 + 클라 로컬 스폰뿐 -> 몹/드랍이 폭증해도 성능적으로 저하가 적다.

1. 서버 드랍 레지스트리 - UNSCurrencyDropSubsystem

- TMap 형태로 DropId와 구조체를 들고 있는형태
- 구조체에는 Location정보도 필요할 거같다. 왜냐하면 오버랩될때 획득이 되어야함
- Durtaion도 같이 클라에 전달해서 만료시간으로 사용
- 습득시 검증 -> 미만료, 아직 습득 안함, 거리등 -> 재화 표시

두가지 고민 경로

- 몹 사망할때 드롭테이블을 엔트리에 등록해서 GameState의 NetMulticast 호출 -> 이게 서브시스템은 액터가 아니라서 멀티캐스트가 불가능함 그래서 GameState로 경유

- 투사체 시스템처럼 플레이어별 onwer-only 프록시 액터 + Client RPC
  - 이게 조금 더 나을것같다 멀티캐스트는 되도록이면 안쓰는 방향이 맞는듯
  - 이쪽이 나아보임

2. 클라 로컬에서 비주얼만 보여주는 액터 - ANSCurrency
   - 클라에서만 스폰
   - 로컬 충돌 스피어, 비동기 메시 로드, 자가 만료 타이머
   - 로컬플레이어가 오버랩되면 즉시 숨기고 PlayerState::Server_CollectCurrency(DropId)같은 함수 호출
     - 예외상황이 있을수 있지만 미미할듯?

3. 재화 인벤토리? 지갑? - UNSCurrencyComponent(PlayerState 부착)

- 임시 재화 : int64, COND_OwnerOnly 복제, OnRep -> HUD 델리게이트
- 런 누적 영구재화(공통/스킬) : 이번 런동안 획득한 재화 모아놓음
- 서버 API : AddTemp(Grade, Amount), TrySpendTemp(Amount), AddRunPermanent(Type, Amount), CommitRunPermanent(Multiplier)
- 델리게이트 : OnTempChanged, OnPermanentChanged (MVVM / HUD)

4. 영구 재화 흐름 (픽업 액터 안거침)

- 두가지 경로로 런 누적에 쌓이고, 런 종료 1회에 실제로 적립
- 경로1. 럭키 드랍(임시 재화랑 같은 경로로)
  - 럭키 드랍시 DropId를 레지스트리에 등록 Type만 Common이나, Skill로
  - 각 클라 Proxy Client_SpawnCurrency
  - 클라 ANSCurrency 스폰 (메시만 공통/스킬 전용으로)
  - 클라 오버랩 -> Server_CollectCurrency(DropId)
  - 서버 검증 -> AddRunPermanentCurrency(Type, Amount) -> 런 누적에 쌓이는

- 경로2. 스테이지 클리어 보상
  - NotifyStageCleared_Implementation(서버)
  - AddRunPermanenetCurrency(Type, Amount) -> 런 누적에 쌓이게

- 경로2-1. 런 종료
  - 런 누적에서 아웃런에서도 사용가능하게 적립
  - CommitPermanentCurrency(multiflier), 클리어시 1배수, 전멸 시 0.5

5. 영구재화 세이브쪽은 추후에 연동
6. 드롭테이블 데이터도 추후 연동

GameState는 서버 클라 둘다 존재 GameMode가 서버에만 존재
서브시스템에서 멀티캐스트 불가한 이유, 액터만 가능한 이유?

RPC는 액터 채널을 통해 네트워크로 전송된다. 액터 또는 그 컴포넌트 만이 복제 채널을 가진다.

# 궁금한거

- tuple은 3개이상의 값에 사용하지만 그정도면 그냥 struct를 쓰자
- USTRUCT(), UCLASS() 매크로를 쓰면 리플렉션 시스템에 등록이 되는데 리플렉션 시스템에 등록되었다고 해서 모두 GC대상인것은 아니다.
- GC대상 조건은 UObject상속 + UCLASS + UPROPERTY()가 함께 작동해야 GC가 제대로 동작한다.
  - UObject상속 : GC가 관리할 수 있는 객체 타입이 됨
  - UCLASS() : 리플렉션 시스템이 등록 -> GC가 이 클래스의 구조를 알 수 있음
  - UPROPERTY() : 멤버 변수 안의 포인터를 GC에게 알림 -> GC가 참조 추적가능

- 직렬화란 메모리 안의 객체 상태를 파일/디스크로 저장하고, 다시 불러오는 것을 말한다.
- 메모리 안의 객체 상태 -> (직렬화) -> 바이트 데이터 -> 파일 저장
- 파일 -> (역직렬화) -> 바이트 데이터 -> 다시 메모리 안의 객체 상태 복원

UE5에서 직렬화가 필요한 상황

- 게임 세이브/로드(USaveGame)
- 에디터에서 레벨을 .umap 저장
- 패키징 시 에셋을 .uasset으로 변환
- 네트워크 리플리케이션(데이터를 바이트로 변환해서 전송)

UCLASS() : 클래스 자체를 리플렉션 시스템에 등록, 리플렉션 시스템에서 '이 클래스가 존재한다'를 알 수 있음
UPROPERTY() : 멤버 변수를 리플렉션에 등록, UOBject\*같은 포인터에는 무조건 붙여야됨(댕글링 방지), 에디터에서 값 조정을 하거나(EditAnywhere), 블루프린트에서 접근해야 할 때(BlueprintReadWrite), 네트워크 리플리케이션이 필요할 때(Replicated)

안붙여도 되는경우는 : int, float, bool같은 순수 C++ 기본 타입, C++내부에서만 쓰고 에디터/블루프린트 노출이 필요없는 변수
, 매 프레임 계산용 임시 변수

UFUNCTION() : 함수를 리플렉션에 등록, GC추적은 당연히 없다 함수니까 GC가 추적하는것은 누가 이 UObject를 가리키고 있는가 즉 포인터(참조)임. 함수 자체는 포인터가 아니고 코드(명령어 집합)이므로 GC가 추적할 대상이 아니다.

UFUNCTION을 쓸까말까는 딱 한가지 C++바깥과 연결이 필요한가 아닌가(블루프린트, 네트워크, 에디터)

1. 블루프린트에서 호출해야할때 BlueprintCallable
2. 블루플니트에서 오버라이드해야 할 때 BlueprintImplementableEvent
3. 네트워크 RPC가 필요할 때 UFUNCTION(Server, Reliable)
4. 델리게이트에 바인딩할 때 (AddDynamic계열만), AddDynamic은 리플렉션시스템에 등록안하면 런타임에서 함수명을 찾을 수가없음 Dynamic으로 등록되기때문에 (함수이름을 문자열로 바인딩 -> 런타임에 리플렉션으로 탐색)
5. 에디터 버튼으로 실행하거나 콘솔 명령어로 쓸 때 UFUNCTION(CallInEditor)
