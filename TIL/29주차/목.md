## 복제문제

서버에서는 정상적으로 증강이 출력되는데 클라이언트에서 O키(디버그용키)를 눌렀을때 증강이 제대로 출력되지 않던 문제
데이터 서브시스템에서 데이터가 모두 존재하면 EnterRun()함수를 호출해서 데이터가 로드되었다는 것을 Enum으로 설정하고 이를 IsRunReady라는 함수로 확인할 수가 있다. 증강쪽 코드에는 IsRunReady()를 확인해서 데이터가 있는지 확인하고 증강을 띄워준다.
허나 아직 데이터 서브시스템을 달아놓지 않았기 때문에 둘다 동작을 안하거나 둘다 동작해야 맞는데 계속 서버만 동작하게끔 되어있었다. 그래서 디버깅을 계속하다가 문제를 찾게 되었다.
문제는 플레이어 컨트롤러에서 인런으로 넘겨주는 Server_RequestStartRun함수였다.

```c++
if (!HasAuthority())
{
	return;
}

AGameModeBase* CurrentGameMode = GetWorld()->GetAuthGameMode();
if (CurrentGameMode && CurrentGameMode->Implements<UNSOutGameInterface>())
{
	INSOutGameInterface::Execute_RequestStartRun(CurrentGameMode);
}

//서버 인런 데이터 로드
if (UNSDataSubsystem* Data = UNSDataSubsystem::Get(this))
{
	Data->EnterRun();
}

Client_NotifyRunStarted();
```

이 함수는 서버에서만 동작하고 Client_NotifyRunStarted를 호출한다. 해당 함수에서는

```c++
if (UNSDataSubsystem* Data = UNSDataSubsystem::Get(this))
{
	Data->EnterRun();
}
```

이렇게 데이터서브시스템이 있으면 EnterRun을 해서 임시로 런으로 넘겨주는 부분이 있는데 이때문에 서버에서는 EnterRun으로 넘어가고 클라에서는 Server_RequestStartRun의 !HasAuthority()를 넘기지 못하고 return되고 있었던 것

- 해결

```c++
for (FConstPlayerControllerIterator It = GetWorld()->GetPlayerControllerIterator(); It; ++It)
{
	if (ANSPlayerController* PC = Cast<ANSPlayerController>(It->Get()))
	{
		PC->Client_NotifyRunStarted();
	}
}
```

- GetPlayerControllerIterator()를 통해서 모든 플레이어컨트롤러를 가져와서 Client_NotifyRunStarted()함수를 호출해서 해결하였다.

## 비스마스킹

- 1 << n 왼쪽으로 n칸 밀겠다 비트에서

숫자 1을 8칸짜리 이진수로 써봤을때
0 0 0 0 0 0 0 1
이렇게 나온다
여기서 << 1을하면
0 0 0 0 0 0 1 0
이렇게 되고 처음에서 << 2를 하면
0 0 0 0 0 1 0 0 이렇게 된다.

- 예시코드 : 부분집합을 순서대로 구할 수 있다.

```c++
for (int i = 1; i < (1 << n); i++){
    string current_comb = "";
    for (int j = 0; j < n; j++){
        if (i & (1 << j)){
            current_comb += numbers[j];
        }
    }
    cout << current_comb << " ";
}
```

# 다른 궁금한점

1. USTRUCT가 GC대상이 아닌 이유
   UE5에서 GC는 UObject 상속 계층을 기반으로 동작함. GC가 객체를 추적하려면 해당 객체가 UObject를 상속받아야 함, USTRUCT는 UObject를 상속받지 않으므로 GC대상이 아님

2. 런타임과 컴파일타임의 차이점 및 특징
   컴파일 타임
   - 소스코드 -> 실행파일로 변환하는 시점
   - 컴파일러가 주체
   - 역할은 문법검사, 타입 검사, 기계어 변환, 최적화
   - 오류시 컴파일 에러(빌드 실패)
   - constexpr, 함수 오버로딩 결정, 템플릿 인스턴스화 등

   런타임
   - 실행파일이 실제로 실행될 때
   - OS/CPU
   - 메모리 할당, 함수 실행, 사용자 입력 처리
   - 런타임 에러(크래시, 예외)
   - new로 메모리 할당, virtual 함수 디스패치, 파일 읽기

   컴파일 타임에 결정될수록 빠르고 안전하다. 컴파일러가 미리 검사해주기 때문

3. 헤더와 cpp는 언제읽고 어떻게 링크되는가?
   컴파일단계
   - cpp파일을 한번에 하나씩 읽음 -> .obj로 변환
   - cpp안에 #include "MyClass.h"가 있으면, 컴파일러가 그 헤더를 복사해서 붙여넣기 함 (전처리기가 처리)
   - 헤더는 이런함수/클래스가 존재한다는 선언만 담고있음
   - cpp는 실제 구현을 담고있음

   링크 단계
   - .obj파일을 하나의 실행파일로 조립
   - A.cpp에서 B.cpp에 정의된 함수를 호출하면, 컴파일단계에서는 이름은 알지만 주소는 모름 상태
   - 링커가 각 .obj를 보고 함수 이름과 실제 주소를 연결해줌ㄴ

   그렇다면 헤더는 언제 검사하는건지?? 컴파일타임에서 헤더 cpp둘다 검사하는건지??
   - 헤더 자체는 컴파일 대상이 아님
   - 전처리단계(컴파일전에 실행) -> #include "MyClass.h"를 만나면 전처리기가 해당 헤더파일 내용을 그 자리에 통째로 복사/붙여넣기 함
   - 즉 컴파일러 입장에서는 .h.과 .cpp가 거대한 .cpp로 합쳐진것처럼 보임

4. virtual 소멸자 동작
   C++에서 포인터 타입이 부모 클래스일때 delete를 호출하면, virtual이 없으면 부모 소멸자만 호출됨, 자식 호출자가 호출되지 않아 자식이 할당한 메모리가 누수됨
   그러면 부모에서 virtual소멸자랑 그냥 소멸자 둘다 있어야 하는건지 ??
   - 그냥 virtual하나만 있으면 됨

5. Delegate가 STRUCT인 이유
   Delegate는 상태를 가진 함수 포인터 묶음, 함수 포인터 + 바인딩된 객체 포인터 + 몇 가지 메타데이터를 하나로 묶은 데이터 컨테이너

6. 전략 패턴이란?
   알고리즘을 교체 가능하게 만드는 설계 패턴, 조건문 덩어리 대신 동작 자체를 객체나 함수 포인터로 분리

   ```c++
   // 전략 패턴 없이 → 조건문 지옥
    void attack(string type) {
        if (type == "sword") { /* 검 공격 */ }
        else if (type == "magic") { /* 마법 공격 */ }
    }

    // 전략 패턴 적용
    class IAttackStrategy {
    public:
        virtual void Execute() = 0;
    };

    class SwordAttack : public IAttackStrategy {
    public:
        void Execute() override { /* 검 공격 */ }
    };

    class MagicAttack : public IAttackStrategy {
    public:
        void Execute() override { /* 마법 공격 */ }
    };

    class Character {
        IAttackStrategy* strategy;
    public:
        void SetStrategy(IAttackStrategy* s) { strategy = s; }
        void Attack() { strategy->Execute(); }  // 런타임에 교체 가능!
    };
   ```
