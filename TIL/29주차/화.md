## 배운 것 요약

드랍된 파츠를 줍고, 슬롯에 장착하고, 기존 파츠를 바닥에 다시 스폰하는 흐름을
테스트하면서 마주친 이슈와 해결 방법을 정리한다.

## 테스트 구성 개요

### 임시 테스트 코드 사용 (삭제 예정)

| 항목                         | 내용                                            |
| ---------------------------- | ----------------------------------------------- |
| `DebugPartData`              | 몬스터 드랍 연동 전 수동으로 파츠 데이터 세팅용 |
| `ServerRequestPickupNearest` | G키로 가장 가까운 드랍 액터를 줍는 임시 RPC     |
| 레벨 블루프린트 G키 바인딩   | 몬스터 드랍 연동 후 삭제 예정                   |

> 몬스터 드랍 연동 완료 후 위 세 가지 모두 제거할 것.

## 테스트 절차

### 사전 준비

**DA_TestArmPart 생성 (없는 경우)**

1. Content Browser 우클릭 → Miscellaneous > Data Asset
2. 클래스: `NSPartDefinition` → 이름: `DA_TestArmPart`
3. 내부 설정:
   - Part Slot: `Arm`
   - Part Name: `Test Arm`
   - Drop Mesh: 아무 스켈레탈 메시
   - Value Range: Common → Min 10, Max 20

**BP_DroppedPart 세팅**

뷰포트에서 드랍 액터 선택 → Details 패널 → Debug > Debug Part Data:

- Definition Ptr: `DA_TestArmPart`
- Current Rarity: `Common`
- Current Value: `15.0`

두 번째 드랍 액터는 첫 번째를 Ctrl+D로 복제 후 위치만 이동.

**레벨 블루프린트 G키 구성**

기존 G키 노드 전부 삭제 후 아래 순서로 재구성:

1. 빈 곳 우클릭 → `G` → Keyboard Events > G
2. Pressed 핀 → `Get Player Controller` (Index 0)
3. → `Get Player State`
4. → `Get Component by Class` (Component Class: `NSPartEquipComponent`)
5. → `Server Request Pickup Nearest`
6. G Pressed 실행 핀 → `Server Request Pickup Nearest` 실행 핀 연결

## 테스트 케이스

### TC-A: 첫 장착 확인

**목표:** G키로 드랍 파츠를 처음 주웠을 때 슬롯에 장착되는지 확인

**절차:**

1. PIE 실행
2. 첫 번째 드랍 액터 반지름 300 이내로 이동
3. G키 입력

**기대 로그:**

```
[PartEquip] 장착 완료 — Slot: 1 | Rarity: 0 | Value: 15.0 | Def: DA_TestArmPart
[PartEquip] GE 없음 — Slot: 1 (EffectClass 미지정)
```

> `GE 없음`은 정상이다. DA에 EffectClass를 지정하지 않았기 때문이며,
> GE 테스트는 EffectClass 지정 후 별도로 진행한다.

---

### TC-B: 교체 + 기존 파츠 드랍 확인

**목표:** 이미 파츠가 장착된 상태에서 다른 드랍 액터를 주우면 기존 파츠가 바닥에 스폰되는지 확인

**절차:**

1. TC-A 완료 상태 유지
2. 두 번째 드랍 액터 근처로 이동
3. G키 입력

**기대 로그:**

```
[PartEquip] 기존 파츠 드랍 — Slot: 1 | Rarity: 0 | Value: 15.0
[PartEquip] 장착 완료 — Slot: 1 | Rarity: 0 | Value: 15.0 | Def: DA_TestArmPart
```

**기대 시각:**

- 플레이어 발 근처에 드랍 액터 새로 스폰됨
- 두 번째 드랍 액터 사라짐

---

## 로그 해석표

| 로그                                    | 의미                                                      |
| --------------------------------------- | --------------------------------------------------------- |
| `장착 완료`                             | EquipPart 정상 동작                                       |
| `기존 파츠 드랍`                        | DropPartInSlot 정상 동작, 교체 발생                       |
| `GE 없음`                               | EffectClass 미지정 — 정상, GE 테스트는 별도 진행          |
| `EquipPart 실패 — Definition 로드 불가` | DebugPartData.DefinitionPtr 미지정. Details에서 DA 재확인 |
| `ServerRequestPickupNearest` 무반응     | 반지름 300 안에 드랍 액터 없음. 더 가까이 이동 필요       |

## 트러블슈팅

### [이슈 1] Make NSPartData 핀이 비어 있음

**증상**

BP에서 `Make NSPartData` 노드를 꺼냈는데 설정 가능한 핀이 하나도 없음.

**원인**

`FNSPartData`의 멤버 변수들이 전부 `BlueprintReadOnly`로 선언되어 있어서,
Make 노드가 쓰기 가능한 핀을 노출하지 않은 것.

Unreal의 Make 노드는 내부적으로 구조체의 각 멤버에 접근해 값을 채우는데,
`ReadOnly`면 외부에서 쓰기가 불가능하므로 핀 자체가 생성되지 않는다.

**해결**

`NSPartTypes.h`의 `FNSPartData` 멤버를 `BlueprintReadWrite`로 변경
→ 에디터 완전 종료 → VS에서 빌드 → 재시작
→ 기존 Make NSPartData 노드 삭제 후 새로 추가

---

### [이슈 2] G키(TryPickup) 후 F키(Initialize) 누르면 pending kill 에러

**증상**

```
Attempted to access BP_NSDroppedPart_C_1 via property ...
but BP_NSDroppedPart_C_1 is not valid (pending kill or garbage)
```

**원인**

두 가지 문제가 중첩되었다.

1. G키(TryPickup)가 액터를 `Destroy`하고 나면 레벨 블루프린트가 들고 있던
   레퍼런스는 이미 무효(pending kill) 상태가 된다.
   이 상태에서 F키(Initialize)가 그 레퍼런스에 접근하면 에러가 발생한다.

2. Initialize 없이 G를 누르면 `StoredInstance.DefinitionPtr`이 비어 있어
   `EquipPart 실패 — Definition 로드 불가` 로그가 남는다.

**해결**

반드시 **F(Initialize) → G(TryPickup)** 순서로 실행해야 했으나,
`DebugPartData` 자동 Initialize 도입으로 F키 자체가 불필요해졌다.
현재는 G키만으로 전체 흐름이 동작하므로 F키 바인딩은 삭제한다.

---

## 정리

- `BlueprintReadOnly` vs `BlueprintReadWrite`는 Make 노드 핀 노출 여부에 직접 영향을 준다.
- 레벨 블루프린트에서 액터 레퍼런스를 들고 있을 때, 해당 액터가 Destroy되면
  레퍼런스는 즉시 무효가 되므로 접근 전 유효성 체크가 필요하다.
- 테스트용 임시 코드(`DebugPartData`, G키 바인딩)는 몬스터 드랍 연동 시 반드시 제거할 것.
