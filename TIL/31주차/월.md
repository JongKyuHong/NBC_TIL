# 트러블슈팅 — 상호작용 시스템 & 드롭 파츠

## 1. 상호작용 프롬프트가 특정 파츠에서만 안 뜸

**증상**  
일부 드롭 파츠에 다가가도 프롬프트 위젯이 표시되지 않음. 다른 파츠는 정상.

**원인**  
`NSInteractionComponent::OnSphereBeginOverlap`에서 겹침 시점에 `CanInteract()`를 검사해 후보 목록에서 제외했음.  
드롭 파츠는 비동기로 데이터(`StoredInstance`)를 채우기 때문에, 겹침 이벤트가 먼저 도착하면 `CanInteract = false` → 후보에서 영구 제외.

**해결**  
`OnSphereBeginOverlap`에서 `CanInteract` 검사 제거. 모든 `INSInteractable` 구현체를 즉시 후보에 추가.  
`UpdateActiveTarget`(Tick)에서 `CanInteract`를 매 프레임 재검사해 활성 대상을 결정.

```cpp
// NSInteractionComponent.cpp — BeginOverlap
// 이전: CanInteract 통과한 것만 추가
// 수정: 구현체면 무조건 추가, 유효성은 Tick에서 판단
if (OtherActor->Implements<UNSInteractable>())
{
    Candidates.AddUnique(OtherActor);
}
```

## 2. 프롬프트 위치가 완전히 엉뚱한 곳에 표시됨

**증상**  
프롬프트 위젯이 파츠/NPC 위에 뜨지 않고 화면 한쪽 구석이나 허공에 표시됨.

**원인 1 — 바운드 계산이 DetectionSphere 포함**  
`ShowPromptFor`에서 액터 바운드(`GetActorBounds`)를 기준으로 상단 Z를 계산했는데, `USphereComponent` 반경이 합산되어 높이가 수백 유닛 이상 튀었음.

**원인 2 — 상의(Body) 파츠 메시 피벗 어긋남**  
`SetupVisual`에서 `MeshComp`를 Z 방향으로 보정해 이동했지만, `DetectionCollision`과 `PromptAnchor`는 루트에 고정된 채 그대로여서 보이는 메시와 위치가 달랐음.

**해결**  
각 액터에 `USceneComponent PromptAnchor`를 추가하고, `INSInteractable` 인터페이스에 `GetPromptWorldLocation()` 메서드를 추가.  
`ShowPromptFor`는 바운드 계산 대신 이 메서드 결과를 사용.  
`SetupVisual`에서 메시 Z 보정 후 `DetectionCollision`과 `PromptAnchor`도 함께 재배치.

```cpp
// NSDroppedPart.cpp — SetupVisual
const FVector MeshCenter(MeshBounds.Origin.X, MeshBounds.Origin.Y, MeshBounds.BoxExtent.Z);
if (DetectionCollision) { DetectionCollision->SetRelativeLocation(MeshCenter); }
if (PromptAnchor)       { PromptAnchor->SetRelativeLocation(MeshCenter + FVector(0, 0, MeshBounds.BoxExtent.Z + 30.f)); }
```

## 3. F 키를 눌러도 파츠 줍기/교체가 안 됨

**증상**  
프롬프트는 뜨는데 F를 눌러도 파츠가 장착되지 않음. 아무리 가까이 붙어도 동일.

**원인**  
`TryPickup`에서 `GetActorLocation()`(루트 위치)과 플레이어 위치 사이의 거리를 비교했는데,  
`SetupVisual`이 `MeshComp`를 Z 보정해도 루트는 그대로라 실제 메시 위치와 루트 위치가 달랐음. 루트 기준 거리가 항상 임계값 초과.

**해결**  
거리 계산 대신 `DetectionCollision->IsOverlappingActor(InstigatorPawn)` 사용.  
프롬프트가 뜨는 조건(DetectionSphere 겹침)과 동일한 기준이므로 데드존 없음.

```cpp
// NSDroppedPart.cpp — TryPickup
if (!DetectionCollision || !DetectionCollision->IsOverlappingActor(InstigatorPawn))
{
    return;
}
```

## 4. `NSInteractionComponent`가 Controller에 붙어 있어 상호작용이 항상 실패

**증상**  
`TryInteract` 호출 시 항상 `ActiveTarget`이 없어서 아무것도 실행 안 됨.  
`OnSphereBeginOverlap`도 한 번도 호출 안 됨.

**원인**  
`NSInteractionComponent`를 `NSPlayerController` 생성자에서 만들었는데,  
`TryInteract`가 Pawn에서 `FindComponentByClass`를 했고 Controller에 붙은 컴포넌트는 Pawn에서 검색 안 됨.  
또한 `DetectionSphere`가 Controller에 붙어 있어 월드에 존재하지 않으므로 겹침 이벤트 자체가 발생하지 않았음.

**해결**  
`NSPlayerCharacterBase` 생성자로 이동. Pawn에 붙어야 물리 위치를 가져 DetectionSphere가 정상 작동.

```cpp
// NSPlayerCharacterBase.cpp — 생성자
InteractionComp = CreateDefaultSubobject<UNSInteractionComponent>(TEXT("InteractionComp"));
```

## 5. `LogStreamableManager: RequestAsyncLoad() called with empty or only null assets!` 경고

**증상**  
파츠 줍기(F)를 누를 때마다 로그 창에 경고 1줄씩 출력됨.  
빈 공간에서 F를 누르면 경고 없음. 파츠 주울 때만 발생.

**원인 추적 과정**

1. 처음에는 `GrantAbilities`의 빈 배열 `RequestAsyncLoad` 의심 → 가드 추가했으나 경고 지속
2. `[LoadHunt]` 로그를 `GrantAbilities`와 `ApplyPartEffect`에 추가 → 경로는 유효하게 찍힘
3. GE/GA를 데이터 에셋에서 비워도 경고 발생 → GE/GA 아님 확인
4. 최종 원인: `NSPartSlotButton::SetPart`에서 `PartIconImage->SetBrushFromSoftTexture(InPartDefinition->Icon)` 호출 시 `Icon`이 None(미설정)이어서 내부적으로 빈 경로로 비동기 로드 요청 발생

**해결**  
`Icon.IsNull()` 가드 추가.

```cpp
// NSPartSlotButton.cpp — SetPart
if (!InPartDefinition->Icon.IsNull())
{
    PartIconImage->SetBrushFromSoftTexture(InPartDefinition->Icon);
    PartIconImage->SetVisibility(ESlateVisibility::HitTestInvisible);
}
else
{
    PartIconImage->SetBrushFromTexture(nullptr);
    PartIconImage->SetVisibility(ESlateVisibility::Hidden);
}
```

**교훈**  
`SetBrushFromSoftTexture`는 내부에서 `RequestAsyncLoad`를 직접 호출함.  
소프트 레퍼런스를 넘기기 전에 반드시 `IsNull()` 검사 필요.

## 6. `DetectionRadius` 값과 실제 스피어 반경이 에디터에서 동기화 안 됨

**증상**  
에디터 Details 패널에서 `DetectionRadius`를 바꿔도 스피어 시각화 크기가 즉시 반영되지 않음.

**원인**  
생성자에서 `SetSphereRadius(InteractRadius)` 한 번만 호출. 에디터 수정 시 재호출 없음.

**해결**  
`OnConstruction`에서 `SetSphereRadius` 재호출. 에디터에서 값 변경 시마다 `OnConstruction`이 재실행됨.

```cpp
void ANSDroppedPart::OnConstruction(const FTransform& Transform)
{
    Super::OnConstruction(Transform);
    if (DetectionCollision) { DetectionCollision->SetSphereRadius(InteractRadius); }
}
```

NPC 베이스(`NSInteractableNPCBase`)도 동일 패턴 적용.

## 7. Server RPC가 `_Implementation` 빌드 에러 또는 미호출

**증상**  
`UFUNCTION(Server, Reliable)` 선언 후 빌드 에러 또는 런타임에 RPC가 실행 안 됨.

**원인**  
UE5에서 `Server` RPC는 함수명이 반드시 `Server_`로 시작해야 UHT가 올바르게 처리함.  
`ServerRequestEquip`처럼 접두사 없이 쓰면 `_Implementation` 생성이 의도대로 안 될 수 있음.

**해결**  
`Server_RequestEquip`, `Server_RequestPickup` 등 `Server_` 접두사로 통일.  
`Client_` RPC도 동일하게 `Client_` 접두사 사용.

## 9. 리슨 서버에서 클라 플레이어 주변의 상호작용 위젯이 호스트 화면에도 표시됨

**증상**  
2인 리슨 서버 환경. 클라이언트 플레이어가 상호작용 액터(NPC 또는 드롭 파츠)에 접근하면,  
클라이언트 화면에는 정상적으로 프롬프트 위젯이 뜨지만 **호스트 화면에도 동일 위젯이 표시**됨.  
클라 플레이어가 멀어지면 호스트 화면의 위젯도 사라짐.

![](https://velog.velcdn.com/images/kyu_/post/4c06c4af-72b7-4606-b8f6-cdaed00daef0/image.png)

**원인**  
`NSInteractionComponent::BeginPlay`에서 **모든 머신의 모든 폰**에 대해 무조건 셋업을 실행했기 때문.

리슨 서버는 모든 플레이어 폰을 서버 프로세스 안에서 관리한다. 문제는 두 가지였다.

1. **오버랩 델리게이트가 비로컬 폰에도 바인딩됨**  
   `BeginPlay`에서 `IsLocallyControlled()` 검사 없이 `DetectionSphere->OnComponentBeginOverlap.AddDynamic(...)` 을 호출해, 호스트 위에서 실행되는 클라 폰의 `NSInteractionComponent`에도 델리게이트가 붙었다.

2. **DetectionSphere 콜리전이 모든 폰에서 활성 상태**  
   생성자에서 `SetCollisionProfileName(TEXT("OverlapAllDynamic"))`만 설정하고 콜리전 자체를 끄지 않아, 비로컬 폰의 스피어도 오버랩 이벤트를 발생시켰다.

결과적으로 호스트 위에서 돌아가는 클라 폰이 오버랩을 감지 → `Candidates`에 추가 → Tick에서 `UpdateActiveTarget` 실행 → 위젯을 호스트 화면에 표시하는 흐름이 발생했다.

`UpdateActiveTarget` 내부의 `GetOwnerController()` 가드(`if (!PC)`)는 컨트롤러 유무만 보는데,  
리슨 서버에서 원격 플레이어 폰도 서버 입장에서 `GetController()`가 non-null `PlayerController`를 반환하므로 이 가드는 원격 폰을 걸러내지 못한다.

**해결**  
컴포넌트에 `EnableLocalInteraction()` 함수를 추가하고, 로컬 전용 셋업을 이 함수 하나로 집중. 비로컬·중복 호출은 즉시 리턴.

변경된 파일과 내용:

**`NSInteractionComponent.h`**

- `EnableLocalInteraction()` public 함수 추가
- `IsOwnerLocallyControlled()` private 헬퍼 추가
- `bLocalInteractionEnabled` 중복 방지 플래그 추가

**`NSInteractionComponent.cpp`**

- 생성자: `DetectionSphere->SetCollisionEnabled(ECollisionEnabled::NoCollision)` 추가  
  → 비로컬 폰의 스피어는 콜리전 쿼리 자체가 발생하지 않음
- `BeginPlay`: 오버랩 바인딩 / 위젯 클래스 세팅 제거, `EnableLocalInteraction()` 호출로 대체  
  → attach 등 머신 공통 셋업만 남음. 스탠드얼론/리슨 호스트 본인 폰은 이미 possess 돼 있어 여기서 활성화됨
- `EnableLocalInteraction()` 구현:

  ```cpp
  void UNSInteractionComponent::EnableLocalInteraction()
  {
      if (bLocalInteractionEnabled) { return; }
      if (!IsOwnerLocallyControlled()) { return; }

      bLocalInteractionEnabled = true;

      if (PromptWidgetClass) { PromptWidgetComponent->SetWidgetClass(PromptWidgetClass); }
      DetectionSphere->OnComponentBeginOverlap.AddDynamic(this, &UNSInteractionComponent::OnSphereBeginOverlap);
      DetectionSphere->OnComponentEndOverlap.AddDynamic(this, &UNSInteractionComponent::OnSphereEndOverlap);
      DetectionSphere->SetCollisionEnabled(ECollisionEnabled::QueryOnly);
  }
  ```

- `IsOwnerLocallyControlled()` 구현: `APawn::IsLocallyControlled()` 사용

**`NSPlayerCharacterBase.h/.cpp`**

- `OnRep_Controller()` 오버라이드 추가
- `PossessedBy`(서버/리슨 호스트 경로)와 `OnRep_Controller`(클라이언트 본인 폰 — 컨트롤러가 리플리케이션으로 도착하는 시점) 양쪽에서 `EnableLocalInteraction()` 호출

`OnRep_PlayerState`가 아닌 `OnRep_Controller`를 진입점으로 쓴 이유: `IsLocallyControlled()`는 내부적으로 `Controller`에 의존하고, 클라이언트 본인 폰에는 `Controller`가 리플리케이트된다. `BeginPlay` 단독으로는 클라이언트에서 컨트롤러가 아직 null일 수 있어 불안정하다.

**결과**

| 경로                             | 활성화 여부                                  |
| -------------------------------- | -------------------------------------------- |
| 스탠드얼론 / 리슨 호스트 본인 폰 | `PossessedBy` 또는 `BeginPlay` 폴백에서 활성 |
| 리슨 서버 위의 원격 플레이어 폰  | `IsLocallyControlled() == false` → 무시      |
| 클라이언트 본인 폰               | `OnRep_Controller`에서 활성                  |

![](https://velog.velcdn.com/images/kyu_/post/1100a687-931d-4dab-82a9-eb99704cfd4b/image.png)

비로컬 폰의 `DetectionSphere`는 `NoCollision` 상태로 남아 오버랩 이벤트 자체가 발생하지 않으며, 오버랩 델리게이트 바인딩과 위젯 인스턴스 생성도 일어나지 않는다.

# 상호작용 시스템 (feature/NPC-Interaction)

> 기록일: 2026-06-22  
> 브랜치: feature/NPC-Interaction  
> 관련 파일: `NSInteractionComponent`, `NSInteractable`, `NSInteractableNPCBase`, `NSPartsNPC`, `NSPetNPC`, `NSPartEquipWidget`, `NSPetUpgradeWidget`, `NSPlayerCharacterBase`

---

## 전체 구조 한눈에 보기

```
[플레이어 폰]
  └── NSInteractionComponent     감지 + 입력 처리 + RPC 담당
        └── DetectionSphere      근접 오버랩 감지
        └── PromptWidgetComponent 화면에 띄우는 상호작용 프롬프트

[NPC 액터]
  └── ANSInteractableNPCBase     NPC 공통 베이스 (ACharacter + INSInteractable)
        └── ANSPartsNPC          파츠 장착 NPC
        └── ANSPetNPC            펫 강화 NPC
```

---

## 상호작용 흐름

```
① 플레이어가 NPC에 접근
   DetectionSphere 오버랩 → Candidates 목록에 추가 → Tick 활성화

② Tick마다 UpdateActiveTarget() 실행
   후보 중 가장 가까운 INSInteractable 대상 → ActiveTarget
   해당 NPC의 PromptAnchor 위치에 프롬프트 위젯 표시

③ F키 입력 → TryInteract()
   클라이언트에서 CanInteract 사전 검사 (반응성용)
   → Server_RequestInteract(Target) RPC 전송

④ 서버에서 Server_RequestInteract_Implementation 실행
   CanInteract 재검증 (서버 권위)
   → 통과 시 Client_OnInteractApproved(Target) RPC 전송

⑤ 클라이언트에서 Client_OnInteractApproved_Implementation 실행
   Execute_OnInteract(Target, PC) 호출
   → ANSPartsNPC or ANSPetNPC의 OnInteract_Implementation 실행
   → 해당 UI 위젯 생성 + AddToViewport
```

## 인터페이스: INSInteractable

모든 상호작용 가능한 액터가 구현해야 하는 인터페이스.

| 함수                       | 설명                                 | 실행 위치                        |
| -------------------------- | ------------------------------------ | -------------------------------- |
| `CanInteract(PC)`          | 이 플레이어가 상호작용 가능한지 판단 | 클라(Tick 표시용) + 서버(검증용) |
| `OnInteract(PC)`           | 실제 상호작용 처리 (UI 오픈 등)      | 클라이언트                       |
| `GetPromptText()`          | 프롬프트에 표시할 텍스트             | 클라이언트                       |
| `GetPromptWorldLocation()` | 프롬프트 위젯이 뜰 월드 위치         | 클라이언트                       |

`Execute_OnInteract(Target, PC)` 형태로 호출하는 이유: UHT가 자동 생성하는 정적 디스패처로, C++ 구현체든 BP 구현체든 올바른 쪽으로 알아서 디스패치한다.

---

## NPC 베이스: ANSInteractableNPCBase

`ACharacter + INSInteractable` 조합. 공통 로직 구현.

- **`CanInteract_Implementation`**: `PlayerState`의 `NSPlayerProgressComponent`에서 `IsNPCUnlocked(NPCId)` 확인. NPCId는 에디터 Details → NPC 카테고리에서 설정.
- **`PromptText`**: 에디터 Details → NPC 카테고리에서 직접 입력.
- **`PromptAnchor`**: 프롬프트 위젯이 뜰 위치. 에디터에서 드래그로 조정 가능.
- **`DetectionCollision`**: `NSInteractionComponent`의 `DetectionSphere`가 오버랩 감지에 사용.

## NSInteractionComponent 주요 설계 포인트

### 비로컬 폰 격리 (EnableLocalInteraction)

리슨 서버에서 서버가 모든 플레이어 폰을 관리하기 때문에, 아무 처리 없이 두면 클라이언트 폰의 상호작용 UI가 호스트 화면에도 표시되는 문제가 생긴다.

해결: 로컬 전용 셋업(오버랩 바인딩 + 콜리전 활성화 + 위젯 클래스)을 `EnableLocalInteraction()` 하나로 집중하고, 비로컬 폰에서는 아무것도 하지 않는다.

```
생성자: DetectionSphere 콜리전 = NoCollision (기본 OFF)

EnableLocalInteraction() 호출 시점:
  - BeginPlay      → 스탠드얼론 / 리슨 호스트 폴백
  - PossessedBy    → 서버 / 리슨 호스트 경로
  - OnRep_Controller → 클라이언트 본인 폰 경로 (컨트롤러 리플리케이션 도착 시)

IsLocallyControlled() == false → 즉시 리턴, 아무것도 안 함
```

| 폰                               | 결과                                        |
| -------------------------------- | ------------------------------------------- |
| 스탠드얼론 / 리슨 호스트 본인 폰 | 활성                                        |
| 리슨 서버 위의 원격 플레이어 폰  | 비활성 (콜리전 OFF, 바인딩 없음, 위젯 없음) |
| 클라이언트 본인 폰               | OnRep_Controller에서 활성                   |

### Tick 최적화

오버랩 중인 후보가 없으면 Tick을 끈다. 후보가 생기면(BeginOverlap) 켜고, 전부 사라지면(EndOverlap) 다시 끈다.

### 서버/클라 역할 분리

| 단계                        | 실행 위치       | 이유                      |
| --------------------------- | --------------- | ------------------------- |
| 오버랩 감지 + 프롬프트 표시 | 클라이언트 로컬 | UI는 로컬에서만           |
| CanInteract (Tick)          | 클라이언트 로컬 | 프롬프트 표시 여부 판단용 |
| TryInteract → Server RPC    | 클라→서버       | 서버에 검증 요청          |
| CanInteract (서버 검증)     | 서버            | 실제 권위 검증            |
| OnInteract → UI 오픈        | 클라이언트      | UI는 서버에서 열 수 없음  |

## UI: NSPartEquipWidget / NSPetUpgradeWidget

| 함수                    | 역할                                                     |
| ----------------------- | -------------------------------------------------------- |
| `OpenForInteractor(PC)` | AddToViewport + 마우스 커서 표시 + GameAndUI 입력 모드   |
| `RequestClose()`        | 변경사항 있으면 저장 확인 다이얼로그, 없으면 즉시 닫기   |
| `CloseWidget()`         | RemoveFromParent + 마우스 숨김 + GameOnly 입력 모드 복구 |

닫기(ESC) 처리: `NativeOnKeyDown`에서 Escape 키 감지 → `RequestClose()` 호출 → `FReply::Handled()` 반환으로 이벤트 소비. PIE에서는 ESC가 세션 종료로 먼저 잡히므로 테스트 시 BP 닫기 버튼 사용.
