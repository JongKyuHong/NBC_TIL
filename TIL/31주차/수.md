## 파츠 아이콘 간헐적 미표시 (패키징 전용)

### 증상

- 인런 중 몬스터 드롭 파츠를 줍고 Tab을 여러 번 열고 닫으면 장착 파츠 아이콘이 간헐적으로 사라짐
- 에디터 PIE에서는 재현 안 됨, 패키징 빌드에서만 발생
- 양쪽 클라이언트 동시에 사라지는 케이스도 목격됨

---

### 시도 1 — 서버에서 EquippedParts 배열이 초기화되는 경로 의심

**단서:** 양쪽 클라이언트 동시에 사라진다 → 클라이언트 개별 문제가 아니라 서버에서 발생하는 문제일 것.

**의심 경로:**

```
CommitCharacterSelection (캐릭터 선택 NPC)
→ UploadLocalProgress()
→ Server_UploadProgress_Implementation()
→ ApplyEquippedPart()
    → EquipComp->ClearAll()   ← 전체 파츠 삭제
    → SaveGame 기준 1개만 재장착
```

인런 중 캐릭터 선택 NPC를 건드리면 SaveGame 기준으로 파츠가 리셋되면서 배열이 일시적으로 비워질 수 있다고 판단.

**추가한 로그:**

- `OnRep_EquippedParts`에 배열 크기 로그
- `NSPartUtils::ResolvePartDefinition`에 DataCache 미스 로그

**결과:** 양쪽 클라이언트 동시 소멸 케이스는 이 경로가 원인일 수 있으나, 주요 재현 케이스(Tab 여러 번)와는 패턴이 달랐음. 보류.

### 시도 2 — 비동기 로드 타이밍 문제 의심 (메모리 미로드)

**단서:** 패키징에서만 발생 + "간헐적" 패턴 → 에디터와 패키징의 에셋 메모리 상태 차이 의심.

에디터는 프로젝트 열 때 레퍼런스된 에셋을 모두 메모리에 올려두지만, 패키징 빌드는 명시적으로 로드하기 전까지 메모리에 없음.

`ApplySlot` 흐름:

```
RefreshEquippedParts → ApplySlot
→ ResolvePartDefinition → Definition 반환 (성공)
→ SlotButton->SetPart(*PartData, Def)
    → SetBrushFromSoftTexture(Def->Icon)  ← 여기가 의심
```

`NSPartDefinition::Icon`이 `TSoftObjectPtr<UTexture2D>`. Definition 에셋이 메모리에 로드되어도 내부의 소프트 참조 텍스처는 별도로 로드해야 함.

**"있다가 없다가"인 이유:** `SetBrushFromSoftTexture`는 내부적으로 비동기 로드를 요청하지만 완료 콜백도 없고 핸들도 보관하지 않음. Tab을 열 때 이전 요청이 완료됐으면 텍스처가 메모리에 있어 표시되고, 아직 완료되지 않았으면 빈 이미지로 표시됨 — 순전히 타이밍에 의존.

**패키징에서만 터진 이유:** 에디터는 에셋이 항상 메모리에 있어 `Icon.Get()`이 항상 유효. 패키징은 쿠킹(파일 포함)과 런타임 로드가 분리되어 있어, 쿠킹된 에셋이라도 명시적으로 로드하지 않으면 `Get()` = null.

---

### 해결

`NSPartSlotButton::SetPart`에서 Icon을 명시적 비동기 로드 후 콜백에서 설정하도록 수정.

**`NSPartSlotButton.h`**

```cpp
TSharedPtr<FStreamableHandle> IconLoadHandle;
```

**`NSPartSlotButton::SetPart` (Icon 처리 부분)**

```cpp
if (UTexture2D* LoadedIcon = InPartDefinition->Icon.Get())
{
    // 이미 메모리에 있으면 즉시 설정
    PartIconImage->SetBrushFromTexture(LoadedIcon);
    PartIconImage->SetVisibility(ESlateVisibility::HitTestInvisible);
}
else if (!InPartDefinition->Icon.IsNull())
{
    // 로드 완료 후 설정, WeakThis로 위젯 소멸 후 크래시 방지
    TWeakObjectPtr<UNSPartSlotButton> WeakThis(this);
    TSoftObjectPtr<UTexture2D> SoftIcon = InPartDefinition->Icon;
    IconLoadHandle = UAssetManager::GetStreamableManager().RequestAsyncLoad(
        SoftIcon.ToSoftObjectPath(),
        [WeakThis, SoftIcon]()
        {
            if (!WeakThis.IsValid()) return;
            if (UTexture2D* Tex = SoftIcon.Get())
            {
                WeakThis->PartIconImage->SetBrushFromTexture(Tex);
                WeakThis->PartIconImage->SetVisibility(ESlateVisibility::HitTestInvisible);
            }
            WeakThis->IconLoadHandle.Reset();
        });
}
```

**`NSPartSlotButton::ClearPart`**

```cpp
if (IconLoadHandle.IsValid())
{
    IconLoadHandle->CancelHandle();
    IconLoadHandle.Reset();
}
```

## 자료구조

### 컨테이너 map 설명 + 시간 복잡도에 대해 설명해주세요

`unordered_map` 비교, 레드블랙트리 vs 균형이진탐색트리, 해시 충돌이 잦으면 `unordered_map` 성능 변화

우선 그냥 map은 레드블랙트리로 이루어져 있다. unordered_map은 해시테이블로 이루어져 있다.
둘의 차이는 크게 키값을 기준으로 정렬이 되느냐 안되느냐의 차이가 있다.
레드블랙트리와 균형이진탐색트리의 차이는 레드블랙트리는 노드를 빨간색, 검은색으로 두고 루트는 항상 검은색 새로운 노드는 빨간색으로 삽입된다. 빨간노드의 자식은 항상 검은색이며 즉, 빨간 노드는 연속될 수 없다. 어떤 노드에서 리프까지의 경로에 있는 검은 노드 수는 모두 동일해야 한다. 이 규칙들이 위반되면 회전과 색변환으로 트리를 재조정한다.
균형 이진탐색트리는 레드블랙트리를 포함하고 있다. 레드블랙트리는 균형이진탐색트리의 한 종류, 다른건 AVL트리인데 좌우 자식 서브트리의 높이 차이가 1이하로 엄격하게 균형을 맞춤 균형이 깨지면 즉시 회전함.
해시 충돌이 잦으면 성능이 낮아진다. 해시 충돌이 발생하면 같은 버킷에 여러키가 링크드리스트로 쌓인다. 결국 원하는 키를 찾기 위해 리스트 전체를 순회해야 하므로 O(N)이 된다.
리해싱으로 인한 O(N), 버킷 대비 데이터가 너무 많아지면 테이블 크기를 늘리고 전체 데이터를 재배치하는 리해싱이 발생. 이때 모든 원소를 다시 처리하므로 O(N)임

### 해시 테이블의 동작 원리를 설명해주세요.

해시테이블은 우선 해시함수와 버킷이 존재한다. 해시함수는 어떤 값이 들어왔을때 해시변환을 통해 해시값을 만들고 이를 인덱스로 사용하는 버킷에다가 저장한다. 해시 충돌이 발생하면 같은 버킷에 링크드리스트형태로 값이 쌓이게 되는데 올바른 해시값을 찾기위해서 탐색을 해야하므로 성능저하가 발생한다. 이를 해결하기 위해서는 여러 방법이 있는데 그중 하나가 체이닝인데 내가 설명한 그방법 충돌이 난경우 링크드 리스트를 연결해서 여러 데이터를 저장하는 방식. 하나는 개방 주소법 충돌이 발생하면 다른 빈 버킷을 찾아 저장하는 방식 여러개 있다.

선형 조사법 : 충돌 발생 시 다음 인덱스로 순차 탐색
이차 조사법 : 탐색 간격을 제곱 형태로 늘려서 클러스터링 완화
이중 해싱 : 충돌 발생 시 두 번째 해시함수로 이동거리를 계산해 탐색, 가장 충돌 분산효과가 좋다.

리해싱 : 만약에 이렇게해도 데이터가 너무 많아져 Load Factor(데이터수/버킷 수)가 일정 기준을 초과하면 테이블 크기 자체를 늘리고 전체 데이터를 새로 해싱하는 방법

### 멀티 쓰레드 환경에서 안정적으로 사용할 수 있는 자료구조

멀티스레드 환경이 위험한 이유 : 여러 스레드가 동시에 같은 데이터에 접근하면 레이스 컨디션이 발생 ( A도 count = 5를 읽고 +1, B도 count=5를 읽고 +1을 하면 실제로는 7이되어야 하는데 실제로 6이 저장되는 문제) 이를 막기 위해 임계 구역 : 한번에 하나의 스레드만 접근해야 하는 코드블록을 보호해야 한다.

멀티스레드 환경에서 안정적으로 사용하려면 ThreadSafe한 자료구조가 필요하다. C++ STL에서 ThreadSafe한 자료구조는 없는데 기존 자료구조를 안전하게 감싸는 방법을 선택하는것이 핵심, 언리얼에서는 존재한다.
접근방법 3가지

1. Mutex로 직접 감싸기

```c++
std::unordered_map<int, int> myMap;
std::mutex mtx;

// 쓰기
{
    std::lock_guard<std::mutex> lock(mtx); // 블록 끝나면 자동 unlock
    myMap[key] = value;
}

// 읽기
{
    std::lock_guard<std::mutex> lock(mtx);
    auto it = myMap.find(key);
}
```

구현이 단순하지만, 모든 읽기/쓰기가 순차처리되므로 스레드가 많아질수록 성능 병목이 생김

2. std::atomic(Lock-Free, 단순타입전용)

int, bool, pointer 같은 단순 타입 하나를 보호할때 사용, Lock없이 CPU명령어 수준에서 원자적 연산을 보장하므로 Mutex보다 빠름

```c++
std::atomic<int> count = 0;
count++; // 여러 스레드에서 동시에 호출해도 안전
```

컨테이너 전체에는 사용할 수 없고 단일 변수 보호 용도로 한정

3. 언리얼 엔진 전용 Thread-Safe 컨테이너

언리얼 엔진은 멀티스레드 전용으로 설계된 컨테이너를 별도로 제공
TQueue : Lock-Free큐, 단일/다중 생산자-소비자 패턴 지원
TLockFreePointerList : Lock-Free 스택 깁만, Task Graph시스템 내부에서 사용
FCriticalSection : 언리얼 래핑 Mutex, TArray / TMap 보호 시 사용

일반적으로 UE5에서 멀티스레드 작업 시 TQueue+필요한 곳에서만 FCriticalSection 조합이 가장많이 쓰임
