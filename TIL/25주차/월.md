# 면접 질문

## UObject 삭제 과정(GC가 어떻게 찾는지)을 설명해주세요

UPROPERTY로 등록된 포인터만 따라감, Mark and Sweep 알고리즘으로 동작하는데

- Mark(표시) : Root Set에서 시작해 `UPROPERTY`로 연결된 모든 `UObject`를 탐색하며 살아있음으로 표시
- Sweep(정리) : 탐색이 끝난 후, 표시되지 않은(Unreachable) 오브젝트를 메모리에서 제거

기본값 60초마다 한 번 실행

- Root Set : GC 탐색의 시작점
  GC는 아무 오브젝트나 기준점으로 삼지 않음, Root Set이라는 특별한 오브젝트 집합에서 탐색을 시작
  Root Set에 포함된 오브젝트는 GC수거대상에서 영원히 제외됨, Root Set에 등록하는 방법은 AddToRoot()이며, 반대로 해제할 때는 RemoveFromRoot()를 호출함

- UPROPERTY로 선언된 포인터가 해당 오브젝트를 참조하고 있는 상태일때 -> GC가 탐색 중 발견 -> Unreachable 플래그 해제 -> 수거 안됨
- UPROPERTY 포인터 변수에 nullptr이 대입되거나, 해당 포인터를 갖고 있는 오브젝트 자체가 파괴될 때 -> GC탐색 경로에서 끊김 -> Unreachable 유지 -> 수거됨
- UPROPERTY가 없는 포인터는 GC입장에서 해당 오브젝트를 아무도 참조하지 않는 것으로 간주하므로, 다음 GC 사이클에서 삭제됨. 그 결과 해당 포인터는 삭제된 메모리를 가리키는 댕글링 포인트가 되어 크래시를 유발함

## UStrcut와 UClass의 차이점

UCLASS는 자아를 가진 객체, USTRUCT는 데이터의 묶음

1. UCLASS
   `UObject`를 상속받는 언리얼의 핵심 단위
   - 참조 방식 : 항상 포인터(\*)를 통해 참조하며, 힙 메모리에 할당됨
   - 가비지 컬렉션 : 시스템이 직접 수명관리를 해줌, UPROPERTY로 선언된 포인터를 통해 도달 가능성을 판단
   - 상속 : 클래스간의 상속이 자유롭고, 블루프린트에서 해당 클래스를 상속받아 새로운 블루프린트 클래스를 만들 수 있음
   - 기능 : 네트워크 복제(Replication), 블루프린트 노드 구현, 컴포넌트 포함 등 무거운 기능을 모두 수행할 수 있음
   - 접두사 : 클래스 이름 앞에 U(일반 객체) 또는 A(액터)가 붙음

2. USTRUCT
   데이터를 저장하기 위한 가벼운 구조체
   - 참조 방식 : 기본적으로 값(Value)타입, 다른 클래스의 멤버 변수로 직접 포함되거나 스택에 생성됨 (만약 UCLASS의 멤버변수로 포함된 경우 UCLASS인스턴스가 힙에 있으므로 구조체도 힙에 함께 저장)
   - 가비지 컬렉션 : GC가 이 구조체 자체를 관리하지 않음, 구조체를 들고있는 `UObject`가 파괴될때 같이 사라짐
     - 구조체 내부에 `UPROPERTY`포인터 변수가 있다면, 그 포인터가 가리키는 객체는 GC로부터 보호받을 수 있음 -> 하지만 구조체 자체가 GC대상은 아님
   - 상속 : 블루프린트에서는 구조체를 상속받을 수 없음
   - 기능 : UCLASS보다 메모리 오버헤드가 훨씬 적고 빠름, 가벼운 데이터 저장(예 : 좌표, 아이템 정보, 스탯 등)에 최적화 되어있음
   - 접두사 : 이름 앞에 F가 붙음(FVector, FItemData)

## USTRUCT는 GC대상인가요?

언리얼 GC는 오직 UObject 기반 객체만 관리함, USTRUCT는 UObject를 상속받지 않으므로 GC수거 대상이 아님
USTRUCT는 GC가 아닌 자신을 소유한 컨텍스트에 의해 해제됨

- UObject의 멤버 변수로 포함된 경우 -> 해당 UObject가 GC에 수거될 때 함께 사라짐
- 로컬변수(스택)로 선언된 경우 -> 함수 스코프가 끝나는 순간 자동 소멸

## TMap과 std::map의 내부 동작 차이

std::map은 레드블랙트리기반, TMAP은 해시 테이블 기반

1. std::map
   - 내부적으로 자가 균형 이진 탐색 트리 : 레드-블랙 트리로 구현
   - 데이터가 삽입될 때마다 키값을 기준으로 정렬 상태를 유지
   - 시간 복잡도 : 탐색, 삽입, 삭제 모두 O(log N)
   - 해시 함수 : 없음, 키 기준 자동 정렬
   - 순회 속도 : 상대적으로 느림
   - 언리얼 리플렉션 : 지원 불가(UHT 인식 불가)

2. TMAP
   - 내부적으로 해시 테이블 기반, TMap은 내부적으로 데이터를 저장할 때 `TSet<TPair<Key, Value>>` 사용, TSet은 내부적으로 TSparseArray를 사용
   - 데이터를 정렬하지 않으며, 키 값을 해싱하여 인덱스를 결정
   - 시간복잡도 : 평균 O(1), 최악의 경우(해시 충돌 시) O(N), Key를 통한 조회는 O(1), Value를 통한 조회는 O(N)
   - 해시 함수 : 필요함 (GetTypeHash 사용)
   - 순회 속도 : 매우 빠름(메모리가 연속되어 있음 Array이므로)
   - 언리얼 리플렉션 : 지원가능(UPROPERTY)

## 언리얼 엔진 대표 3가지 String 종류

1. FName
   - 에셋 관리와 식별자 용도로 만들어진 가장 가벼운 문자열
   - 빌드 시 해시값으로 변환되어 전역테이블에 저장
   - 비교 연산이 단순한 정수 비교 수준으로 매우 빠름
   - 대소문자 구분 안함
   - 불변(Immutable) : 한 번 생성된 FName은 수정할 수 없음
2. FText
   - 다국어 현지화를 위한 문자열, 3가지 중 가장 무거움
   - 내부적으로 문자열 테이블의 키로 작동, 빌드 시 해당키에 맞는 언어별 텍스트로 자동 교체
   - 불변(Immutable) : 내용을 직접 수정하기보다는 Format 함수를 통해 새로운 FText를 생성하는 방식을 권장
   - 표시 전용 : 비교연산에 적합하지 않으며, 오직 보여주기 위한 데이터
3. FString
   - 자유로운 조작이 가능한 가변 문자열
   - 파일 경로 추출, 문자열 파싱, 이름 합치기등 동적인 조작이 필요한 모든 상황에 사용
   - 가변성(Mutable) : std::string처럼 문자를 추가, 삭제, 교체하는 조작이 쉽다.

## Thread들이 공유하는 자원

Thread는 하나의 프로세스내에서 실행되는 흐름의 단위, 프로세스가 실행중인 프로그램의 '거대한 틀'이라면 쓰레드는 '일꾼'

1. Thread가 공유하는 자원(Shared Resources)
   프로세스에게 할당된 메모리 영역 중 다음 세가지는 모든 쓰레드가 함께 사용함
   - 코드 영역 : 실행할 프로그램의 명령어들이 저장된 공간, 모든 쓰레드는 같은 프로그램을 실행하므로 이 영역을 공유함
   - 데이터 영역 : 전역변수(Global)나 정적변수(Static)가 저장되는 곳, 어떤 쓰레드에서든 이 변수들에 접근하고 수정할 수 있음
   - 힙 영역 : new나 malloc을 통해 동적으로 할당받은 메모리 공간, 한 쓰레드가 생성한 객체라도 주소값(포인터)만 알면 다른 쓰레드에서 접근할 수 있음

2. 쓰레드가 개별적으로 가지는 자원(Private Resource)
   쓰레드가 서로 간섭받지 않고 독립적으로 관리해야 하는 '개인공간'도 있음
   - 스택 영역 : 함수의 매개변수, 지역 변수, 복귀 주소 등이 저장됨. 쓰레드마다 독립적인 함수 호출 흐름을 가져야 하므로 공유하지 않음
   - PC 레지스터 (Program Counter) : 쓰레드가 현재 코드의 어느 부분을 실행하고 있는지 가리키는 포인터, CPU는 여러 쓰레드를 아주 빠르게 번갈아 실행하므로, 각 쓰레드는 자신이 어디까지 일했는지 기억해야 함
   - 레지스터 세트 : 현재 연산 중인 값들을 저장하는 CPU 내부의 임시 저장소

## 프로세스와 스레드 차이 + 스레드 생성이 가벼운 이유

자원을 점유하는 방식

- 프로세스 : 실행 중인 프로그램의 인스턴스, 운영체제로부터 독립된 메모리 영역을 할당받음(Code, Data, Heap, Stack), 프로세스끼리는 서로의 메모리에 직접 접근할 수 없으며 통신하려면 IPC(Inter Process Communication)라는 복잡한 절차가 필요함
- 스레드 : 프로세스 내에서 실행되는 흐름의 단위. 프로세스가 할당받은 자원 중 Code, Data, Heap영역을 공유하고, 자신만의 Stack과 PC레지스터, 레지스터 셋 전체만 따로 가짐

### 스레드 생성이 가벼운 이유

스레드를 흔히 LWP(Light Weight Process)라고 부르는 이유는 프로세스 생성 시 발생하는 거대한 비용을 건너뛰기 때문

1. 자원 할당의 생략
   프로세스를 새로 만드려면 운영체제는 해당 프로그램을 위한 새로운 가상 주소 공간을 설계하고, 메모리를 할당하고, 실행코드를 로드해야함. 반면, 스레드는 이미 만들어진 프로세스의 메모리 공간에 '숟가락만 얹는격'
   단순히 작은 스택공간만 할당해주어도 즉시 일을 실행할 수 있다.

2. 컨텍스트 스위칭의 효율성
   CPU가 일을 하다가 다른일로 넘어갈 때, 이전 작업의 상태를 저장하고 새 작업의 상태를 불러오는 것을 '컨텍스트 스위칭'이라고 함
   - 프로세스 전환 : 메모리 주소 체계 자체가 바뀌어야 함, 이 과정에서 CPU의 캐시메모리(TLB 등)를 비워야 하므로 성능 저하가 큼
   - 스레드 전환 : 같은 프로세스 안에서 움직이므로 메모리 주소 체계를 바꿀 필요가 없음, 캐시 데이터도 그대로 재사용할 수 있어 전환속도가 압도적으로 빠름

3. 데이터 통신 비용의 절감
   프로세스끼리 데이터를 주고받으려면 시스템콜을 해야하며 이 과정은 매우 느림, 스레드는 한 집(프로세스)에 같이 살고 있으므로 별도의 절차없이 공유 변수를 통해 데이터를 즉시 주고받을 수 있음

## 컨텍스트 스위칭이 무엇인지 또, 컨텍스트 스위칭의 수행 작업에 대해 설명

컨텍스트 스위칭이란 멀티태스킹 환경에서 CPU가 한 프로세스(또는 스레드)의 실행을 멈추고, 다른 프로세스로 제어권을 넘기는 과정, CPU는 한번에 하나의 일만 할 수 있지만 이 스위칭 작업이 워낙 빠르게 일어나기 때문에 사용자는 여러 프로그램이 동시에 실행되는것처럼 느끼게 됨

### 컨텍스트란?

CPU가 해당 프로세스를 실행하기 위해 필요한 모든 정보를 말함
레지스터셋(PC, SP등)이 바로 이 컨텍스트의 핵심, 이 정보를 알고 있어야만 나중에 다시 돌아왔을때 어디까지 했더라? 라면 하던일을 이어서 할 수 있음

### 수행 작업

CPU가 프로세스 A에서 B로 넘어갈때 내부적으로 다음과 같은 정교한 작업이 일어남

1. 실행 중단 및 상태 보존 : 상태(레지스터 값, 프로그램 카운터 등)를 프로세스 A의 PCG(Process Control Block)에 저장
2. 프로세스 상태 변경 : 프로세스 A의 상태를 '실행중'에서 '준비' 또는 '대기'로 바꿈
3. 스케줄러의 선택 : 운영체제의 스케줄러가 다음에 실행할 프로세스 B를 결정
4. 다음 프로세스 정보 로드 : 새로 실행할 프로세스 B의 PCB에서 이전에 저장해두었던 컨텍스트 정보를 CPU 레지스터로 복구함
5. 실행 재개 : 프로세스 B의 프로그램 카운터(PC)가 가리키는 지점부터 명령어를 다시 실행

### 컨텍스트 스위칭은 언제 발생하는가?

- 멀티 태스킹 : 정해진 시간이 다 되어 다른 프로세스에게 CPU를 넘겨야 할 때
- 인터럽트 : 입출력 요청이나 하드웨어 이벤트가 발생하여 급하게 처리해야 할 때
- 사용자 전환 : 실행중인 프로세스가 입출력(I/O) 완료를 기다리며 스스로 대기 상태로 들어갈 때

### 오버헤드와 성능 저하

컨텍스트 스위칭은 필수적이지만, 시스템 입장에서는 '비용'

- 순수 오버헤드 : CPU가 실제로 사용자 코드를 실행하는 것이 아니라, 오직 정보를 저장하고 불러오는데 시간을 쓰기 때문
- 캐시 오염 : 새로운 프로세스가 실행되면 이전에 사용하던 CPU 캐시 데이터나 TLB정보들이 무의미 해지거나 새로 채워져야 함, 이 과정에서 Cache Miss가 발생하며 일시적으로 성능이 떨어질 수 있음

## 페이지 폴트에 대해 설명하세요

프로그램이 사용하려는 메모리 페이지가 현재 실제 메모리(RAM)에 올라와 있지 않을 때 발생하는 하드웨어 인터럽트(또는 예외)입니다. 마치 오류처럼 느껴질 수 있지만 가상 메모리 시스템이 부족한 RAM을 효율적으로 쓰기위해 아주 자연스럽게 수행하는 데이터 요청 과정

1. 페이지 폴트 발생 과정
   1. CPU의 요청 : CPU가 특정 가상 주소에 접근하려고 함
   2. MMU의 확인 : MMU가 페이지 테이블을 확인했는데, 해당 페이지의 Valid Bit(유효 비트)가 0임 (즉, RAM에 없음)
   3. 트랩 발생 : CPU는 즉시 하던일을 멈추고 운영체제(OS)에 "이 데이터가 RAM에 없으니 가져와줘"라고 신호를 보냄, 이게 페이지 폴트

2. 운영체제 해결 단계(OS의 뒷수습)
   페이지 폴트가 발생하면 OS는 다음과 같이 움직임
   1. 주소 검사 : 프로그램이 접근하려는 주소가 정말 유효한지 확인(잘못된 주소면 Segment Fault 크래시 발생)
   2. 빈 공간 찾기 : RAM에 비어있는 물리 공간(Frame)이 있는지 찾음
      - 만약 꽉 찼다? Clock이나 LRU같은 페이지 교체 알고리즘으로 누군가를 쫓아내고 자리 만듬
   3. 디스크 읽기 : 보조 기억장치(SSD/HDD)의 스왑영역에서 필요한 데이터를 RAM으로 복사해 옴(가장 시간이 오래걸리는 구간)
   4. 페이지 테이블 업데이트 : 데이터를 다 옮겼으면 페이지 테이블의 해당 항목에 실제 물리 주소를 적고, Valid Bit를 1로 바꿈
   5. 명령어 재실행 : 이제 준비가 끝났으므로 아까 멈춘 그 명령어를 다시 실행, 이번에는 RAM에 데이터가 있으므로 성공

3. 페이지 폴트가 성능에 미치는 영향
   페이지 폴트는 시스템 성능에 아주 치명적일 수 있다.
   - 디스크 I/O의 병목 : RAM 접근 속도와 SSD/HDD 접근 속도는 수천~수만배 차이남, 페이지 폴트가 자주 발생하면 CPU는 아무것도 못하고 디스크가 일을 끝내기를 기다려야 함
   - 스래싱(Thrashing) : 만약 메모리가 너무 부족해서 페이지를 가져오자마자 다른 페이지를 내보내야 하는 상황이 반복되면, 시스템은 실제 작업보다 페이지 교체에만 모든 CPU자원을 소모하게 됨, 게임이 갑자기 심하게 버벅거리거나 멈추는 주범 중 하나

### 언리얼 개발자라면

- 에셋 로딩 전략 : 한꺼번에 너무 큰 에셋을 메모리에 올리려고 하거나, 메모리 관리가 안되어 RAM이 부족해지면 이때 발생하는 페이지 폴트때문에 게임에 프레임 드랍(Stuttering)이 생김
- 데이터 친화적 설계 : 데이터들이 메모리상에서 여기저기 흩어져 있으면(불연속적), 새로운 페이지를 계속 불러와야 하므로 페이지 폴트 확률이 높아짐, 반대로 TArray처럼 데이터를 연속적으로 배치하면 한 페이지 안에 많은 데이터가 들어있어 폴트를 줄일 수 있음

## CPU 캐시 계층, 캐시 히트/미스에 대해 설명해주세요.

CPU는 초당 수십억번의 연산을 수행할 정도로 빠르지만 RAM은 그 속도를 따라가지 못함, 이 속도 차이를 극복하기 위해 CPU내부에 두는 아주 빠르고 작은 메모리가 바로 캐시(Cache)

### 캐시 계층 구조

캐시는 속도와 용량의 트레이드오프에 따라 여러 단계로 나뉨

1. L1 캐시
   - 특징 : CPU 코어 바로 옆에 붙어 있는 가장 빠른 캐시
   - 구조 : 보통 L1i(Instruction - 명령어)와 L1d(Data - 데이터)로 분리되어 있음, 명령어를 읽는 동안 데이터를 동시에 처리하기 위함
   - 용량 : 수십 KB정도로 매우 작음

2. L2 캐시
   - 특징 : L1보다는 느리지만 용량이 조금 더 큼
   - 구조 : 현대 CPU에서는 보통 각 코어마다 독립적으로 할당되는 경우가 많음
   - 용량 : 수십 KB ~ 수 MB정도

3. L3 캐시
   - 여러 개의 코어가 공유(Shared)해서 사용하는 캐시
   - 구조 : 코어 외부, 하지만 CPU칩 내부에 위치하며 RAM으로 가기 전 마지막 보루 역할을 함
   - 용량 : 수십 MB이상으로 캐시 중 가장 큼

### 캐시 히트와 미스

- 캐시 히트
  CPU가 필요한 데이터가 캐시에 이미 존재하는 상태
  RAM까지 갈 필요없이 즉시 데이터를 가져올 수 있어 연산 속도가 비약적으로 빨라짐

- 캐시 미스
  필요한 데이터가 캐시에 없어 아래 단계 캐시나 메인메모리에서 데이터를 가져와야 하는 상태, 캐시 미스가 발생하면 L1 → L2 → L3 → RAM 순서로 계층을 내려가며 데이터를 탐색, 데이터를 가져오는 동안 CPU는 아무것도 못하고 기다려야 하므로 성능 저하(Stall)가 발생

### 캐시가 효율적인 이유

참조 지역성때문

- 시간 지역성 : 최근에 접근했던 데이터에 다시 접근할 가능성이 높다는 원리 (반복문의 변수)
- 공간 지역성 : 접근한 데이터 근처에 있는 데이터에 접근할 가능성이 높다는 원리(배열의 연속된 요소)

!!! 배열이 연결리스트보다 성능이 좋은 이유중 하나가 바로 이 공간 지역성 덕분, 배열은 메모리에 다닥다닥 붙어있어 한 번 읽을 때 주변 데이터가 캐시에 한꺼번에 올라가기(Cache Line) 때문

## 32비트 OS vs 64비트 OS 차이

가장 결정적인 차이는 한 번에 처리할 수 있는 데이터의 양과 메모리 주소의 길이

1. 주소 길이의 차이 (RAM의 인식 한계)
   - 32비트 OS : 주소 번호표가 32(2^32)까지만 있음. 계산하면 약 42억개, 즉 최대 4GM RAM까지만 인식 가능
   - 64비트 OS : 주소 번호표가 64(2^64). 이론상 16엑사바이트(EB)라는 어마어마한 양의 RAM을 인식할 수 있음, (사실상 현재기술로는 한계가 없다)

2. 연산 능력과 데이터 처리량
   단순히 메모리 주소만 길어진게 아니라 데이터를 실어 나르는 도로의 폭 자체가 넓어짐
   - 레지스터 크기 : CPU내부의 임시 저장소인 `레지스터`의 크기가 2배로 커짐 32비트 CPU가 한 번에 4바이트를 처리할 때, 64비트 CPU는 8바이트를 한 번에 처리
   - 복잡한 연산 : 수치 계산이 많은 게임 엔진이나 고해상도 영상 편집 프로그램에서 64비트는 훨씬 더 정밀하고 빠른 연산을 가능하게 함

3. 소프트웨어 호환성 (하위 호환)
   - 64비트 OS : 64비트 전용 프로그램은 당연히 잘 돌아가고, 32비트 프로그램도 에뮬레이션을 통해 구동할 수 있음
   - 32비트 OS : 64비트로 설계된 프로그램을 절대 실행할 수 없음. 큰상자를 작은 구멍에 넣는 격

## 컴퓨터 메모리 계층 구조

1. 레지스터
   위치 : CPU내부
   특징 : CPU가 현재 직접 연산중인 데이터를 아주 잠깐 저장하는 공간
2. 캐시 메모리
   위치 : CPU칩 내부 또는 근처
   특징 : 자주 사용하는 데이터를 미리 담아두어 RAM까지 가는 횟수를 줄임
3. 메인 메모리
   위치 : 메인보드 슬롯
   특징 : 실행중인 프로그램 코드와 데이터가 올라와 있는 곳. CPU가 직접 접근할 수 있는 마지막 단계의 메모리. DRAM(Dynamic RAM)을 사용하며, 전원이 꺼지면 데이터가 사라짐
4. 보조 기억장치
   위치 : 메인보드 연결
   특징 : 데이터를 영구적으로 저장, 얘만 비휘발성, CPU가 직접접근 불가

### 왜 이렇게 복잡하게 나눔?

CPU가 데이터를 처리하는 속도와 데이터를 저장소에서 가져오는 속도 사이의 거대한 간극을 메우기 위해 여러 단계의 징검다리를 두었다고 생각하자

## TCP vs UDP 차이

### TCP

- 연결 지향형(1:1통신)
- 신뢰성 높음
- 순서 보장
- 속도 비교적 느림(제어 로직 때문)
- 특징 : 흐름 제어, 혼잡 제어 제공

3-way handshake라는 통신 전 확인 과정을 거쳐야 함
ACK/NACK : 잘받았는지 확인을 해주어야 다음 데이터를 보냄
흐름 제어 : 받는 쪽의 속도가 느리면 보내는속도를 조절해 서버가 터지는것을 방지함
주 사용처 : 웹서핑(HTTP/HTTPS), 파일전송(FTP), 이메일(SMTP) 등 데이터가 단 1바이트도 틀리면 안되는 곳

### UDP

- 비연결형(1:1, 1:N, N:N가능)
- 신뢰성 낮음(데이터 유실 가능성 있음)
- 순서 보장안함(보낸 순서와 다를 수 있음)
- 속도 매우 빠름(오버헤드 적음)
- 특징 : 단순한 헤더, 던지고 끝
  주 사용처 : 실시간 스트리밍, 온라인 게임(캐릭터 위치 정보), VoIP(인터넷 전화)

### 게임 개발에서의 관점

캐릭터 이동/회전은 UDP, 아이템 획득/채팅/결제등은 TCP로 유실없이 사용
Modern Trend :최근에는 UDP의 속도를 챙기면서도, 필요한 부분에서만 신뢰성을 얹은 사용자 정의 프로토콜(QUIC, Reliable UDP)를 직접 구현해서 쓰는 추세, 언리얼의 네트워킹 시스템도 기본적으로 UDP 위에서 신뢰성 옵션을 선택할 수 있게 설계되어 있음

## 오브젝트 풀링이 자주 사용되는 이유(메모리 관점)

게임개발 특히 성능 최적화가 필수적인 환경에서 '메모리 관리의 효율성'을 극대화 하기 위해 사용하는 기법

1. 메모리 단편화 방지
   메모리를 빈번하게 할당(new)하고 해제(delete)하면 메모리 공간에 구멍이 생김 이를 단편화라고 함
   - 문제 상황 : 총 여유 메모리가 100MB가 있더라도 작은 객체들이 여기저기 흩어져 할당되어 있으면 정작 큰 데이터(50MB 크기의 텍스트 파일)를 올릴 연속된 공간이 부족해 할당에 실패할 수 있음
   - 풀링의 해결책 : 풀링은 시작할 때 필요한 만큼의 메모리를 커다란 덩어리(Contiguous Block)로 미리 잡아둠. 그 안에서 객체를 활성화/비활성화 하기 때문에 메모리 공간이 지저분해지는 것을 원천 봉쇄

2. 할당 및 해제 오버헤드 감소
   컴퓨터 입장에서 메모리를 새로 할당하는 작업은 꽤 '비싼' 작업
   - 할당 과정 : new 키워드를 호출하면 OS는 메모리 맵을 뒤져서 적절한 크기의 빈 공간을 찾아야 함, 만약 적당한 곳이 없으면 메모리를 정리하거나 스왑하는 과정이 추가되어 시간이 더 걸림
   - 풀링의 해결책 : 이미 생성된 객체의 '상태 값'만 초기화해서 재사용. CPU 입장에서는 복잡한 메모리 탐색 과정 없이, 이미 알고 있는 주소의 값만 바꾸면 되므로 연산량이 획기적으로 줄어듬

3. CPU 캐시 히트율 향상
   - 공간 지역성 : 메모리 풀은 객체들을 메모장에 나란히 배치하는 경우가 많음 CPU가 풀에 있는 한 객체에 접근할 때 주변 객체들도 함께 L1/L2 캐시에 올라올 가능성이 큼
   - 효과 : 만약 객체들이 메모리 여기저기 파편화 되어있다면 매번 RAM까지 가서 데이터를 가져와야 하지만(Cache Miss), 풀링된 객체들은 캐시에 머무를 확률이 높아 연산 속도가 비약적으로 빨라짐

4. SpawnActor의 무거운 비용
   SpawnActor를 호출하는 것은 단순히 메모리를 할당하는 것보다 훨씬 복잡한 과정을 거칩니다.
   - 컴포넌트 초기화 : 액터에 붙은 컴포넌트를 생성하고 등록해야함
   - 물리 및 렌더링 등록 : 월드의 물리엔진(Chaos)에 콜리전 등록하고, 렌더링 스레드에 보낼 정보를 구성
   - 생명 주기 함수 호출 : `PostInitializeComponents`, `OnConstruction`, `BeginPlay` 수많은 초기화 함수가 순차적으로 실행

5. 언리얼 가비지 컬렉션의 특징
   주기적으로 월드의 모든 UObject를 훑는 방식을 사용
   - 스캔 부하 : `UObject`개수가 많아질수록 GC가 누가쓰레기인지 검사하는 시간이 길어짐
   - 히치현상 : 대량의 오브젝트를 Destroy하면 다음 GC사이클에서 처리할 양이 폭주하여 프레임이 뚝 끊기는 현상이 발생
   - 풀링의 이점 : 액터를 파괴하지 않고 재사용하면 월드의 `UObject`총개수가 일정하게 유지. 이는 GC스캔범위를 예측가능한 수준으로 묶어두어 급격한 성능 저하를 방지

## 언리얼에서 오브젝트 풀링이 자주 사용되는 이유

1. 메모리 단편화의 원천 봉쇄
   메모리를 빈번하게 할당(SpawnActor)하고 해제(Destroy)하면, 메모리 공간이 마치 '구멍 뚫린 치즈'처럼 변함
   - 문제 상황 : 파편화가 심해지면 전체 여유공간은 충분해도, 정작 큰 데이터(고해상도 텍스처, 대규모 배열 등)를 올릴 연속된 빈 공간이 없어 할당 실패나 성능저하가 발생함
   - 언리얼에서의 해결 : 풀링은 게임 시작 시 TArray등을 통해 거대한 메모리 블록을 한번에 미리 확보함. 그 안에서 객체의 활성화/비활성화만 반복하므로, 엔진의 할당자(FMalloc)가 메모리 맵을 지저분하게 해집어 놓는 일을 원천적으로 막음

2. 엔진 레벨의 무거운 오버헤드 회피
   언리얼에서 객체를 새로 만드는 작업은 단순히 메모리할당 이상의 '행정 절차'가 필요
   - 할당 과정 : SpawnActor를 호출하면 엔진은 UClass정보를 탐색하고, 이름을 생성하여 등록하며, 속성값(CDO)을 복사하는 등 수많은 CPU 사이클을 소모함
   - 풀링의 해결 : 이미 모든 행정 절차가 끝난(Constructed)객체를 재사용, CPU입장에서는 복잡한 탐색과 등록 과정없이, 이미 알고 있는 메모리 주소의 '값(상태)'만 초기화하면 되므로 연산량이 획기적으로 줄어듬

3. CPU캐시 히트율 극대화
   - 공간 지역성 : 풀링을 통해 객체들을 메모리상에 나란히 배치하면 CPU가 한 객체를 처리할때 L1/L2캐시에 주변 객체들을 데이터까지 함께 올라올 확률이 비약적으로 높아짐
   - 성능 이득 : 데이터가 여기저기 흩어져 있어 매번 RAM까지 다녀와야 하는 캐시미스를 줄임으로써 수천개의 총알이나 파티클을 처리할 때 연산속도를 수 배 이상 높일 수 있음

4. 언리얼 특화 : 가비지 컬렉션(GC) 및 물리 엔진 부하 경감
   - GC 스파이크 방지: 대량의 액터를 Destroy하면 다음 GC 사이클에서 '도달 가능성 분석'을 할 때 처리할 쓰레기가 폭주하여 히치(Hitch, 툭 끊김) 현상이 발생합니다. 풀링은 객체 수를 일정하게 유지해 GC의 작업량을 안정화합니다.
   - 컴포넌트 및 물리 등록 생략: SpawnActor 시 발생하는 컴포넌트 부착, 물리 엔진(Chaos) 콜리전 등록, 렌더링 프록시 생성 과정을 생략할 수 있습니다. 단순히 SetActorHiddenInGame(false)와 SetActorEnableCollision(true)만으로 훨씬 가볍게 객체를 월드에 등장시킵니다.

# 궁금한거

## TLB

- Translation Lookaside Buffer
- 가상 주소를 물리주소로 바꿔주는 주소 변환 전용 캐시 메모리
  - 프로그램은 실제 메모리 주소가 아닌 '가상 주소'를 사용함 -> 실제 메모리 주소로 바꾸려면 메모리에 있는 '페이지 테이블'을 뒤져야 하는데 메모리 접근은 CPU입장에서 너무 느림
- 역할 : 자주 사용하는 주소 변환 정보를 CPU바로옆(MMU 내부)에 저장해 둠, TLB Hit, TLB Miss등이 있음

### MMU

Memory Management Unit, 가상 주소를 실주소로 바꿔주는 하드웨어 길잡이, CPU와 메모리사이에 위치한 하드웨어 장치 (CPU칩안에 내장되어있음)

- 주요 임무 : CPU는 프로그램이 사용하는 가상주소를 보냄, 하지만 실제 데이터는 RAM의 물리주소에 있음, MMU는 이 사이에서 주소를 번역해주는 역할을 함
- 보호 기능 : 특정 프로세스가 허용되지 않은 메모리 영역(커널 영역이나 다른 프로그램의 메모리)에 접근하려고 하면 이를 차단하여 시스템을 보호

- PageTable : 메모리 주소 변환 장부, MMU가 주소를 번역할때 참고하는 장부라고 보면됨, 이게 양이 좀 있어서 RAM에 존재함

## 메모리/캐시 교체 알고리즘

- Least Recently Used(LRU) : 메모리 관리나 캐시에서 어떤 데이터를 쫓아낼지 결정할때 쓰는 알고리즘
- FIFO
- LFU
- NUR(Not Used Recently) : 참조비트, 변조 비트를 두어 최근에 사용/수정 안 된것을 찾음, LRU와 비슷하지만 훨씬 적은 비용으로 효율적인 교체가 가능함, Clock 알고리즘이라고도 함(리눅스, 윈도우에서 많이 씀)

## 스케줄링 알고리즘

- RR : Round Robin 돌아가며 똑같은 시간만큼 기회를 줌
- Multi-Level Feedback Queue (MLFQ) : 현대 OS가 주로 쓰는 방식, 여러개의 줄(Queue)를 세워놓고, 우선순위가 높거나 응답이 중요한 작업을 먼저 처리하며 상황에 따라 우선순위를 동적으로 조절

## CPU의 스레드

물리 코어 vs 논리 스레드(하이퍼스레딩)

- 코어 : 실제로 계산을 수행하는 '일꾼의 몸'
- 하드웨어 스레드 : 코어라는 일꾼이 가진 '작업대', '손'
  - 하이퍼 스레딩기술을 쓰면 일꾼(코어)은 한 명인데 작업대(스레드)를 두개 둠
  - 한쪽 작업대에서 재료가 오기를 기다리는 동안(메모리 대기 시간), 일꾼은 놀지 않고 다른 작업대에 있는 일을 처리
  - 결과적으로 CPU입장에서는 일꾼이 두 명인것처럼 보임

- 프로세스-스레드(소프트웨어적) : 운영체제가 관리하는 작업의 단위
- 코어-스레드(하드웨어적) : CPU 하드웨어가 실제로 명령어를 처리하는 단위

## 게임 엔진의 주요 스레드와 역할

- 게임 스레드 : 게임의 두뇌, 사용자 입력처리, 캐릭터 상태 업데이트(Tick), 게임 로직 실행, GAS처리등이 여기서 일어남
- 렌더링 스레드 : 게임 스레드에서 계산된 결과(위치, 애니메이션 등)를 넘겨받아 GPU까 이해할 수 있는 명령어로 변환, 어떤 물체를 먼저 그릴지, 그림자는 어떻게 할지 계산하는 '감독' 역할을 함
- RHI 스레드 (Render Hardware Interface) : 렌더링 스레드의 명령을 실제로 DirectX, Vulkan 같은 그래픽 API에 전달하는 최종 통로
- 오디오 스레드 : 사운드 믹싱, 공간 오디오 효과
- 워커 스레드 : 물리 연산, 애니메이션 블루프린트 업데이트, 비동기 에셋 로딩 등 "누가해도 상관없는 병렬작업"들을 나누어 가짐

## 보조기억장치의 스왑영역 / 가상 메모리

스왑영역은 실제 메모리(RAM) 공간이 부족할 때를 대비하여, 운영체제가 하드디스크나 SSD의 일부 공간을 마치 RAM인 것처럼 빌려 쓰는 구역을 말함, 윈도우에서는 가상메모리, 페이지파일이라고 부름

- 작동 방식 : RAM에 자리가 없으면 OS는 현재 당장 쓰지 않는 데이터(비활성 페이지)를 이 스왑영역으로 옮기고, 그 빈자리에 급하게 필요한 데이터를 올림. 나중에 쫓겨났던 데이터가 다시 필요해지면 다시 RAM으로 가져옴
- 성능 : 아무리 SSD가 빨라도 RAM보다는 수십배 느림. 따라서 시스템이 스왑 영역을 빈번하게 사용하기 시작하면 전체적인 속도가 급격히 떨어지는 현상을 겪게 된다.

## 스터터링과 히치

Stuttering vs Hitching
스터터링은 화면이 미세하게 떨리거나 지직거리는 느낌, 프레임이 일정하지 않고 미세하게 튀는 현상
히치는 진짜 그냥 멈추고 다시 돌아오는 현상
