# 언리얼 5의 세이브 시스템

USaveGame클래스를 핵심으로 하는 데이터 저장/불러오기 프레임워크. 복잡한 파일I/O를 직접 다루지 않아도 되며, 플랫폼에 상관없이 일관된 방식으로 동작

- USaveGame클래스 : 저장할 데이터를 담는 컨테이너 역할의 UObject기반 클래스. 저장하고 싶은 변수에 UPROPERTY()를 붙이면 자동으로 직렬화
- UGameplayStatics : 실제 저장/불러오기를 수행하는 유틸리티 함수를 제공하는 클래스
- 슬롯 : 각 저장 파일을 식별하는 문자열 이름 + 유저 인덱스(int32)조합. 슬롯을 여러개 만들면 여러개의 세이브파일을 독립적으로 관리할 수 있음
- UPROPERTY(SaveGame) : 이 변수를 세이브 시스템에서 직렬화(저장/불러오기) 대상으로 포함시켜라

### 저장 파일 위치

- 에디터 모드에서 .sav파일은 Saved\SaveGames\{슬롯 이름}\sav 에 저장
- 실제 빌드에서는 AppData\Local\{프로젝트명}\Saved\SaveGames에 저장

### 주요 함수

- CreateSaveGameObject(TSubclassOf<USaveGame>) : SaveGame 객체 인스턴스 생성
- SaveGameToSlot(USaveGame\*, SlotName, UserIndex) : 슬롯에서 데이터 저장
- LoadGameFromSlot(SlotName, UserIndex) : 슬롯에서 데이터 불러오기
- DoesSaveGameExist(SlotName, UserIndex) : 슬롯이 이미 존재하는지 확인
- DeleteGameInSlot(SlotName, UserIndex) : 슬롯 삭제

우리는 멀티라 슬롯 필요하지 않을듯

### 저장 방식 및 특징

- 바이너리 직렬화 : 기본 저장 방식은 바이너리 포맷이며, 가볍고 빠름
- JSON 저장도 가능 : FJsonObjectConvertor 등을 활용하면 JSON형태로도 저장할 수 있어 디버깅에 유리
- 플랫폼 독립적 : PC, 콘솔, 모바일 등에서 동일한 코드로 동작, 내부적으로 플랫폼에 맞는 최적의 저장 경로를 자동 선택
- 암호화 가능 : SaveGameToMemory() / LoadGameFromMemory()로 바이너리 배열을 얻은 뒤 암호화 알고리즘을 적용하면 세이브 파일 변조를 어느정도 방지 할 수 있음

### GameInstance와의 연동

세이브 데이터는 레벨이 바뀌어도 유지되어야 하므로 GameInstance에서 세이브/로드 함수를 관리하는 패턴이 일반적.
