# 멀티플레이 게임 개발

## NetRole

### Authority, Proxy

- 서버에 스폰된 액터가 가진 NetRole 속성 값은 언제나 Authority
- 게임에 중대한 영향을 끼치는 로직은 NetRole이 Authority일때 수행해야한다. (서버이므로)
- NetRole이 Authority인 액터가 클라이언트로 복제되었을때, 클라이언트에 복제된 액터의 NetRole 속성값은 Proxy이다.

### NetRole 종류

- None
  - 보통의 경우 리플리케이션 되지 않는 액터
  - LocalRole은 Authority고 RemoteRole이 None이라면 서버에서 스폰되고, 클라이언트로 리플리케이션 되지않는 액터라고 보면 된다.

- Authority
  - 게임에 중대한 영향을 끼칠 수 있는 권한을 가진 액터
  - 서버에서 스폰된 액터가 LocalRole으로 Authority를 가질 수 있다.
  - ex) GameMode

- Autonomous Proxy
  - Authority 액터의 복제본
  - 서버로부터 데이터를 수신 받아서 동기화도 되면서, 서버로 송신도 가능
  - ex) PlayerController

- Simulated Proxy
  - Authority 액터의 복제본
  - 서버로부터 수신받아서 동기화 당하기만 한다.
  - ex) PlayerCharacter

### 각 액터에 따른 특징

- 게임 모드는 HasAuthority() 함수를 호출할 필요가 없다.
- UI관련 로직은 클라이언트에서만 수행한다.

## RPC

- Remote Procedure Call
- 호출하는 PC와 실행하는 PC가 달라도 되게끔 해주는 통신 기법
- 언리얼에서는 주로 사운드, 파티클등 게임에 큰 영향을 끼치지 않는 일시적인 효과들에 사용하는 편
- 호출하는곳과 실행되는 곳이 다르다.

### Call

- 정적인 의미
- 컴파일 타임에 어떤 함수인지, 호출하는곳과 실행하는 곳이 정해져 있어야 한다.
- 일반적인 전역함수와 멤버 함수가 이에 해당한다.
- 직접적으로 함수를 호출하고 실행해야 한다.

### Invoke

- 동적인 의미를 가짐
- 런타임에 어떤 함수인지 호출하는 곳과 실행하는 곳이 어딘지 정해짐
- 함수 포인터, 동적 바인딩, RPC 같은 개념들이 이에 해당
- 간접적으로 함수를 호출하고 실행해야 한다.
- RPC를 Invoke라고 표현한다.
- 호출하는거다 이런식으로 생각하자

### ActorOnwerShip

- 네트워크 멀티플레이가 적용되려면 액터는 서버에서 스폰되고 bReplicated가 true여야 한다.
- 서버에서 스폰된 후 SetOwner(PlayerController) 함수를 호출하여 Client-Owned Actor가 된다.
- PlayerController가 LocalPlayerController와 같아면 Client-Owned Actor고, 아니면 Owned by different Actor
- SetOwner함수 호출이 없으면 Server-Owned Actor

### Multicast, Server, Client 키워드

- Netmulticast : 서버를 포함한 모든 클라이언트에서 해당 RPC를 실행시켜주세요
- Server : 서버에서 해당 RPC를 실행시켜주세요
- Client : 클라이언트에서 해당 RPC를 실행시켜주세요

### RPC가 자주쓰이는 케이스

- RPC가 클라이언트에서 호출되고, 서버에서 실행되어야 하는 경우에는 Server키워드를 사용해야 한다.
  - 해당 ClientConnection이 소유하고 있는 액터에서 RPC가 호출되어야 한다. (캐릭터의 무기라던지)

- RPC가 서버와 모든 클라이언트에서 실행되어야 하는 경우에는 NetMulticast 키워드를 사용해야 한다.
  - 서버에서 호출해야함
  - 부하가 굉장히 심하므로 자주 사용하는것은 권장되지 않는다.

- RPC가 서버에서 호출되고 클라이언트에서 실행되어야 하는 경우에는 Client키워드를 사용해야 한다.
  - 서버에서 호출해야 한다.

### WithValidation

- UFUNCTION() 매크로와 함께 사용되는 키워드
- 서버에서 실행되는 RPC의 경우에 해당 키워드를 작성하는것이 권장된다.
- 서버 실행 로직은 무조건적으로 신뢰되기 때문에, 위변조를 막기위한 방어막 역할을 한다.
- 이 키워드가 붙은 RPC를 구현할때는 \_Implementation()함수와 \_Validation()함수로 나누어 진다.
  - Validation()함수는 해당 RPC가 서버에서 실제로 실행할지 말지 결정

### Reliable

- UFUNCTION 매크로와 함께 사용
- RPC는 기본적으로 Unreliable 키워드를 기준으로 동작
  - 게임에 영향이 없는 이펙트 사운드 등은 Unreliable
  - 충돌, 데미지, 스폰처럼 게임에 큰 영향이 있는 키워드는 Reliable

## Property Replication

- 리플리케이션 : 생성된 액터의 정보를 네트워크 내 다른 클라이언트에게 복제하는 작업을 레플리케이션이라고 함
- RPC, Property Replication 두가지 방법
- 항상 Authority에서 proxy로만 가능
- 우리가 원하는 속성만을 다른 클라이언트에게 복제하는것을 프로퍼티 레플리케이션이라고 함

### 설정 방법

- bReplicates를 true로
- UPROPERTY에 REplicated 키워드 추가
- GetLifetimeReplicatedProps() 함수에 네트워크로 복제할 속성 추가
  - DOREPLIFETIME 매크로를 사용해 복제할 속성 명시

# 코드카타

## 골드바흐 파티션

```c++
// https://www.acmicpc.net/problem/17103

#include <iostream>
#include <vector>

using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(NULL);

    int T;
    cin >> T;
    int maxValue = 1000001;
    vector<bool> isPrime(maxValue, true);
    isPrime[0] = isPrime[1] = false;
    for (int i = 2; (long long) i * i <= maxValue; i++)
    {
        if (isPrime[i])
        {
            for (int j = i * i; j <= maxValue; j += i)
            {
                isPrime[j] = false;
            }
        }
    }
    vector<int> Prime;
    for (int i = 2; i <= maxValue; i++)
        if (isPrime[i]) Prime.push_back(i);


    for (int t = 0; t < T; t++)
    {
        int N;
        cin >> N;

        int start = 0;
        int end = Prime.size() - 1;
        int count = 0;

        while (end >= 0 && Prime[end] > N) end--;

        while (start <= end)
        {
            int sumV = Prime[start] + Prime[end];

            if (sumV == N)
            {
                count++;
                start++;
                end--;
            } else if (sumV < N)
            {
                start++;
            } else
            {
                end--;
            }
        }

        cout << count << "\n";
    }

    return 0;
}
```

- 오랜만에 접한 에라토스테네스체 문제
- 정렬되어있는 데이터이기 때문에 인덱스를 양쪽에서 트래킹하면서 문제를 풀었다.
