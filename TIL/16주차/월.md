# 멀티플레이어 게임 개발

## ETextCommit

```c++
void OnChatInputTextCommitted(const FText& Text, ETextCommit::Type CommitMethod);
```

- CXChatInput의 헤더에서 이러한 메소드를 정의했다.
- 여기서 ETextCommit은 어떤식으로 입력을 마쳤는가를 알려주는 열거형 정보
  - 즉, 타이핑 후에 엔터를 쳤는지, 다른곳을 클릭해서 나갔는지등을 알려준다.

### ETextCommit::Type의 주요값

- OnEnter : 사용자가 엔터키를 눌러서 입력을 마침 -> 메시지 전송
- OnUserMovedFocus : 포커스가 나간경우 -> 전송하지 않고 취소하거나 그대로 두거나
- OnCleared : 프로그램 코드에서 강제로 비움 -> 로직상 초기화 상황
- Default : 그외 기본상황 -> 거의 사용안됨

## 리플렉션 시스템에서 meta

```c++
UPROPERTY(meta = (BindWidget))
TObjectPtr<UEditableTextBox> EditableTextBox_ChatInput;
```

- WBP에서 만든 위젯 이름과 C++변수명이 일치하면 자동으로 이 포인터에 해당 위젯을 대입
- 컴파일 타임 체크 가능 : 만약 C++클래스를 상속받은 블루프린트에 해당이름의 위젯이 없다면 컴파일 에러를 띄운다. 일종의 안전 장치
- meta를 쓸때는 이름을 잘 확인하고 위젯에서 IsVariable을 꼭 체크하자

### 메타 키워드

- AllowPrivateAccess = "true" : private영역에 있는 변수를 블루프린트 에디터에서 볼 수 있게 허용, 수정도 가능
- ToolTip = "설명" : 에디터에서 해당 변수에 마우스 올렸을때 설명
- ClampMin = "0", ClampMax = "100" : 에디터 슬라이드로 조절할 수 있는 값의 범위 지정
- BindWidgetOptional : BindWidget과 같지만 블루프린트 해당 위젯이 없어도 에러를 내지 않고 일단 작동

## 서버 종류

### P2P Server

- Peer to Peer Server
- 한 컴퓨터가 서버이자 클라이언트 역할을 한다.
- 다크소울, 토렌트

### Listen 서버

- 클라리언트이자 서버인 방장(Host)가 있고, 나머지는 모두 클라이언트 역할
- P2P의 일종이다 (마인크래프트, 어몽어스)

### Dedicated Server

- 서버를 담당하는 컴퓨터가 따로 있는 경우, 서버-클라이언트 구조
- 서버만 담당하는 애가 있다.
- 배틀그라운드

## Dedicated Server의 실행 흐름도

- PIE를 했거나, Server.exe를 실행해서 서버 프로세스가 실행
- 실행할 때 Open {Level이름}?Listen 명령어가 인자로 전달 해당 레벨을 열어둠
- 이때 Socket이 생성되며 다른 PC가 접속 가능하게끔 한다.
- Listen 명령어가 없다면 싱글플레이
- Level에는 월드 세팅이 있고 거기에서 GameMode, GameState정보가 있다. 이를 통해 생성
- 중요한건 GameMode 액터는 서버 컴퓨터에만 존재
  - GameMode와 Server를 동일 시 해도 된다.

- 클라이언트는 서버의 IP주소와 포트번호로 접속 시도, 서버는 접속 시도하는 클라이언트에게 Level정보를 넘김
- 클라이언트도 해당 Level을 열고, Level을 여는데 성공했다고 데디케이트 서버에게 알림
- Level을 여는데 성공한 클라이언트 전용 PlayerState, PlayerController, PlayerCharacter가 서버에 생성된다.
- 이것이 다시 Client1에 복제, GameState도 복제 (서버에서 클라이언트로)
- 다른 클라이언트가 들어와도 마찬가지로 반복
- 클라이언트 - 클라이언트는 통신불가 무조건 서버를 통해야 함

## NetMode

- 해당 게임 프로세스가 네트워크 상에서 어떤 역할을 하고 있는지를 의미
- 데미지를 입어서 HP가 줄어드는 로직을 어느 컴퓨터에서 처리해야할지 고민해보자
  - 클라이언트 컴퓨터에서 한다면?
  - 해킹에 취약 해짐, 핵이 만들어질때 서버와 클라이언트가 주고받는 패킷을 분석함
  - 패킷을 찾아서 원하는 값을 서버로 보내면 끝
  - 이러한 해킹 방지를 위해서 데미지 처리 로직은 서버에서 처리해야함

### 종류

- StandAlone
  - 게임이 원격 클라이언트의 연결을 허용하지 않는 서버로 실행중
  - 싱글플레이, 로컬플레이 게임에 사용
  - 서버측 로직과 클라이언트 로직 모두 실행

- Client
  - 게임이 네트워크 멀티플레이 세션에서 서버에 연결된 클라이언트로 실행
  - 서버측 로직을 실행하지 않음, 서버로부터 복제된 Proxy를 보여주는 역할

- Listen Server
  - 게임이 네트워크 멀티플레이어 세션을 호스팅하는 서버로 실행 중
  - 원격 클라이언트의 연결을 수락하고 로컬 플레이어를 서버에 직접 배치, 즉 서버도 직접 게임에 참여
  - 이 모드는 캐주얼 협동 및 경쟁 멀티플레이어에 자주 사용

- Dedicated Server
  - 게임이 네트워크 멀티플레이 세션을 호스팅하는 서버로 실행
  - 원격 클라이언트의 연결을 허용하지만 로컬플레이어가 없음
  - 이 모드는 지속적이고 안전한 대규모 멀티플레이어가 필요한 게임에 주로 사용

### NetMode에 따른 액터 위치

- 서버에만 존재하는 액터 : 게임 모드
- 서버와 모든 클라이언트에 존재하는 액터 : 배경 액터와 폰
- 서버와 (Owning) 클라이언트에만 존재하는 액터 : 플레이어 컨트롤러
- 클라이언트에만 존재하는 언리얼 오브젝트 : UI

## NetConnection

- 다른 PC와의 연결이 발생하면 UNetConnection객체도 생성
- 서버에 클라이언트가 접속하면 서버에는 ClientConnection 객체가 추가
- 반대로 클라이언트에는 ServerConnection 객체가 생성
- 두 PC는 UNetConnection객체를 통해 통신 (클라이언트는 하나의 ServerConnection을 가지고, 서버는 여러개의 ClientConnection을 가짐)
- UNetDriver는 생성된 UNetConnection을 관리한다.
  - 서버에서는 접속된 클라이언트 수만큼의 UNetConnection을 관리하고
  - 클라이언트에서는 서버와의 ServerConnection 하나만을 관리한다.

## Ownership

- 하나의 ClientConnection은 하나의 PlayerController를 소유함
- 즉 PlayerController의 Owning 객체는 ClientConnection이다.
- PlayerController가 핑의하는 폰의 Owner객체는 해당 PlayerController로 설정
- 폰에 무기 액터가 생성되고, 무기 액터의 Owner 속성에 해당 폰을 설정할 수도 있다.
- ClientConnection에서부터 무기 액터에 이르는 소유 관계를 패밀리라고 부름
- 소유관계속에 있는 액터가 본인의 Owning Connection을 얻으려면 AActor::GetNetConnection()함수를 호출하면 된다.
- 후에 RPC와 Property Replication과 관계가 있다.

## NetDriver

- 언리얼 네트워크 통신에서 로우레벨 동작들을 관리하는 클래스
- 싱글플레이에서는 UNetDriver객체가 생성되지 않음
- 멀티플레이에서만 UWorld::Listen() 함수를 통해 UNetDriver 객체가 생성된다.
- 멀티플레이에 참여하는 각 PC마다 UNetDriver객체가 생성된다.

# 언리얼엔진, UCLASS와 게임의 구성

## GameInstance

- 생명주기가 게임실행 ~ 게임종료
- 레벨전환같은것에 영향받지 않음
- 네트워크 관점에서는 서버, 클라이언트 각각 가지고 있음, 복제 대상이 아니다.

### 언제 쓰는가?

- 레벨을 넘어 유지되어야 하는 데이터
- 로그인/계정정보, UI플로우 상태
- 매치메이킹, 세션 연결 정보(온라인 서브시스템 연동)
- 로딩화면/전역 UI 매니저같은 "전역 서비스"

### GameInstanceSubsystem

- 책임이 너무 커져서 분리해서 사용하는게 좋다.

### 많이 하는 실수

- 공유해야 하는 데이터를 넣으면 안됨
- 레벨 전환시 삭제되어야 하는 객체를 들고있으면 안됨

## GameState

- 공유상태를 담는 클래스
- 게임모드가 규칙을 집행, GameState는 그 결과로 나온 현재 상태를 모든 클라이언트에 전파하는 역할

## PlayerState

- 플레이어 1명의 공유상태를 담은 클래스
- 다른사람도 봐야하는 플레이어 정보 (킬/데스/점수/닉네임/핑/팀 같은것들)

## HUD

- 전통적인 UI 진입점 (언리얼5에서는 PlayerController에다가 UI진입점을 두는 경우도 있다.)
- 화면에 그리는 UI를 담당
- 일반적으로 로컬에 존재, 서버나 다른 클라이언트에 내 HUD가 복제되는 개념이 아님

## PlayerController

- 플레이어의 입력 처리와 소유 관점의 중심
- 언리얼 5에서 인핸스드서브시스템을 이용해서 입력 관리를 했던것처럼, UI도 컨트롤의 주체이기 때문에 이쪽으로 오는 경향도 있다. (특히 모바일은 더더욱 UI가 조작을 담당하므로)

## 상속

- Is-a 상속
  - AActor - APawn - ACharacter
  - ACharacter is Pawn=
  - APawn\* Pawn = Character : 이게 가능
  - 인간은 포유류다

- Has-a 상속
  - AActor <-> UComponent
  - 누구를 가지고 있다.
  - 부품처럼 사용
  - 인간은 심장을 가지고 있다.

## Component

- Actor에 붙이는 부품
- UActorComponent : 모든 액터의 어머니, 로직 중심(Transform 없음)
- USceneComponent : Transform있음 (Attach 가능)
- UPrimitiveComponent : 렌더/충돌/물리까지 포함하는 실체(StaticMeshComponent 등)
- 재사용성과 분리를 높여주지만 남용하면 의존성 관리가 어려워질 수 있다.

## BeginPlay, EndPlay

- BeginPlay : Actor/Component가 실제로 게임 월드에서 플레이를 시작할때 한 번 호출되는 이벤트
  - 월드 접근, 다른 Actor찾기, 타이머/입력 연결등은 여기서
- EndPlay : Actor가 월드에서 제거되거나 플레이가 끝날 때 호출
  - 종료 이유가 존재하며(레벨 전환, Destroy, 게임 종료 등), `EndPlayReason`으로 구분

### 실무 포인트

- 타이머 해제, 델리게이트 언바인딩, 외부 참조 정리같은 "수명 정리"는 EndPlay가 안전한 편
- C++로 소멸자는 언리얼 오브젝트 수명 모델(GC)상 기대한 타이밍에 호출되지 않을 수 있으므로, 정리는 EndPlay/BeginDestroy 계열을 우선시

## 생성자와 소멸자, CDO

- 언리얼의 `UObejct` 계열은 엔진이 클래스 기본값과 인스턴스 초기화를 위해 생성자를 다양한 상황에서 호출
- 기본 컴포넌트 생성, 기본값 설정, 리플렉션 기반 프로퍼티 기본 세팅과의 정합 유지
- Class Default Object : UCLASS마다 1개 존재하는 클래스 기본 인스턴스, 리플렉션 시스템에 등록된 정보를 바탕으로 엔진이 생성한 인스턴스
- 언리얼은 CDO를 복제해서 사용 이걸 프로토타입 패턴이라고 함

### CDO를 사용하는 이유

- 메모리 절약
  - 캐릭터가 천마리 있어도 기본 이동속도는 다 같을 수 있음, 이때 CDO를 참조해서 가져오기 때문에 관리가 편하다.
  - 인스턴스 생성 가속 : 새로운 캐릭터를 만들때 엔진은 처음부터 로직을 계산하지 않고 CDO에서 복사해서 사용
  - 에디터와의 연결 : 블루프린트 창에서 값을 바꾸면 엔진은 해당 클래스의 CDO값을 바꿈, 그러면 이후에 생성되는 모든 캐릭터는 바뀐 기본값을 들고 태어남

### CDO 생성 시점

- 엔진 초기화 및 모듈 로드 단계
- 엔진이 켜지거나 DLL(프로젝트 모듈)이 로드
- 클래스 등록, 리플렉션 시스템(UHT가 만든 코드들) 이 이 클래스가 존재함을 엔진에 알림
- 생성자 호출 : 앤진은 C++의 생성자를 딱 한 번 실행
- CDO 완성 : 이 생성자 호출을 통해 만들어진 객체가 바로 CDO

### CDO 생성 시점에서 하면 안되는 일

- CDO가 생성되는 시점에는 게임세상(World)가 존재하지 않음
- GetWorld()혹은 다른 액터를 찾으려고 하면 안됨, 런타임 데이터 기반 초기화(세이브 로드 등)
- CreateDefaultSubobject로 컴포넌트를 붙이거나, 변수에 숫자/에셋 경로같은 순수 데이터 기본값을 넣는 역할만
- 다른 액터 접근은 BeginPlay에서

## 조합 0의개수

- https://www.acmicpc.net/problem/2004

```c++
long long calculate(long long targetNumber, int p)
{
    long long count = 0;
    while (targetNumber >= p)
    {
        count += targetNumber / p;
        targetNumber /= p;
    }
    return count;
}

// 조합은 n! / m!(n-m)!이기도 하다.
long long countTwo = calculate(n, 2) - calculate(m, 2) - calculate(n-m, 2);
long long countFive = calculate(n, 5) - calculate(m, 5) - calculate(n-m, 5);
```

- 위 주석처럼 조합은 nCm일때 n!/m!(n-m)! 이다.
- calculate함수는 팩토리얼에 있는 모든 p의 개수를 구할 수 있다.
  - 첫시기에 targetNumber가 10이고 p가 2면 5가 나온다. 10아래숫자들중 2를 가진 수가 5개임 (2, 4, 6, 8, 10)
  - targetNumber를 p로 나눠줌
    - 그 이유는 4, 8처럼 2를 여러개 가지고 있는 수가 있다.
    - targetNumber는 5가되고 그 아래 숫자들로 부터 2의 개수를 센다 (2, 4) + 2
    - 이는 마치 2를 여러개 가지던 4와 8도 2로 나누어져서 같이 계산된것처럼 보임
    - 여기서 4는 또 2를 가지므로 2로 나누어서 계산하면 (2) + 1 되어서 calculate(10, 2)는 총 8이나옴
    - 이는 10팩토리얼에서 가지는 2의 개수와 동일하다.

```c++
}
```

- while (cin >> n) 으로 입력이 계속되는 문제를 풀 수 있다.
- 1로만 이루어진 자릿수가 몇백 몇천이 될 수도 있어서 나머지 연산으로 계속해서 current를 줄여주었다.
  - n으로 나눈 나머지가 0일때만 체크하면 되기때문에 저게 가능하다.
- 예를들어 7에 대해서 생각해보면
- 1일때 7로 나누면 나머지 1
- 1에 10을 곱하고 1을 붙이면 11 이때 7로 나누면 나머지가 4가 나온다.
- 이 4에 10을 곱하고 1을 붙이기때문에 41이 되고 이를 7로 나누면 나머지 6
- 61을 7로 나누면 5 -> 51을 나누면 2 -> 21을 나누면 나머지 0 그래서 6번째 자리수가 채택되는식
- 거대한 1로 이루어진 수가있고 이것을 n으로 계속해서 나누어준다고 생각하면 된다.
