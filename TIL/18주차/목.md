## Relevancy

** 연관성 있는 애들만 복제 **

### 왜 필요한가?

통신량이 너무 빡셈, 이 액터가 저 클라이언트랑 연관이있나? 를 판단해서 연관성 있는 액터만 복제

### Viewer, ViewTarget

Viewer는 클라이언트 커넥션이 소유하고 있는 PlayerController
ViewTarget은 그 playerController가 빙의한 Pawn
이게 연관성의 판단 기준이 된다. `이 액터가 내 캐릭터(Pawn)과 관련이 있는가?`

### 액터(AActor) 연관성 판단 기준

AActor 레벨에서 복제 여부를 결정하는 속성, 대충 여러가지들이 있음

### IsNetRelevantFor()

연관성 판단 함수, 오버라이드해서 커스텀 판단 로직을 만들 수 있다. 대부분의 경우는 Super::IsNetRelevantFor() 를 호출하면 자동으로 판단, 진짜 추가조건이 필요할때만 오버라이드 (예시에서는 로그 출력)

### 흐름

서버의 복제 루프 (매 NetUpdateFrequency 주기)

↓ 모든 Replicated 액터를 순회

↓ 각 클라이언트 커넥션에 대해 IsNetRelevantFor() 호출

↓ true → 복제함
false → 이 클라이언트에겐 복제 안 함 (대역폭 절약)

## NetPriority

** 대역폭이 한정된 상황에서 어떤 액터를 먼저 보낼까? 를 결정 **

### NetPrioirty

대역폭이 한정되어 있기에 NetPriority값이 높은 액터부터 우선 전송
순서를 정하는데만 사용

### 가중치

가중치가 있어서 실제 NetPriority값은 고정이 아님, GetNetPriority() 함수가 매 복제 주기마다 상황에 맞게 계산
예를들어 AActor의 GetNetPriority()함수를 보면 직전 패킷을 보내고 경과시간을 담은 Time변수가 있는데 얘가 커질수록 즉 오래기다렸을수록 우선순위가 높아진다. 그래서 낮은 우선순위의 액터도 무한정으로 굶지는 않는다.

포화상태도 있는데 이는 아무리 NetPriority값이 높아도 이번 주기에 복제못할수도있음(대역폭이 꽉차서) 그러면 다음 주기에 더 높은 우선순위를 가짐

## NetDormancy

** 복제는 하고싶은데 자주 바뀌지 않는 액터** 를 위한 최적화 기법

### NetDormancy

안바뀌는 액터는 재워라, 휴면 상태가 된 액터는 Property Replication도, RPC도 일절 동작하지 않음
플레이어 캐릭터처럼 매 프레임 값이 바뀌는 액터에는 오히려 깨웠다가 재웠다가 하면서 오버헤드가 발생한다.

### DORM_Initial

가장 실용적 패턴, 스폰시 바로 휴면상태, 값이 바뀔때만 복제, 복제하려면 명시적으로 깨워야함 FlushNetDormancy() 호출

### FlushNetDormancy()

** 잠깐 깨워서 복제하고 다시 재움 **

### Conditional Property Replication

DOREPLIFETIME_CONDITION() 조건을 달아서 복제 여부를 제어

## 챕터 5 최적화 도구

```
복제 대상 선정   →  IsNetRelevantFor()   (연관성 있는 클라이언트만)
복제 주기 조절   →  NetUpdateFrequency   (얼마나 자주)
복제 우선순위    →  NetPriority          (대역폭 부족 시 누가 먼저)
복제 완전 중단   →  NetDormancy          (안 바뀌는 액터는 아예 재우기)
복제 조건 제어   →  DOREPLIFETIME_CONDITION (누구에게 보낼지 세밀하게)
```

## 서버 RPC 규칙

```c++
UFUNCTION(Server, Reliable, WithValidation)
void ServerRPCSpawnLandMine();
```

- Server : 클라이언트가 호출 -> 서버에서 실행
- Reliable : 반드시 전달 보장(패킷 유실 시 재전송)
- WithValidation : 검증 함수 필요

RPC규칙에서 실제 ServerRPC함수에 \_Implementation() 붙이는것처럼 WithValidation을 쓰면 \_Validate() 써야됨 (검증 로직)

## 멀티캐스트 RPC

### RPC 선택 기준

** 무엇을 보여줄지 **

사망 UI를 띄워야 한다
-> ClientRPC, 내 사망 UI는 나의화면(OwningClient)에서만 띄우면 되기때문

지뢰 폭발 이펙트를 재생해야됨
-> Netmulticast RPC
폭발 이벤트는 모든 사람의 화면에서 보여야 하기때문에, 물론 데디케이트 서버는 화면이 없으니까 예외처리해도됨

BeginOverlap() 함수는 여러 PC에서 각각 호출되기때문에 누가 먼저 호출된다는 보장이 없어서 서버에서만 로직을 처리, 폭발같은 이펙트는 시각효과일뿐이기 때문에 Unreliable을 써서 네트워크 부하 줄이기

## RPC와 Property Replication

### 문제 1.

1. 지뢰가 폭발하고 다시 폭발하지 않도록 bIsExploded 변수를 만들어서 MulticastRPC에서 관리
2. 멀리서 지뢰를 터뜨리면 다른 클라이언트에 전파가 안되어서 다른 클라이언트가 오면 다시 또 터짐
3. NteCullDistanceSquared범위 밖에서 지뢰를 터뜨리면 Multicast RPC가 전달이 안되고, 나중에 범위로 들어왔을때 또 터짐

### 문제 2.

bAlwaysRelevant = true -> 항상 모든 클라에 연관성이 있다.
거리 문제는 해결되지만 늦게 들어오는 클라이언트가 있으면 폭발당시의 Multicast RPC를 못받았으므로 그 클라이언트 눈에는 지뢰가 안터진걸로 보임 다가가면 터짐
bIsExploded를 Replicated로 해결, RPC는 `이벤트`, Property Replication은 `상태`

### 문제3.

Property Replication적용시 폭발 안됨 -> 클라에서 직접 분기해서 서버에서는 상태 변경만 해주고, 클라에서 직접 이펙트 재생하는 식으로 해결

RPC와 Replication은 타이밍이 독립적이다. 즉, 서버에서 값 변경 후 RPC호출해도, 클라이언트에 값이 먼저 도착할 수도 있고 RPC가 먼저 도착할 수도 있다. 이 실행순서가 보장이 안되어서 의존관계를 만들면 안됨

| 상황                   | 올바른 선택                | 이유                           |
| ---------------------- | -------------------------- | ------------------------------ |
| 폭발 이펙트 재생       | Multicast RPC (Unreliable) | 일회성 이벤트, 놓쳐도 무방     |
| 폭발 여부 상태 저장    | UPROPERTY(Replicated)      | 상태는 항상 최신값 유지 필요   |
| 폭발 후 머테리얼 변경  | ReplicatedUsing + OnRep\_  | 복제 시점에 자동으로 로직 실행 |
| bool 복제 시 자료형    | uint8 bFlag : 1            | bool보다 복제/직렬화에 안전    |
| RPC + 복제 타이밍 의존 | ❌ 절대 금지               | 순서 보장 안 됨                |

## 애니메이션 동기화

AnimInstance가 캐릭터 애니메이션을 담당, 멀티플레이에서는 이게 복제되지 않음 대신 AnimInstance가 읽는 캐릭터의 이동 데이터(Velociy, bIsFalling 등)가 복제되면, 그 값을 AnimInstance가 읽어서 각 클라이언트에서 자동으로 올바른 애니메이션을 재생 (모든 Tv가 같은 채널을 보고있으면 TV자체가 복제되지 않더라고 화면은 똑같이 나온다.)

BP_PlayerCharacter > Details > Replicate Movement = true
Replicate Movement는 기본값이 true이고 이게 있으면 위치, 회전, Velocity 등이 자동으로 복제됨
NativeUpdateAnimation에서 이 복제된 Velocity를 읽으므로 별도 복제 코드 없이도 모든 클라이언트에서 동일한 애니메이션 변수를 갖게 됨

## 공격 동기화

TSet : 같은 액터를 여러번 히트할 수 있어서 중복을 허용하지 않는 Set을 사용

클라이언트에서 처리할 수 있는 로직은 최대한 클라이언트에서 처리해서 체감 반응성을 높이고, 데미지/상태 변경처럼 치트 가능성이 있는 로직만 서버에서 처리하는 것이 좋은 멀티플레이 설계

### 결론

- 공격처럼 상태가 바뀌는 로직은 서버에서 -> 클라에서 처리하면 핵생길수도
- 모든 사람들이 봐야하는 애니메이션은 Multicast RPC
- 상태는 Replicated
- 서버를 무조건 거치는 구조의 단점은 반응성인데 실무에서는 이를 해결하기위해 노력한다.
  - 클라에서 애니메이션 재생 (예측)
  - 서버에서 데미지/히트 판정만
  - \_Validate()에서 치트 방지 검증 로직 추가

## 컴포넌트 복제

1. 액터는 bReplicates = true 하나면 되지만 컴포넌트는 두가지가 필요하다.

```c++
// 생성자에서 복제 활성화
SetIsReplicatedByDefault(true);

// 속성에 ReplicatedUsing 선언 + DOREPLIFETIME 등록
UPROPERTY(ReplicatedUsing = OnRep_CurrentHP)
float CurrentHP;
```

2. 컴포넌트는 Net Role이없어서 Owner액터의 Role을 따라감
3. 모든 속성을 Replicated하면 낭비, 초기 이후 안바뀌면 COND_InitialOnly로 최초 1회만 복제

## 패키징

언리얼은 무엇을 패키징할지 Target.cs로 정의함 기존적으로 Editor타겟만 있어서 Server, Client를 만들어야 한다.

Editor : 에디터 실행용 (개발용, 에디터 기능 포함)
Client : 클라이언트 전용 (UI, 렌더링 있음, 서버 기능 제거)
Server : 서버 전용 (렌더링없음, 경량화, 서버로직만 포함)

패키징 하기전에 맵을 등록해줘야함 등록안되면 패키징 결과물에 포함 안됨

# 챕터 10

## 멀티플레이 기본 지식

### 서버의 종류

- P2P : 모든 참가자가 클라이언트이자 서버, 다크소울, 토렌트
- Listen Server : 방장이 클라이언트 + 서버 역할 동시에, 나머지는 클라이언트, 마인크래프트, 어몽어스
- Dedicated Server : 게임 로직 전담 서버PC분리, 모두 순수 클라이언트, 롤, 배그

### Dedicated Server 접속 흐름

1. 서버에서 Level열기 + Listen명령 실행
2. Socket생성 (외부 접속 가능상태)
3. 클라이언트가 Open [서버IP] 실행
4. 서버가 자신의 Level 정보 전달
5. 클라이언트도 동일한 Level 오픈
6. 클라이언트 레벨 오픈 성공 알림
7. 서버가 해당 클라이언트 전용 PlayerController, Pawn만들고 클라에 복제

### 세션 서비스

IP를 직접 치지 않아도 되게 해주는 매칭 중계 서비스

서버 PC -> Level열기 + 자신의 IP를 중앙 서버에 등록
클라 -> 게임 참가 요청 -> 중앙 서버가 IP반환 -> 접속

스팀, EOS(Epic Online Services)등이 이 역할

## 복습

1. Run Under One Process

true (기본) : 서버/클라이언트 모두 하나의 프로세스에서 실행 -> 에셋 공유, 빠름
false : 서버/클라이언트가 완전히 분리 -> 진짜 멀티환경

정확한 디버깅을 위해 false로

2. NetMode

이 프로세스가 뭔지, 현재 실행중인 게임 프로세스가 네트워크에서 어떤 역할인지를 나타냄

StandAlone, DedicatedServer, ListenServer, Client

3. NetClient, NetDriver

어떻게 연결되는지

UNetDriver : 네트워크 로우레벨 통신 관리, 싱글플레이에서는 생성 안됨
UNetConnection : 다른 PC와의 연결 1개에 대응하는 객체

서버의 UNetDriver -> 클라이언트 수만큼 UNetConnection 관리
클라의 UNetDriver -> ServerConnection 하나만 관리

NetMode는 결국 UNetDriver의 상태 (서버인지 클라인지)에 따라 결정

4. Role

이 액터가 어느 PC에 있는지, 같은 월드안에서도 액터마다 역할이 다를수 있기 때문에 Role이 필요
Authority : 서버에 스폰된 원본액터
Proxy : 클라에 복제된 허상 액터

Local Role : 현재 내 PC에서 이 액터의 역할
Remote Role : 연결된 상대 PC에서의 이 액터의 역할

None : 액터없음
Authority : 서버에 있는 원본 액터
SimulatedProxy : 복제된 액터(다른사람)
AutonomousProxy : 복제된 액터이지만 OwningClinet가 직접 제어 가능(내 캐릭터), 이 역할이어야 입력을 받아서 서버로 RPC를 보내는 것이 허용, 다른 사람 캐릭터는 SimulatedProxy이르모 입력을 받을 수 없다.

5. RPC

내 PC에서 호출되지만 다른 PC에서 실행되는 함수

Call : 컴파일 타임에 실행 위치가 고정(일반 함수)
Invoke : 런타임에 실행 위치가 결정(RPC, 함수 포인터, 동적 바인딩) 그래서 RPC는 호출한다가 아니고 Invoke한다고 함

RPC는 주로 게임에 큰 영향이 없는 일시적인 효과에 사용한다. 코스메틱 : 사운드,파티클,이펙트
영향이 많은건 Property Replication사용

6. Ownership

RPC 실행 조건의 핵심, 실제로 실행될지 여부는 액터의 소유권에 달려있음

- Client Owned Actor : PlayerController가 대표적
- Owned by different client : 다른 클라이언트 소유 캐릭터
- Server-Owned Actor : SetOwner()없이 서버에서 스폰된 액터

Server RPC 조건: 클라이언트 소유 액터에서 호출해야 서버로 전달됨
Client RPC 조건: 서버에서 해당 ClientConnection 소유 액터를 통해 호출해야 함
왜 PlayerController에서 RPC를 호출하나? -> Server RPC는 ClientConnection이 소유한 액터에서 호출해야함

7. Property Replication

특정 속성값이 서버에서 변경될 때 자동으로 모든 클라이언트에 복제되는 기능
항상 Authority -> Proxy방향만 가능, 클라에서 값바꿔도 서버에 복제되지않음 서버에서 값을 바꿔야 클라로 복제
