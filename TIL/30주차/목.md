## 1. 파츠 드롭 스폰 — `SpawnInWorld` 추출

### 배경

기존에 바닥에 파츠를 떨어뜨리는 코드는 `UNSPartEquipComponent::SpawnDroppedPart`(private) 딱 하나였다.
이 함수는 **플레이어가 파츠를 교체할 때 기존 파츠를 발 아래 떨구는 경로 전용**으로 설계된 것이며, `UNSPartEquipComponent`는 `PlayerState`에 부착된 컴포넌트다.

문제는 **몬스터 드롭**이다. 몬스터가 죽는 상황엔 플레이어가 주체가 아니다. 이 상황에서도 `ANSDroppedPart`를 월드에 스폰하는 로직이 필요한데, 기존 코드를 쓰려면 아래 같은 어색한 경로를 거쳐야 했다:

```
몬스터 사망
  → 임의 플레이어의 PlayerState 획득
  → PlayerState → PartEquipComponent 접근
  → SpawnDroppedPart() 호출
```

### 문제점

1. **불필요한 의존성**: 스폰이라는 행위(World에서 Deferred 스폰)는 플레이어/PlayerState와 아무 관련이 없다. 그런데 이 로직이 `UNSPartEquipComponent` 안에 묻혀 있어서, 몬스터 드롭 오케스트레이터(GameMode)가 굳이 PlayerState를 경유해야 하는 구조가 된다.

2. **스폰 패턴 중복 위험**: `SpawnActorDeferred → Initialize → FinishSpawning` 규약을 지켜야 하는데(`Initialize` 전에 `FinishSpawning`하면 `StoredInstance`가 비어서 비주얼/픽업 모두 고장), 이 로직이 한 곳이 아닌 두 군데 이상에 흩어지면 한 군데서 규약을 놓치는 실수가 생긴다.

3. **"닭과 달걀" 문제**: 인스턴스 메서드(비-static)로 만들면, 이 함수를 호출하려면 이미 `ANSDroppedPart` 인스턴스가 하나 있어야 한다. 그런데 이 함수가 하는 일이 바로 새 `ANSDroppedPart`를 만드는 것이다. 팩토리 함수는 static이거나 외부 매니저에 두는 것이 자연스럽다 (언리얼의 `UGameplayStatics::SpawnEmitterAtLocation` 등과 동일한 패턴).

### 해결

`ANSDroppedPart`에 **static 팩토리 헬퍼 `SpawnInWorld`**를 추가했다:

```cpp
// NSDroppedPart.h
static ANSDroppedPart* SpawnInWorld(UWorld* World, TSubclassOf<ANSDroppedPart> Class,
    const FNSPartData& Part, const FVector& Location);
```

- `Class`가 null이면 `ANSDroppedPart::StaticClass()`로 폴백
- 내부에서 `SpawnActorDeferred → Initialize → FinishSpawning` 규약을 완전히 처리
- 실패 시 `nullptr` 반환

기존 `UNSPartEquipComponent::SpawnDroppedPart`는 이 헬퍼를 호출하는 한 줄로 축소했다:

```cpp
// NSPartEquipComponent.cpp
void UNSPartEquipComponent::SpawnDroppedPart(const FNSPartData& Part, const FVector& Location)
{
    ANSDroppedPart::SpawnInWorld(GetWorld(), DroppedPartClass, Part, Location);
}
```

### 결과

| 경로 | 변경 전 | 변경 후 |
|------|---------|---------|
| 장착 교체 시 드롭 | `EquipComponent::SpawnDroppedPart` 직접 구현 | 내부에서 `SpawnInWorld` 호출 — 의미/책임 그대로 유지 |
| 몬스터 드롭 (신규) | 경로 없음 | `ANSDroppedPart::SpawnInWorld` 직접 호출 — PlayerState 무관 |
| 스폰 규약 보장 위치 | `EquipComponent::SpawnDroppedPart` 안 | `SpawnInWorld` 한 곳 |

## 2. GameMode 재화 Multiplier 에디터 노출

### 배경

런 종료 시 영구 재화(공통/스킬)를 확정할 때, 클리어/전멸에 따라 획득량에 배율을 적용한다.

- 클리어: 100% → `ClearMultiplier = 1.0f`
- 전멸(실패): 50% → `FailMultiplier = 0.5f`

이 수치가 코드에 하드코딩되어 있었다.

### 변경

`ANSRunGameMode`에 `EditDefaultsOnly`로 노출해 에디터(BP 디테일 패널)에서 조정 가능하게 했다:

```cpp
// NSRunGameMode.h
UPROPERTY(EditDefaultsOnly, Category = "Currency")
float ClearMultiplier = 1.0f;

UPROPERTY(EditDefaultsOnly, Category = "Currency")
float FailMultiplier = 0.5f;
```

디테일 패널 **Currency** 카테고리에서 수치를 수정할 수 있다. 코드 재컴파일 없이 밸런스 조정 가능.

### FFastArraySerializer

일반 TArray + UPROPERTY(Replicated)로 배열을 리플리케이트하면, 배열 원소 하나만 바뀌어도 배열 전체를 직렬화해서 보냄.

FFastArraySerializer는 델타(변경분)만 전송하는 UE 전용 구조:

- 원소 추가/변경 → 해당 항목만 전송
- 원소 삭제 → 삭제 마커만 전송
- 원소별로 PostReplicatedAdd / PostReplicatedChange / PreReplicatedRemove 콜백도 제공

# 알고리즘

```c++
void convert(string& m, unordered_map<string, string>& words){
    for (auto& [key, value] : words){
        auto it = m.find(key);
        while ((it = m.find(key, it)) != string::npos){
            m.replace(it, key.length(), value);
            it++;
        }
    }
}
```

- string에서 특정 string을 찾는거 find
- it로 저장해놓았다가 replace로 해당 문자열을 다른 문자열로 변경함
- replace는 하나만 적용되어서 while문을 돌면서 모두 변경하는식

## 언리얼 엔진 용어 정리

### 오브젝트

오브젝트는 언리얼엔진의 가장 기본적인 클래스. 오브젝트는 구성단위 역할을 함. 에셋에 필수적인 기능을 많이 포함하고 있음, 거의 모든것이 오브젝트로부터 상속받거나 함수 기능을 사용함, C++에서 `UObject`는 모든 오브젝트의 베이스 클래스, 가비지 컬렉션, 언리얼 에디터에 변수를 노출하기 위한 메타데이터 지원, 로딩 및 저장직렬화 등의 기능을 구현함

### FSoftObjectPaths와 TSoftObjectPtr

FSoftObjectPath은 에셋 전체 이름으로 된 스트링이 들어있는 단순한 구조체, 클래스에 이 유형의 프로퍼티를 만들면 에디터에는 마치 UObject\* 프로퍼티인 양 나타남. 쿠킹과 리디렉터도 제대로 처리되므로, SoftObjectPath가 있다면 디바이스에서의 정상 작동도 보장됨
TSoftObjectPtr은 기본적으로 FSoftObjectPath를 감싸는 TWeakObjectPtr이며, 에디터 UI에서 특정 클래스만 선택되게끔 제한시킬 수 있도록 하기 위해서 특정 클래스로 고정. 참조된 에셋이 메모리에 존재한다면 TSoftObjectPtr.Get()은 그것을 반환함, 존재하지 않는다면 ToSoftObjectPath()를 호출하여 참조하는 에셋을 찾아내어 아래 설명된 메서드를 사용해서 로드한 다음 TSoftObjectPtr.Get()을 다시 호출하여 역참조함
TSoftObjectPtr과 SoftObjectPath는 아티스트나 디자이너가 레퍼런스를 수동 셋업하는 경우에는 좋지만, 특정 조건을 만족하는 에셋질의 작업을 모든 에셋 로드없이 하려는 경우, 에셋 레지스트리와 오브젝트 라이브러리를 사용하는것이 좋다.

### 에셋 레지스트리와 오브젝트 라이브러리

에셋 레지스트리는 에셋에 대한 메타데이터를 저장하여 해당 에셋에 대한 검색 및 질의를 가능케 해주는 시스템. 에디터에서는 콘텐츠 브라우저에 정보를 표시하기 위해 사용되나, 게임플레이 코드에서 현재 로드되지 않은 게임플레이 에셋에 대한 메타데이터 질의를 하는데도 사용할 수 있음 -> 결론 에셋 자체를 로드하지 않고도 에셋의 정보만 검색하는 시스템

로드되지 않은 에셋 그룹을 가지고 작업하는 가장 쉬운 방법은 ObjectLibrary임. 특정 폴더의 에셋들을 한번에 모아 다루기 쉽게 해주는 컨테이너, 해당 폴더의 로드된 오브젝트 목록이나 로드되지 않은 에셋의 FAssetData목록을 모아서 관리하는 도우미 객체

### StreambleManager와 비동기 로딩

FStreamableManager는 게임 전역 유일무이(singleton)오브젝트로 비동기 로딩을 함. 비동기 로드를 하고나면 콜백이 호출되는데 이때 콜백에서 바로 .Get()을 해서 쓴뒤 끝낼수도있지만 이후에도 계속 써야 한다면 어딘가에 실제 객체 포인터를 멤버로 저장하는식으로 하드하게 잡아줘야 한다.
