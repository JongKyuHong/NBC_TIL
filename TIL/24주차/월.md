# 개념 공부

## 생성자와 소멸자의 역할

- 생성자의 역할

객체가 생성될때 자동으로 호출되는 특수 멤버 함수, 객체의 초기 상태를 설정하는 역할을 함
멤버 변수 초기화, 리소스 확보(파일 핸들, 네트워크 소켓 오픈, 메모리 동적 할당)등 객체가 작동하는데 필요한 자원 획득

- 소멸자의 역할

소멸자는 객체의 수명이 다했을때 호출되며, 객체가 사용하던 자원을 안전하게 정리하는 역할
메모리 및 리소스 해제, 메모리 누수 방지(명시적으로 자원 해제)를 수행

## inline함수는 언제 사용되며 컴파일러가 실제로 어떻게 처리하는가

inline함수는 함수 호출 시 발생하는 오버헤드를 줄이기 위해 사용하는 함수입니다. 일반적인 함수는 호출 시 제어권이 함수의 주소로 이동했다가 다시 돌아오지만, 인라인 함수는 컴파일 타임에 함수의 본문 코드가 호출지점에 직접 치환됩니다.

- 컴파일러의 처리

컴파일러가 판단하기에 함수가 너무 복잡(재귀, 루프등)하거나 코드가 길면 인라인화를 거절하고 일반 함수로 처리
인라인 함수는 호출부에 코드가 박히기 때문에 헤더파일에 정의를 포함해야함

## C++에서 함수 포인터를 사용하는 예제를 작성하고 이유를 설명

함수포인터는 함수의 실행코드가 시작되는 메모리 주소를 저장하는 변수, 일반 변수가 데이터를 저장한다면 함수 포인터는 '동작'을 저장하며 이를 통해 실행중에 호출할 함수를 동적으로 결정할 수 있음

```c++
int Add(int a, int b) {return a + b;}
int Multiply(int a, int b) {return a * b;}
int main(){
    int (*operationPtr)(int, int);

    operationPtr = Add;
    std::cout << "결과 : " << operationPtr(10, 5) << std::endl;

    operationPtr = Multiply;
    std::cout << "결과 : " << operationPtr(10, 5) << std::endl;

    return 0;
}
```

- 사용하는 이유

콜백 구현 : 특정 이벤트가 발생했을때 실행될 함수를 등록해두고 나중에 호출할 수 있음
전략 패턴 : 동일한 인터페이스에서 상황에 따라 다른 알고리즘을 선택해 실행할 수 있어 코드의 유연성이 높아짐
코드 역전 : 함수를 호출하는 쪽에서 구체적인 함수 이름 대신 포인터만 알고 있으면 되므로, 모듈간의 의존성을 줄일 수 있음

## 함수 오버로딩이란?

동일한 이름의 함수를 매개변수 타입, 개수, 순서를 다르게 하여 여러개 정의하는것, 유사한 작업을 수행하는 함수들에게 일관된 이름을 부여함으로써 코드의 가독성과 편의 증대
오버라이딩은 부모클래스에 정의된 가상함수를 자식에서 동일한 시그니처로 다시 정의하는것

## C++에서 std::array와 C배열 차이점 2가지 이상

1. std::array는 STL 컨테이너 인터페이스를 따르므로 begin(), end(), at()같은 메서드를 지원하고, 범위 기반 for문이나 STL 알고리즘들과 자연스레 연동됨
2. std::array는 크기가 타입 자체에 포함됨, C배열을 함수의 매개변수로 넘기면 첫번째 원소를 가리키는 포인터로 변해버려서 기존 배열의 크기를 모름

```c++
void printSize(int arr[]){
    std::cout << sizeof(arr);
}

int main(){
    int arr[5] = {1, 2, 3, 4, 5};
    std::cout << sizeof(arr);
    printSize(arr);
}

```

C배열의 성능(스택 할당, 연속메모리)은 그대로 유지하면서 타입 안정성과 STL 생태계 호환성을 더한 현대 C++의 대안

## RAII의 개념을 설명하고 장점을 설명

RAII(Resource Acquistion Is Initialization)란 자원의 획득과 해제를 객체의 생명주기에 묶는 패턴
생성자에서 획득해서 -> 소멸자에서 해제
객체가 스코프를 벗어나면 소멸자가 자동으로 호출되므로, 자원 해제를 직접 신경 쓸 필요가 없음

- 장점

1. 자원 누수 방지 해제 코드를 빠뜨리거나 예외 or early return으로 건너뛰는 실수가 원천 차단됨
2. 예외 안정성 보장 예외가 던져져도 스택이 풀리면서 소멸자가 반드시 호출되므로 자원이 항상 정리됨
3. 코드 가독성 향상 자원 해제 코드를 직접 작성할 필요가 없어 비즈니스 로직에만 집중할 수 있음

## 실수 자료형 사용 시 발생문제(부동소수점 오차 등)를 설명

컴퓨터는 모든 수를 2진수로 저장하기 때문에 우리가 쓰는 십진수 소수중 상당수는 이진수로 정확하게 표현이 안된다.
부동소수점 : 소수점이 고정되지 않고 둥둥 떠다닌다는 뜻, 소수점을 앞뒤로 움직여서 아주 크거나 작은 수도 표현이 가능
컴퓨터는 이를 이진수로 구현하기 때문에 부호, 지수부, 가수부가 나뉨(지수부란 소수점 위치, 가수부란 유효숫자)
십진수 정수를 이진수로 변환할때는 2로 나누면서 나머지를 읽는식으로 변환된다. 십진수 소수부는 반대로 2를 곱하면서 정수부를 읽음
0.625를 변환한다고 치면
0.625 x 2 = 1.25 -> 1읽고 0.25남김 // 0.25 x 2 = 0.5 -> 0읽고 0.5 남김 // 0.5 x 2 = 1.0 -> 1읽고 0.0남김 (소수부 0 -> 종료) 0.625 = 0.101(이진수)
근데 0.1같은 숫자는 0.2가 계속 등장해서 반복되기 때문에 오차가 생김

## vector의 push_back을 하면 어떤일이 벌어질까?

capacity가 꽉차지 않았으면 뒤에 추가됨, 만약에 capacity가 꽉찼다면 재할당을 하는데 보통 크기의 2배로 함(size는 실제 저장된 원소 수, capacity는 실제 할당된 메모리 공간) -> 재할당은 O(n) 전체 원소를 새 메모리로 복사해야하므로 비용이 크다, 주소도 당연히 바뀜.
그래서 reserve를 사용하면 됨 -> 미리 공간 확보, 초기화는 값을 미리 넣어놓는

## ++it와 it++의 동작 차이

```c++
Iterator& operator++(){
    this->ptr++;
    return *this // 자기 자신 반환 (복사 없음)
}

Iterator operator++(int){
    Iterator tmp = *this; // 현재 상태 임시 복사
    this->ptr++;
    return tmp; // 복사본 반환
}
```

후위는 증가전 값을 돌려줘야 하므로 복사본을 하나 만들어야 함
int는 복사비용이 거의 없고 컴파일러가 최적화하므로 체감차이가 없다.
하지만 이터는 다름

후의 증가는 증가 전 값을 반환해야 하므로 내부적으로 임시 객체를 하나 복사합ㄴ디ㅏ. int는 컴파일러가 최적화해서 차이가 없지만, map이나 list같은 무거운 이터레이터는 복사비용이 발생하므로 전위 증가를 쓰는것이 습관적으로 좋습니다.

## std::unique_ptr 와 std::shared_ptr 차이를 설명

unique_ptr은 단독 소유, 소유권 이전은 move로만 가능하다. 복사 자체가 컴파일 단계에서 차단, 오버헤드 없음 -> 내부적으로 그냥 포인터 하나

shard_ptr은 공동 소유(참조 카운팅), 내부적으로 참조 카운트를 관리, 여러곳에서 소유권을 공유할 수 있고, 카운트가 0이 되는 순간 자동해제, 순한참조 시 메모리 누수가 발생할 수 있어 weak_ptr을 통해 해결해야함

## find() vs binary_search()

find는 선형 탐색 O(N), 정렬 불필요, 찾으면 iterator반환-> 위치, 값 접근가능, 못찾으면 end()반환
binary_search는 이진 탐색 O(log N), 반드시 정렬 필요, bool반환 -> 있냐 없냐만 알 수 있음, 위치나 값 접근 불가
데이터가 정렬안되어있고 값, 위치 알아야 한다 -> find()
데이터가 정렬되어있고 존재만 필요 -> binary_search()
데이터가 정렬되어있고 위치까지 알아야 한다 -> lower_bound()

## pair vs tuple

pair는 딱 2개만 묶음, .first .second로 접근, map의 원소 타입
tuple 개수 제한 없이 묶음, 3개이상 묶을때, get<N>으로 접근 -> 인덱스라 직관성이 떨어짐, 개수 제한 없음

```c++
// pair — 이름으로 접근 (직관적)
p.first;
p.second;

// tuple — 인덱스로 접근 (덜 직관적)
std::get<0>(t);
std::get<1>(t);
std::get<2>(t);
```

## STL list::sort가 따로 존재하는 이유에 대해 설명

기존 sort는 list에 못씀, sort는 내부적으로 임의접근이 필요함 list는 앞뒤로만 이동이 가능하고 중간으로 바로 점프가 불가능
list::sort는 어떻게 정렬하는가? list는 노드의 포인터를 바꾸는 방식으로 정렬함, 원소를 복사/이동하지 않고 링크만 재연결

```
[3] -> [1] -> [4] -> [2]

[1] -> [2] -> [3] -> [4]

원소 자체는 메모리에서 이동 없음
```

내부적으로는 merge sort(합병 정렬)를 사용, merge sort는 임의접근없이 순차접근만으로 동작

## STL std::map에 대해 설명

내부구조는 레드블랙트리로 구현되어있음, 모든 연산이 O(log n), 삽입할때마다 자동 정렬 유지, 노드 기반이라 메모리가 연속적이지 않음
