## 1. 멀티플레이 — 게임 라운드 UI 표시

### 구현 목표

라운드 정보는 **모든 클라이언트가 동일하게 알아야 하는 정보**이므로 `GameState`에서 관리하고, `ReplicatedUsing`으로 값 변경 시 클라이언트 UI를 자동 갱신한다.

### 구현

```cpp
// JKGameStateBase.h
protected:
    UFUNCTION()
    void OnRep_IncreaseRound();

private:
    UPROPERTY(ReplicatedUsing = OnRep_IncreaseRound)
    int32 CurrentRound = 0;
```

```cpp
// JKGameModeBase.cpp
void AJKGameModeBase::IncreaseRound()
{
    AJKGameStateBase* JKGS = GetGameState<AJKGameStateBase>();
    if (IsValid(JKGS) == false) return;
    JKGS->SetCurrentRound(JKGS->GetCurrentRound() + 1);
}
```

### 설계 포인트

- `CurrentRound` 초기값 = `0` → `BeginPlay()`에서 `IncreaseRound()` 호출 → UI에 **1**로 표시
- `ResetGame()` 호출 시에도 `IncreaseRound()`를 함께 호출하여 라운드가 리셋될 때 UI도 함께 갱신
- **GameMode는 서버 전용**이므로 라운드 증가 로직은 GameMode에서, 클라이언트 동기화는 GameState의 Replication이 담당

## 2. 멀티플레이 — 턴 표시 초기화 문제 해결

### 문제 상황

현재 내 턴인지 표시하는 UI가 **게임 시작 시 갱신되지 않는** 버그 발생.  
`GameState::BeginPlay()`에서 `OnRep_CurrentTurn()`을 직접 호출했지만 동작하지 않았다.

![](https://velog.velcdn.com/images/kyu_/post/ca5aa897-4915-4b3b-aa4b-8596bb48d402/image.png)

### 원인 분석 — UE_LOG로 단계별 추적

```cpp
void AJKGameStateBase::OnRep_CurrentTurn()
{
    APlayerController* PC = UGameplayStatics::GetPlayerController(GetWorld(), 0);
    if (IsValid(PC) == false) return;
    UE_LOG(LogTemp, Warning, TEXT("1"))   // ✅ 출력됨

    AJKPlayerController* JKPC = Cast<AJKPlayerController>(PC);
    if (IsValid(JKPC) == false) return;
    UE_LOG(LogTemp, Warning, TEXT("2"))   // ✅ 출력됨

    AJKPlayerState* JKPS = JKPC->GetPlayerState<AJKPlayerState>();
    if (IsValid(JKPS) == false) return;
    UE_LOG(LogTemp, Warning, TEXT("3"))   // ❌ 첫 턴에만 출력 안됨
    ...
}
```

![](https://velog.velcdn.com/images/kyu_/post/8eca3384-7ec0-4742-94bf-a175a5411ed4/image.png)
- 첫턴과 두번째턴 로그


**복제 순서 문제:**

```
[클라이언트 기준 초기화 순서]

GameState 생성
    └─ BeginPlay() 호출
          └─ OnRep_CurrentTurn() 실행
                └─ ❌ 이 시점에 PlayerState가 아직 복제 미완료
                       → GetPlayerState() == nullptr
```

`GameState`의 `BeginPlay()`가 호출되는 시점에는 `PlayerState`가 클라이언트에 아직 복제되지 않은 상태였다.  
언리얼 엔진에서 액터 간 복제 순서는 보장되지 않으므로, **PlayerState가 복제 완료된 시점을 명시적으로 잡아야 한다.**

### 해결 방법 — `OnRep_PlayerState()` 활용

`PlayerController`에는 `PlayerState`가 클라이언트에 복제 완료된 순간 호출되는 `OnRep_PlayerState()`가 존재한다.  
이 콜백에서 UI 갱신 함수를 호출하면 타이밍 문제가 해결된다.

```cpp
// JKPlayerController.cpp
void AJKPlayerController::OnRep_PlayerState()
{
    Super::OnRep_PlayerState();

    AJKGameStateBase* JKGS = GetWorld()->GetGameState<AJKGameStateBase>();
    if (IsValid(JKGS) == false) return;

    JKGS->OnRep_CurrentTurn();  // PlayerState 복제 완료 후 UI 갱신
}
```

> ⚠️ `OnRep_PlayerState()`는 **클라이언트 전용** 콜백이다. 서버에서는 호출되지 않으므로 서버 로직과 혼용하지 않도록 주의할 것.

![](https://velog.velcdn.com/images/kyu_/post/f2c435e4-9b86-4b6b-8eb7-6c2d3f1dca79/image.png)

### 핵심 개념 정리

> 멀티플레이에서 특정 액터에 의존하는 초기화 로직은  
> **해당 액터의 복제 완료 콜백 시점**에 실행해야 한다.

## 3. 베이직 반 — 스마트 포인터

### 개념 체계

| 구분 | 타입 | 역할 | 언리얼 타입 |
|------|------|------|------------|
| 인스턴스 **소유** | Strong | 레퍼런스 카운트 증가, 소멸에 관여 | `TObjectPtr<T>` |
| 인스턴스 **참조** | Weak | 레퍼런스 카운트 미증가, 소멸에 미관여 | `TWeakObjectPtr<T>` |
| 리소스 **소유** | Hard Reference | 리소스 로드/언로드에 관여 | `TSoftObjectPtr` (Hard) |
| 리소스 **참조** | Soft Reference | 리소스 로드/언로드에 미관여 | `TSoftObjectPtr` |

- Reference Count가 0이 되면 자동 소멸 → 스마트 포인터의 핵심
- `TObjectPtr` = "내 것"을 소유, `TWeakObjectPtr` = "남의 것"을 참조하는 것으로 구분하면 이해하기 쉽다

### 언리얼에서 주의할 점 — UPROPERTY 필수

```cpp
// ❌ UPROPERTY 없으면 GC가 레퍼런스 카운트를 추적하지 못함 → 댕글링 포인터 발생
TObjectPtr<AActor> MyActor;

// ✅ 반드시 UPROPERTY를 붙여야 GC가 수명을 관리
UPROPERTY()
TObjectPtr<AActor> MyActor;
```

### TUniquePtr\<T\>

- **단일 소유권**: 복사 불가, 이동만 가능
- `MoveTemp()`: 캐스트 연산만 수행하며 기존 참조를 RValue 상태로 전환
- `&&` (RValue Reference): "이 값을 RValue로 취급하겠다"는 선언
- LValue에 대입되는 순간 소유권이 이전되고 기존 참조가 끊어진다

```cpp
TUniquePtr<int> A = MakeUnique<int>(10);
TUniquePtr<int> B = MoveTemp(A);  // A의 소유권이 B로 이전, A는 nullptr
```

### RAII (Resource Acquisition Is Initialization)

> **자원의 획득 = 객체의 초기화**  
> 객체가 소멸될 때 소멸자에서 자원을 자동 해제하는 C++ 관용 패턴

- STL 컨테이너, 스마트 포인터가 대표적인 RAII 구현체
- 언리얼의 `TUniquePtr`, `TSharedPtr`도 RAII 기반

## 4. 코드카타

### BOJ 1107 - 리모컨 (브루트포스)

[문제 링크](https://www.acmicpc.net/problem/1107)

**접근:** 사용 가능한 버튼 조합으로 만들 수 있는 모든 숫자를 탐색하여 최소 이동 횟수를 구한다.

**핵심 반례:**

```
입력: 889번 채널, 고장 버튼: 0 2 3 4 5 6 7 8 9
사용 가능 버튼: 1만 사용 가능

초기 오답: 789 → 오답 (같은 자릿수에서만 탐색했기 때문)
정답: 226 (1111에서 -222 이동)
```

**원인:** 목표 채널보다 **자릿수가 더 긴 숫자에서 역방향으로 도달하는 경우**를 고려하지 않았다.  
**해결:** `N의 길이 + 1` 자리까지 탐색 범위를 확장하여 해결.

```cpp
void recursion(const string& tmp, const string& N)
{
    if (!tmp.empty())
    {
        minValue = min(minValue, abs(stoi(N) - stoi(tmp)) + (int)tmp.length());
    }
    // N보다 한 자리 더 긴 경우까지 탐색
    if ((int)tmp.length() >= (int)N.length() + 1) return;

    for (int i = 0; i <= 9; i++)
    {
        if (buttons[i] == 0)
            recursion(tmp + to_string(i), N);
    }
}
```

**회고:** 로직을 짠 뒤 **엣지 케이스를 체계적으로 찾는 연습**이 필요하다. 특히 경계값, 역방향 접근 가능성을 의식적으로 검토해야 한다.

### BOJ 2096 - 내려가기 (슬라이딩 윈도우 DP)

[문제 링크](https://www.acmicpc.net/problem/2096)

**제약:** 메모리 **4MB 제한** → 2차원 배열 DP 불가

**접근:** 이전 행 결과만 1차원 배열로 유지하는 슬라이딩 윈도우 방식으로 메모리 절약

```cpp
currMaxDp = max(prevMaxDp, prevMaxDp) + a;[3]
currMaxDp = max({prevMaxDp, prevMaxDp, prevMaxDp}) + b;[1][3]
currMaxDp = max(prevMaxDp, prevMaxDp) + c;[3][1]
// 최소도 동일한 구조로 처리
```

**슬라이딩 윈도우 vs 투 포인터 정리:**

| | 슬라이딩 윈도우 | 투 포인터 |
|---|---|---|
| 구간 크기 | **고정** | **가변** |
| 용도 | 고정 크기 구간의 최적값 탐색 | 조건을 만족하는 가변 구간 탐색 |

## 오늘의 핵심 요약

1. **멀티플레이 복제 순서:** 다른 액터에 의존하는 초기화는 해당 액터의 복제 완료 콜백(`OnRep_PlayerState` 등)에서 처리할 것
2. **스마트 포인터:** 소유(`TObjectPtr`) vs 참조(`TWeakObjectPtr`) 구분, UPROPERTY 없으면 GC가 추적 불가
3. **브루트포스 반례:** 경계 너머의 역방향 접근 가능성을 항상 검토
4. **메모리 제한 DP:** 슬라이딩 윈도우로 이전 상태만 유지
