# 면접 질문

## 실시간 멀티플레이 race condition상황 & 방지 전략에 대해 설명

두개 이상의 엔티티(클라이언트나 서버 스레드)가 공유된 데이터에 동시에 접근하여 수정하려고 할 때, 실행 순서에 따라 결과가 달라지는 현상을 말함
특히, 네트워크 지연(Latency)이 존재하는 게임 환경에서는 데이터의 무결성을 깨뜨리는 치명적인 버그의 원인이 됨

### 실시간 멀티 플레이에서의 대표적인 상황

1. 아이템 선정
   두 명의 플레이어가 땅에 떨어진 전설 아이템 위로 동시에 지나감
   상황 : 클라이언트 A와 B모두 내가 아이템을 먹었다는 신호를 보냄
   결과 : 적절한 방지 전략이 없다면, 서버는 모두에게 아이템을 지급하거나, 한 명은 먹었는데 필드에는 아이템이 여전히 남아있는 등의 논리적인 오류가 발생함

2. 체력 0과 사망 처리
   두명의 플레이어가 동시에 한 적에게 결정타를 날릴때
   상황 : 거의 같은 밀리초단위로 데미지 패킷이 들어옴
   결과 : 적의 HP가 마이너스가 되면서 사망 처리가 두 번 실행되어 경험치나 전리품이 중복 지급될 수 있음

3. 뱅킹 및 거래
   거래 창에서 `확인`버튼을 동시에 누르거나, 창고에서 아이템을 빼는 행위
   상황 : 한 아이템을 인벤토리와 창고 양쪽에 동시에 존재하게 만드려는 시도

### 방지 전략

멀티플레이 게임에서는 하드웨어적인 락(Lock)뿐만 아니라 아키텍처 설계를 통한 방지가 병행되어야 함

1. 서버 권위 구조(Server Authoritative Architecture)
   가장 근본적인 해결책. 클라이언트는 "나는 아이템을 먹었다"라고 통보하는 것이 아니라, "내가 아이템을 먹어도 될까?" 라고 요청만 함
   - 작동 방식 : 서버는 요청을 받은 순서대로 처리하며, 첫 번째 요청자에게 아이템을 할당한 후 뒤이은 요청은 "이미 처리된 아이템"이라며 거절

2. 뮤텍스 및 크리티컬 섹션
   멀티스레드로 동작하는 서버 코드 내부에서 데이터 수정을 보호할때 사용
   - 작동 방식 : 특정 데이터를 수정하는 동안 다른 스레드가 접근하지 못하도록 잠금장치를 검
   - 언리얼 예시 : FScopeLock등을 사용하여 특정 코드 블록의 동시 실행을 막음

3. 원자적 연산(Atomic Operations)
   복잡한 락 없이도 한 번에 하나의 연산만 수행되도록 보장하는 CPU 수준의 기법
   - 작동 방식 : `std::atomic`이나 언리얼의 `FPlatformAtomics`를 사용하여 변수 값을 증가시키거나 비교 후 교체(Compare-and-Swap)함
   - 장점 : 락(Lock)보다 가볍고 빠름

4. 시퀀스 넘버 및 타임스탬프
   패킷의 순서를 보장하여 뒤늦게 도착한 패킷이 이전 데이터를 덮어쓰지 않게 함
   - 작동 방식 : 모든 요청에 번호를 매겨, 서버가 이미 10번 패킷을 처리했다면 지연때문에 늦게 온 9번 패킷은 무시

### 언리얼에서의 실전 대응

- Validation(RPC검증) : ServerRPC_WithValidation 을 사용하면 서버가 요청을 실행하기 전에 이 요청이 논리적으로 가능한지(예 : 플레이어가 아이템 근처에 정말 있는가?)를 먼저 체크
- GAS : GAS는 예측(Prediction)시스템을 내장하고 있음. 클라이언트에서 먼저 실행하되, 서버에서 "그 아이템은 이미 누가 먹었어"라는 결과가 오면 클라이언트의 상태를 강제로 되돌리는(Rollback)처리를 정교하게 수행
- Replication Pausing : 특정 중요 데이터가 동기화되는 동안에는 관련 로직이 실행되지 않도록 흐름을 제어

## OOP에 대해서 설명해주세요

데이터와 그 데이터를 처리하는 루틴을 하나로 묶은 '객체(Object)'들의 집합으로 파악하는 프로그래밍 패러다임

1. 캡슐화
   데이터(속성)와 그 데이터를 조작하는 코드(메서드)를 하나의 단위인 '클래스'로 묶는 것을 의미함
   - 정보은닉 : 외부에서 객체 내부의 핵심 데이터에 직접 접근하는것을 막음
   - 이전 : 객체 내부 구현이 바뀌어도 이를 사용하는 외부 코드를 수정할 필요가 없어 유지보수가 용이

2. 상속
   이미 정의된 상위 클래스의 특징을 하위 클래스가 물려받는 것
   - 언리얼 예시 : AActor라는 부모 클래스가 가진 위치정보, 물리 기능 들을 ACharacter가 그대로 물려받고, 여기에 '걷기', '점프'같은 기능을 추가하는 식
   - 이점 : 코드의 재사용성을 높이고 계층적인 구조를 설계할 수 있음

3. 추상화
   불필요한 세부 구현은 숨기고, 사용자에게 필요한 핵심 기능만을 노출하는 것
   - 구현 방식 : C++의 순수 가상함수나 언리얼의 인터페이스(UInterface)가 대표적

4. 다형성
   하나의 인터페이스나 함수가 상황에 따라 서로 다른 방식으로 동작하는 능력
   - 오버라이딩 : 부모의 함수를 자식에서 재정의하는 것
   - 오버로딩 : 같은 이름의 함수를 매개변수만 다르게 하여 여러 개 정의하는 것
   - 이점 : 코드의 유연성을 높여주며, 구체적인 타입을 몰라도 부모 타입 포인터만으로 다양한 자식 객체들을 일관되게 다룰 수 있음

## 캡슐화란 무엇인가? 장점 2가지를 설명해주세요

캡슐화는 OOP의 한 원칙으로 데이터(속성)와 데이터를 다루는 함수(메서드)를 하나의 단위(클래스)로 묶는것을 말함, 단순히 묶는것에 그치지 않고 외부에서 객체의 내부 상태에 직접 접근하지 못하도록 제한하고 오직 허용된 통로(인터페이스)를 통해서만 상호작용 가능하게 만드는것이 핵심

### 장점

1. 데이터 보호 및 무결성 유지
   객체 내부의 중요한 데이터를 외부에서 함부로 수정하지 못하도록 보호할 수 있음
   설명 : 멤버변수는 private으로 관리하고, SetSpeed같은 메서드를 통해서만 수정하게 되면 speed를 음수로 만드려는 부적절한 인풋도 검증로직으로 막을 수 있음(무결성 지킴)

2. 유지보수성 및 유연성 향상
   객체 내부의 구현이 바뀌어도 해당 객체를 사용하는 외부 코드는 수정할 필요가 없음
   설명 : 만약에 내부 데이터저장방식을 배열에서 Map으로 변경하더라도 불러서 사용하는 외부에서는 GetData()로 불러서 사용하기만 하면 됨

## 가상 함수와 VTable

virtual 키워드가 붙은 함수는 어떻게 런타임에 자기 자신을 찾아갈까??

virtual 함수는 실행중에 결정되는 동적 바인딩을 사용함, 그 핵심은 VTable(가상 함수 테이블)이 있음
VTable : 클래스마다 하나씩 생성되는 '함수 주소록', 이 테이블에는 그 클래스가 호출할 가상 함수들의 실제 메모리 주소가 적혀있음
VPtr : 객체(인스턴스)마다 내부에 몰래 생성되는 포인터, 이 포인터는 해당 클래스의 VTable을 가리킴

### 찾아가는 과정

1. 프로그램 실행 중 가상 함수를 호출하면, 객체 내부의 VPtr을 확인
2. VPtr이 가리키는 VTable로 점프
3. VTable에 적힌 실제 함수의 주소를 읽어서 그 코드를 실행

결론 : 이 과정덕에 부모 포인터로 자식 객체를 기리키고 있어도, 실제 객체의 VPtr을 따라가서 자식이 오버라이딩 함수를 정확히 찾아낼 수 있음

## 스마트 포인터

TSharedPtr, TUniquePtr, TWeakPtr의 차이점과 왜 UObject에는 이들을 사용하면 안되는지
대답 : TSharedPtr은 소유권을 공유, TUnique는 독점 소유권 소유권 이전만 가능, TWeakPtr은 소유권을 가진다기 보다는 멀리서 관측만 하고 있음, TSharedPtr을 사용하면 순환참조 문제가 생길 수 있는데 이때 TWeakPtr로 해결할 수 있음

### UObject에 사용하면 안되는 결정적 이유

- 이중 관리 문제 : UObject는 언리얼 엔진의 가비지 컬렉터가 수명을 관리. 여기에 TSharedPtr을 사용하면 GC는 지우고 스마트 포인터는 아직 참조중인 충돌이 발생
- 리플렉션 시스템 : 언리얼의 GC는 UPROPERTY() 매크로가 붙은 포인터를 추적하며 메모리를 관리. 스마트 포인터는 이 리플렉션 시스템과 호환되지 않아 GC가 해당 객체를 유효한 객체로 인식하지 못하고 지워버릴 수 있음

결론 : UObject 계열은 엔진의 가비지 컬렉션 시스템이 수명을 관리하므로 UPROPERTY()를 사용해 참조를 유지해야 하며, 스마트 포인터(TSharedPtr 등)는 일반 C++ 클래스나 구조체의 수명을 관리할 때만 사용해야 함

## Cast vs StaticCast

언리얼의 Cast<T>와 C++ 표준 static_cast의 차이점은 무엇이며, 왜 언리얼에서는 전자를 권장할까

- static_cast : 컴파일 시점에 타입을 변환, 위험성이 있다. 부모 타입을 자식 타입으로 강제 변환할 때(DownCasting), 실제 객체가 그 자식 타입이 아니더라도 일단 변환해버림. 이는 런타임에 크래시를 유발할 수 있음
- Cast : 언리얼의 리플렉션 정보(UClass)를 사용하여 런타임에 실제 타입을 검사. 만약 변환하려는 타입이 실제 객체와 맞지 않으면 nullptr을 반환함

왜 Cast를 권장할까? 런타임에 안전하게 타입을 확인할 수 있고, 조건문에서 안전하게 사용가능, dynamic_cast보다 언리얼의 Cast가 엔진 시스템에 최적화되어 훨씬 빠름

### 추가 : Cast가 안전한건 알겠는데 Tick같은 곳에서 수십번씩 Cast를 호출한다면 성능문제가 없을까?

언리얼의 Cast는 내부적으로 최적화되어 있지만, 리플렉션 정보를 확인하는 연산이 포함되므로 Tick같은 빈번한 루프내에서 남발하는것은 피해야 함. 이럴때는 Beginplay에서 미리 캐싱해 두거나, 인터페이스(UInterface)를 사용해 직접적인 타입 캐스팅없이 함수를 호출하는 방식으로 설계하는것이 좋다.

## GameMode vs GameState

멀티플레이 환경에서 이 둘의 결정적인 차이(존재 위치, 역할)은 무엇인가

GameMode는 게임의 규칙을 관리함. 해킹 방지를 위해 서버에만 존재한다. 클라이언트는 게임모드에 접근할 수 없음(nullptr 반환)
GameState는 공유정보 게임의 상태를 관리. 현재 점수, 남은 시간 등 모든 플레이어가 알아야 하는 정보는 서버와 클라이언트 모두에 존재하며 서버로부터 복제됨

### GameMode에만 있는 로직을 클라이언트가 알고 싶을땐 어떻게 해야하는가?

서버가 GameState나 PlayerState에 저장하여 리플리케이션되도록 설계해야 함, 만약 특정 행동을 요청해야 한다면 Server RPC를 통해 서버의 GameMode 로직을 실행하도록 요청해야 함

## Actor의 생명주기

액터가 생성될 때 PreInitializeComponents, PostInitializeComponents, BeginPlay중 어느 것이 먼저 실행될까?

Pre : 컴포넌트 초기화 직전, Post : 액터의 모든 컴포넌트가 생성을 마치고 초기화된 직후, BeginPlay : 액터가 월드에 완전히 배치되어 게임 루프에 참여할 준비가 끝났을때 호출

Pre -> Post -> BeginPlay

## CDO

언리얼의 모든 클래스는 실행 시점에 단 하나의 '기본 객체'를 가짐. 이것의 용도는 무엇일까?
모든 클래스의 기본값을 정의하는 역할을 한다. 에디터에서 설정할때 기본으로 가지는 값

### 왜필요?

- 템플릿 : 새로운 액터를 스폰할 때, CDO라는 원본 틀을 그대로 복사해서 만듬, 덕분에 생성 속도가 매우 빠름
- 직렬화 : 에디터의 '디테일 패널'에서 값을 수정하면, 그 값은 CDO에 저장됨. 이후 생성되는 모든 인스턴스는 그 수정된 CDO값을 기본값으로 가짐

### 직렬화

메모리에 살아있는 객체를 '냉동 보관'하거나 '택배로 보내는' 기술이라고 이해하면 쉽다.

1. 직렬화란?
   메모리상에 존재하는 복잡한 객체(Object)의 상태를 파일로 저장하거나 네트워크로 전송할 수 있는 일련의 바이트 스트림(0과 1의 나열)으로 변환하는 과정을 말함
   반대로 이 바이트 스트림을 다시 메모리 상의 객체로 복구하는것을 역직렬화라고 함

2. 언리얼에서의 직렬화
   - 에디터와 에셋 : 블루프린트나 액터의 변수 값을 수정하고 저장을 누르면 엔진은 그 객체의 현재 상태를 직렬화하여 .uasset이나 .umap 파일로 하드디스크에 기록함
   - 네트워크 리플리케이션 : 서버에 있는 액터의 위치나 체력정보를 클라이언트에 보낼 때, 객체 자체를 보낼 수 없음 이때 데이터를 패킷(바이트 나열)으로 직렬화해서 보내고, 클라이언트는 이를 받아 자신의 메모리에 있는 객체에 역직렬화하여 반영함
   - 복사와 붙여넣기 / 되돌리기 (Undo/Redo) : 에디터에서 객체를 복사하면 엔진은 해당 객체를 텍스트나 바이너라 형태로 직렬화하여 클립보드에 보관

- CDO와 직렬화의 관계
  변경점만 저장한다. 우리가 액터를 월드에 배치하고 특정 변수 하나만 수정했다면, 엔진은 그 액터의 모든 정보를 저장하지 않고, 이 액터는 CDO와 비교했을때 이 변수만 다르다라는 정보만 직렬화하여 저장함

## 드로우 콜

CPU가 GPU에게 이거 그려라 라고 보내는 명령

- 성능 저하의 이유 : CPU와 GPU는 서로 다른 장치, CPU가 명령을 보낼때마다 하드웨어 간의 통신 부하가 발생
  아무리 고성능 그래픽카드라도 수만번의 드로우콜을 받으면 처리속도가 떨어짐

- 드로우 콜을 줄이는 법

1. 인스턴싱 -> ISM/HISM 똑같이 생긴 나무 1000그루를 그린다면 나무를 그리라는 명령을 천번 보내는게 아닌 나무 천개를 이 좌표에다가 그려라라고 명령 한번으로 끝내는 기술
2. 엑터 머징 -> 여러개의 개별 스태틱 메쉬 액터를 하나의 메쉬로 합쳐버리는 것
3. HLOD(Hierarchical Level of Detail) : 멀리 있는 여러개의 액터를 하나의 단순한 메쉬로 뭉쳐서 보여주는 방식

## Tick 최적화

Tick을 줄이는게 CPU 최적화의 핵심

- FTimerHandle : 굳이 매 프레임 확인할 필요 없는 로직(0.5초마다 체력 회복)은 타이머를 사용, Tick보다 훨씬 가벼움
- 이벤트 기반 설계 (Delegates) : '문이 열렸나?' 를 매 프레임 감시하는것이 아닌 문이 열리는 순간에만 이벤트를 발생시켜 반응하게 만듬
- 인터페이스 (Interface) : 특정 상황에서만 호출되는 함수를 인터페이스로 저장해두면, 불필요한 Tick 없이 필요한 시점에만 통신할 수 있음
- Tick Interval : 꼭 필요한 Tick이라도 PrimaryActorTick.TickInterval = 0.1f; 와 같이 설정해 실행 빈도를 낮출 수 있음

## 언리얼 인사이트

언리얼 인사이트는 게임의 성능 병목을 찾는 종합 진단 도구임.

- 핵심 기능

1. Timing Insights : CPU 스레드별로 어떤 함수가 얼마나 오래 걸리는지 타임라인으로 보여줌
2. Memory Insights : 메모리 할당과 해제가 어디서 빈번한지, 메모리 누수가 있는지 실시간으로 추적
3. Asset Loading : 게임 로딩 중에 어떤 에셋이 병목을 일으키는지 확인할 수 있음

## C++과 블루프린트

C++은 컴파일 시점에 CPU가 직접 읽을 수 있는 기계어로 번역. 반면 블루프린트는 실행 중에 가상머신(BPVM)이 바이트코드를 한 줄씩 읽어 해석하는 '인터프리터' 방식을 거치기 때문에 오버헤드가 발생함
