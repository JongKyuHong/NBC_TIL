# EDAssetManager만들기

AssetManager를 상속받는 커스텀 AssetManager를 만들고자 하였다. 공식문서와 라이라를 참고해서 만들었다.
EDAssetManger는 기존 AssetManager를 확장해서 만들것이고, DefaultEngine.ini에 AssetManagerClassName을 추가하여 기존 엔진단에 있는 AssetManager가 아니고 커스텀된 AssetManger를 사용하게 된다.

## 알아야하는 개념

Primary Asset : Asset Manager에 등록된 에셋이라고 보면된다.(FPrimaryAssetId로 식별가능)
Second Asset : Primary Asset이 참조하는 모든 에셋, AssetManager가 Primary Asset로드 시 자동으로 함께 로드된다.
Asset Bundle : UPROPERTY에 meta=(AssetBundles="BundleName")으로 그룹화한 Secondary Asset 목록, "UI"번들에는 아이콘/섬네일만, "InGame" 번들에는 메시/애니메이션 등록
FStreamableHandle : 로드된 에셋을 메모리에 고정시키는 핸들, TSharedPtr이 소멸되면 에셋이 GC대상이되므로 멤버변수로 유지 필요

# EDAssetManager 만들면서

**동기 비동기 선택기준**
동기 : 로딩 화면 중, 또는 아주 작은 에셋에 사용, 게임 스레드를 블로킹 -> 프레임 드랍/히치 발생

## StartInitialLoading

게임 플레이 시작전 가장 이른 초기화 시점

```
엔진 시작
    ↓
UEngine::Init()
    ↓
UAssetManager 생성 (DefaultEngine.ini 설정 기반)
    ↓
StartInitialLoading() 자동 호출  ← 여기
    ↓
게임 플레이 시작
```

**Super::StartInitialLoading**

부모 클래스(AssetManager)의 함수로, 내부에서 작업 수행

```
Super::StartInitialLoading()
    ↓
1. DefaultEngine.ini / Project Settings 읽기
   (Primary Asset Types to Scan 항목)
    ↓
2. 등록된 경로를 디스크에서 탐색
   (/Game/Data/Items 안에 있는 에셋들 스캔)
    ↓
3. 찾은 에셋들을 내부 테이블에 등록
   (FPrimaryAssetId → FSoftObjectPath 매핑 구성)
    ↓
4. 완료 → 이제 GetPrimaryAssetPath() 등 사용 가능
```

## 템플릿 함수

컴파일러가 템플릿을 인스턴스화 하려면 컴파일 타임에 전체 정의(구현부)를 볼 수 있어야 함 그래서 헤더파일에 템플릿 함수를 작성

**C++ 컴파일 과정** : Separate Compilation Model(분리 컴파일 모델)을 사용한다. 즉, `.cpp`파일 하나하나가 독립적으로 컴파일되어 각 `.obj`파일이 되고, 마지막에 링커가 이 오브젝트 파일을 하나로 합치는 구조
일반 함수의 경우 이 방식이 잘 동작, main.cpp가 add.cpp의 선언만 보고 컴파일을 마친 뒤 링크 단계에서 실제 구현을 찾아서 연결하면 되므로

**템플릿이 특수한 이유**
템플릿은 함수나 클래스가 아니다. `코드를 찍어내는 청사진`임, 실제 코드는 특정 타입이 주어졌을때 컴파일 타임에 비로소 생성, 인스턴스화는 링크 타임이 아닌 컴파일 타임에 일어나서 문제가 생긴다.
컴파일러가 main.cpp를 컴파일할 때 add<int> 인스턴스를 생성하려고 한다. 근데 이 시점에 컴파일러가 보고 있는건 add.h뿐이라 구현부가 없다.
반면 add.cpp를 컴파일할때는 구현부가 있지만, add<int>를 누가 호출하는지(어떤 타입이 필요한지) 전혀 모름, 두 정보가 같은 컴파일 단위에 존재하지 않으므로
결과적으로 링크 단계에서 에러가 발생한다.

헤더에 두면 왜 해결? : 헤더파일은 #include를 통해 전처리 단계에서 해당 .cpp에 그대로 복사 그래서 main.cpp가 #include "add.h"하는순간 컴파일러는 헤더를 볼 수 있음 그래서 구현부 + 호출타입을 볼 수 있다.

### 왜 동기로드만 템플릿함수로??

동기 함수는 함수가 return될때 이미 메모리에 에셋이 올라와있어서 캐스팅해서 바로 전달해 줄 수 있음
비동기 함수는 함수가 return될때 메모리에 올라와 있는지 아닌지를 모름 아직 로드중일수도 그래서 당장 AssetType\*를 줄수가없어서 대신에 나중에 완료될것을 추적하는 FStreamableHandle을 반환, 비동기는 나중에 콜백안에서 GetPrimaryAsset같은 함수 부를때 캐스팅됨 그래서 GetPrimaryAsset도 템플릿임

## 스레드 구조

UE5에서 에셋 로딩과 관련된 스레드는 크게 2가지가 있다.

**게임 스레드** : 게임 로직, 틱, 렌더링 명령, 여기가 막히면 게임 프리징/히칭
**비동기 로딩 스레드** : 백그라운드에서 디스크 I/O 전담, 파일읽기, 패키지 파싱, 메모리 적재

```
LoadSynchronous() 호출
        ↓
RequestSyncLoad() 호출
        ↓
① RequestAsyncLoad() 호출  ← "비동기 I/O 요청"이 이 부분
  (Async Loading Thread에게 "이거 읽어줘" 요청을 던짐)
        ↓
② Request->WaitUntilComplete() 호출  ← "블로킹 대기"가 이 부분
  (게임 스레드가 Async Loading Thread 작업 완료할 때까지 멈춰서 기다림)
        ↓
③ 로드 완료 → UObject* 반환
```

현대 OS의 디스크 I/O시스템 자체가 비동기 I/O요청이 훨씬 효율적이라고 한다.
내부적으로 비동기 I/O 요청 -> 완료 대기 조합이 단순 동기보다 빠르다.

## 메모리의 특징과 종류

**RAM의 용량과 성능 : 책상의 크기가 속도를 결정**
CPU의 성능이 아무리 뛰어나도 RAM용량이 부족하면 전체 시스템은 병목현상에 빠지게 된다.

**용량이 크면 왜 유리할까?**
잦은 로딩 방지 : RAM용량이 적으면 실행할 프로그램을 보조기억장치에서 가져오는 일이 잦아져 실행 시간이 길어짐
동시 실행 능력 : 용량이 충분하면 많은 데이터를 미리 RAM에 올려둘 수 있어, 여러 프로그램을 동시에 빠르고 쾌적하게 실행할 수 있음

**용량 증설의 한계**

RAM용량이 커지면 일정 수준까지는 속도가 향상되지만 필요이상으로 커진다고 해서 비례하여 계속 빨라지지는 않음

**DRAM / SRAM**

DRAM : Dynamic Ram, 시간이 지나면 데이터가 동적으로 사라짐, 저렴하고 집적도가 높아 대용량화에 유리함, 주기억장치(일반적인 RAM)
SRAM : Static Ram, 전원 공급중에는 데이터가 변하지 않음, 매우 빠르지만 비싸고 집적도가 낮음, 캐시메모리

### 언리얼 엔진 Insight 에셋 관리와 메모리 최적화

**레벨 스트리밍과 RAM점유**

언리얼에서 레벨을 로드하는 과정은 보조기억장치(.uasset)의 데이터를 DRAM(RAM)으로 복사하는 과정
문제 : 에셋이 너무 많아 RAM용량을 초과하면 하드웨어는 보조기억장치를 메모리처럼 쓰는 '페이징'을 히작하고, 이때 프레임 드랍이 발생
해결 : 월드 파티션 기능을 통해 현재 플레이어에게 필요한 데이터만 RAM에 올리도록 설계해야함

**텍스처 스트리밍**
모든 고해상도 텍스처를 한꺼번에 RAM에 올리면 금세 용량이 바닥남
언리얼은 플레이어와의 거리에 따라 텍스처의 정밀도를 조절하여 RAM사용량을 유동적으로 관리, 하드웨어의 휘발성 공간을 효율적으로 쓰는 기술

**캐시 적중률과 SRAM**

코드 최적화시에 TArray를 강조하는 이유가 메모리에 연속적으로 배치되기 때문임
CPU는 DRAM에서 데이터를 가져올때 주변 데이터까지 한꺼번에 SRAM(캐시)로 담아옴
데이터가 뭉쳐있으면 캐시에서 바로 찾아 쓸 확률이 높아져 연산 속도가 비약적으로 상승

## 메모리 주소 공간과 보호 기법

**물리주소 vs 논리주소 : 하드웨어와 소프트웨어의 다른 시선**

1. 물리주소 : 메모리 하드웨어가 사용하는 주소, 정보가 실제로 저장된 하드웨어상의 고유한 절대 주소를 의미
2. 논리주소 : CPU와 실행중인 프로그램이 사용하는 주소, 각 프로그램에 독립적으로 부여된 0번지부터 시작하는 상대적인 주소 공간, 프로그램마다 같은 논리주소를 가질 수 있으며, CPU는 이 논리 주소를 해석하고 연산

**주소변환의 마법사 MMU(Memory Management Unit)**

CPU가 사용하는 논리 주소와 메모리가 사용하는 물리 주소 체계가 다르기 때문에, 둘 사이를 이어주는 실시간 번역기가 필요, 이 역할을 담당하는 하드웨어가 MMU

변환원리 : 베이스 레지스터
MMU는 CPU가 발생시킨 논리주소에 베이스 레지스터 값을 더하여 물리주소로 변환
베이스 레지스터 : 실행중인 프로그램이 적재된 가장 작은 물리 주소(시작점)을 저장
논리 주소 : 프로그램의 시작점으로부터 떨어진 거리(오프셋) 역할을 한다.

**메모리 보호 기법 한계레지스터**

여러 프로그램이 메모리에 적재되었을때 특정 프로그램이 자신의 영역에서 벗어나 다른 프로그램의 데이터를 조작하거나 삭제하면 시스템 전체에 치명적인 오류가 발생, 이를 방지하기 위한 장치가 한계 레지스터

정의 : 실행중인 프로그램이 가질 수 있는 논리 주소의 최대 크기를 저장
안전 범위 : 프로그램의 물리 주소 범위는 `베이스 레지스터 값` 이상, `베이스 레지스터 + 한계 레지스터 값` 미만이 됨

보호 과정
CPU가 메모리에 접근하기전 접근하려는 논리주소가 한계 레지스터 값보다 작은지 항상 검사
만약 논리주소가 한계 레지스터보다 크다면 이는 다른 프로그램의 영역에 침범하려는 위험한 시도로 간주 인터럽트가 발생하여 해당 프로그램을 즉시 중단

**언리얼 엔진 Insight : Access Violation의 정체**

잘못된 포인터 접근 (Memory Access Violation)
C++에서 잘못된 메모리 주소를 참조하거나 유효하지 않은 포인터를 사용할 때 발생하는 크래시는 사실 하드웨어가 시스템을 보호하고 있다는 증거
하드웨어 레벨 : CPU가 요청한 논리 주소가 MMU 검사 과정에서 **한계 레지스터** 범위를 넘었음을 감지한 것입니다.
결과 : 운영체제는 해당 프로세스에 예외를 던지고 실행을 강제로 종료하여 다른 에셋 데이터나 엔진 코드가 오염되는 것을 막습니다.

**정리 : 독립된 실행 공간 확보**

논리 주소의 독립성 : 모든 프로그램은 자신만의 0번지를 사용하므로 다른 프로그램의 물리적 위치를 신경쓰지 않아도 됨
MMU의 실시간 변환 : 베이스 레지스터를 활용해 논리 주소를 실제 물리 주소로 1초에 수억번씩 변환하여 상호작용
한계 레지스터의 안전망 : 프로그램이 할당된 영역을 벗어나지 못하도록 물리적으로 격리하여 시스템의 안전성 보장

언리얼 프로젝트가 커질수록 수많은 DLL과 플러그인이 메모리에 복잡하게 적재되는데
이들이 서로 엉키지 않고 독립적인 공간에서 최상의 성능을 낼 수 있는 비결은, CPU 내부에서 MMU가 베이스 레지스터와 한계 레지스터를 통해 쉼 없이 주소를 체크해주기 때문이다.

# 알고리즘

## 레드-블랙 트리

자가 균형 이진 탐색트리로서 데이터 삽입, 삭제 시 트리의 높이를 일정 수준으로 유지하여 최악의 경우에도 탐색,삽입,삭제에 O(log N)의 시간복잡도를 보장하는 강력한 자료구조
std::map, std::set 등이 내부적으로 레드-블랙 트리를 사용한다.

### 규칙

1. 모든 노드는 레드 혹은 블랙 노드
2. 루트는 항상 블랙
3. 모든 리프(NIL)는 항상 블랙 (실제 데이터가 없는 끝노드)
4. 레드 노드의 자식은 반드시 블랙 (레드가 연속 2번 나올 수 없음)
5. 임의 노드에서 리프노드까지 가는 모든 경로에는 동일한 개수의 블랙 노드가 있다. (Black - Height)가 같다.

균형 맞추는 법

1. 회전 : 노드들의 위치를 좌/우로 바꾸어 트리 높이 조절
2. 색상 변경 : 부모나 삼촌 노드의 색을 바꿔 4번을 해결

## 에이스타

가장 비용이 적은 노드를 빠르게 뽑기위해 내부적으로 이진탐색트리(레드블랙트리) 혹은 우선순위큐(힙)을 사용
다익스트라의 확장판, 현재까지 온거리 + 목표까지 남은 예상거리를 더해 가장 유망한곳부터 탐색하는 최단경로 알고리즘

f(n) = g(n) + h(n)

### 동작 과정

1. open list(우선순위큐 or 레드블랙트리)에 탐색 예정인 노드를 담음 f(n)이 작은 순으로
2. closed list : 이미 방문이 끝난 노드들을 담음
3. open list에서 f(n)이 가장 작은 노드를 꺼냄, 목표지점이면 종료, 아니면 주변 이웃노드 확인
4. 이웃노드가 더 저렴한 경로로 갱신될 수 있다면 g(n), h(n)을 계산해 open list에 넣음
