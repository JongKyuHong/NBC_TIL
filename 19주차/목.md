# 라이라 프로젝트 분석

## ULyraAssetManager

UAssetManager를 게임 전용으로 확장한 싱글톤 에셋 관리 클래스
에셋을 직접 들고 있지않고 경로만 들고있다가 필요시에 로드하는 구조, 메모리를 효율적으로 관리하는게 목적

```c++
struct FLyraBundles
{
	static const FName Equipped;
};
```

번들은 에셋을 묶어서 그룹 단위로 로드/언로드하는 언리얼의 개념
Equipped는 "현재 장착된" 에셋들의 번들 이름 태그로, AssetManager->LoadPrimaryAssets() 호출 시 어떤 번들을 로드할지 지정할 때 사용

```c++
UCLASS(Config = Game)
```

Config = Game으로 지정해줌으로써 DefaultGame.ini에서 이 클래스의 UPROPERTY(Config) 변수들을 읽어 올 수 있다.

```c++
static ULyraAssetManager& Get();
```

싱글톤 접근자, static으로 선언해서 전역인스턴스를 반환, 참조를 직접 반환하므로 nullptr 체크 X

```c++
template<typename AssetType>
static AssetType* GetAsset(const TSoftObjectPtr<AssetType>& AssetPointer, bool bKeepInMemory = true);

template<typename AssetType>
static TSubclassOf<AssetType> GetSubclass(const TSoftClassPtr<AssetType>& AssetPointer, bool bKeepInMemory = true);
```

GetAsset은 로드된 에셋 객체의 포인터를 반환하는 함수, bKeepInMemory가 true면 LoadedAsset에 등록해 GC가 처리하지 못하게 함
GetSubclass도 비슷하게 동작하는데 클래스자체를 로드할때 사용

```c++
static void DumpLoadedAssets();
```

메모리에 로드된 전체 에셋 목록을 출력해주는 디버깅 함수

```c++
const ULyraGameData& GetGameData();
```

게임 전역 데이터를 반환, 없다면 동기 로드 후 반환

```c++
const ULyraPawnData* GetDefaultPawnData() const;
```

PlayerState에 PawnData가 없을 때 쓸 기본 폰 데이터 반환

```c++
template <typename GameDataClass>
const GameDataClass& GetOrLoadTypedGameData(const TSoftObjectPtr<GameDataClass>& DataPath)
{
    if (TObjectPtr<UPrimaryDataAsset> const * pResult = GameDataMap.Find(GameDataClass::StaticClass()))
    {
        return *CastChecked<GameDataClass>(*pResult);
    }

    // Does a blocking load if needed
    return *CastChecked<const GameDataClass>(LoadGameDataOfClass(GameDataClass::StaticClass(), DataPath, GameDataClass::StaticClass()->GetFName()));
}
```

GameDataMap에서 해당 클래스 타입의 키로 검색해서 이미 로드된적이 있으면 캐시된것을 반환
CastChecked는 타입이 맞지않으면 크래시, 반드시 올바른 타입이어야함을 보장
캐시-온-디멘드(Cache on Demand)패턴이라고 한다. 처음 요청시 로드, 이후에는 GameDataMap에서 즉시 반환

```c++
static UObject* SynchronousLoadAsset(const FSoftObjectPath& AssetPath);
```

Soft 경로 받아서 즉시 메모리에 로드하고 (동기) UObject\* 반환

```c++
void AddLoadedAsset(const UObject* Asset);
```

- 스레드를 안전하게 LoadedAsset에 추가, TSet구조라서 중복없음

```c++
virtual void StartInitialLoading() override;
```

엔진 초기화 초반에 호출, DoAllStartupJobs()로 StartupJobs 목록을 순차 실행

```c++
UPrimaryDataAsset* LoadGameDataOfClass(
    TSubclassOf<UPrimaryDataAsset> DataClass,
    const TSoftObjectPtr<UPrimaryDataAsset>& DataClassPath,
    FPrimaryAssetType PrimaryAssetType);
```

GetOrLoadTypedGameData에서 내부적으로 호출되는 실제 블로킹 로드 실행 함수
PrimaryAssetType : 에셋 매니저가 어떤 카테고리로 추적할지

### 궁금한거

1. UPROPERTY(Config) 변수는 무조건 ini파일에등록해야됨 일반 UPROPERTY()는 에디터에서 직접 할당하거나 코드에서 동적으로 할당
2. 로딩화면에서는 잠깐 멈춰도 사용자가 인지하기 힘듬 LoadGameDataOfClass가 동기적으로 호출 SynchronousLoadAsset()을 호출해서, 대신 이 함수는 절대 런타임에서 호출하면 안됨(히치발생 : 일시적으로 뚝 끉기는 현상)

# CPU 성능

## 클럭

CPU의 심장 박동, 컴퓨터의 모든 부품은 '클럭신호'라는 일정한 리듬에 맞춰 일사분란하게 움직임
클럭속도가 높아진다 = CPU가 명령어 사이클(인출-해독-실행)을 더 빠르게 반복한다. 이는 곧 게임의 연산속도와 직결

## 코어와 멀티 코어

클럭속도가 요리사 한명의 기초체력을 높이는거라면, 코어를 늘리는것은 주방에 요리사를 더 고용하는것과 같다.
코어 : CPU내부에서 실제로 명령어를 실행하는 독립적인 하드웨어 부품, 현대의 CPU는 내부에 여러개의 코어를 포함하고 있음

멀티코어 프로세서 : 여러개의 코어를 포함한 CPU를 말함, 8코어 CPU는 동시에 8개의 서로다른 명령을 처리할 수 있는 일꾼이 있는 셈

성능의 비례성 : 코어수가 늘어난다고해서 연산속도가 코어 수에 비례하여 증가하지는 않음

## 스레드

'실행 흐름의 단위' 하드웨어 레벨과 소프트웨어 레벨에서의 스레드는 의미가 다름

**하드웨어적 스레드 (논리 프로세서)** : 하나의 코어가 동시에 처리하는 명령어의 단위,
멀티스레드 CPU : 하나의 코어안에 레지스터 세트(PC, 스택포인터 등) 을 여러개 두어 마치 코어가 여러개인것처럼 보이게 만드는 기술
논리 프로세서 : 8코어 16스레드 CPU라면 실제 부품(코어)는 8개지만 프로그램 입장에서는 한 번에 16개의 ㅁ여령어를 처리하는 부품이 있는것처럼 보임

**소프트웨어적 스레드** : 하나의 프로그램 안에서 독립적으로 실행되는 작업 단위
멀티태스킹 : 워드 프로세서에서 입력내용출력 스레드, 맞춤법검사 스레드, 자동저장 스레드가 동시에 실행되는것과 같다.
유연성 : 한번에 하나씩 처리하는 1코어 1스레드 CPU도 소프트웨어적 스레드는 수십개를 실행할 수 있다.

하드웨어 스레드는 요리사가 가진 손의개수, 요리사(코어)는 한명이지만 두손을(2스레드)를 쓰면 더 효율적으로 일할 수 있음
소프트웨어 스레드는 요리사가 해야 할 레시피 목록, 요리사가 한명뿐이라도 할일 목록이 10개라면 아주 빠르게 번갈아 가며 처리해 마치 동시에 하는것처럼 보임

## 언리얼 엔진 Insight : 병렬 처리와 성능 최적화

언리얼엔진은 현대 CPU의 멀티코어/멀티스레드 구조를 극한까지 활용하도록 설계된 거대한 시스템

### 게임 스레드와 렌더 스레드의 분리

언리얼 엔진은 CPU의 여러 코어를 동시에 활용하기 위해 주요 작업을 나눔

- Game Thread : 게임 로직, 캐릭터 이동, AI판단 등 처리
- Render Thread : 화면에 그릴 데이터를 준비하여 GPU로 보냄
- Worker Threads : 나나이트 처리, 에셋 비동기 로딩등을 병렬로 처리

무거운 로직은 워커 스레드로 분산 시키고, 코어마다 처리할 명령어를 적절히 분배하는것이 언리얼 최적화의 핵심

# 배열

기억할건 메모리는 언제나 1차원 그래서 1차원 배열은 당연히 메모리에 빈틈없이 줄서있고 다차원배열들도 1자로 한줄로 저장됨
언리얼에서는 TArray를 써야 GC가 추적가능, 또 직렬화라는 개념 게임의 저장이나 네트워크 전송을 할떄 배열 내용을 아주 쉽게 파일로 바꾸거나 보낼수 있도록 설계되어있음, 작동원리는 vector랑 유사

# 알고리즘

## 세그먼트 트리

특정 구간의 합, 최솟값, 최대값을 효율적으로 구하고, 중간에 배열의 값이 변할때도 빠르게 대응하기 위한 자료구조, 일반적인 배열로 구간합을 구하면 O(N)지만 세그먼트 트리를 이용하면 데이터 업데이트와 구간 쿼리를 모두 O(log N)으로 해결가능

### 구조

완전 이진 트리 구조
리프 노드 : 실제 데이터
부모 노드 : 자식 노드들의 최소나 최대값 저장

### 동작

1. 루트 노드는 전체 구간 정보 가짐
2. 왼쪽, 오른쪽 자식을 구간을 반씩 나눠 저장
3. 원하는 구간의 합을 구할때 트리 타고 내려가서 해당 구간에 완전히 포함되는 값만 더해서 반환
4. 배열의 특정 인덱스 값이 바뀌면 해당 리프노드부터 루트 노드까지 경로상에 있는 모든 부모 노드의 값을 갱신(트리 높이가 log N이므로 빠르게 가능)
