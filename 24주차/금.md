# 면접 질문

## std::list의 삽입 시간 복잡도

- 내가 알기로는 링크드 리스트라서 맨끝 노드의 rear를 새로삽입하는 값으로 하고 새로들어가는 값의 front를 기존 맨뒤에 달기만 하면되어서 O(1)
- 끝에 삽입은 O(1)인데 중간 삽입은 포인터 연결만 바꾸면 되니 삽입 자체는 O(1)이지만 삽입 위치를 찾는 탐색이 O(n)임

## std::vector vs std::list 내부 구조 및 장단점

- std::list는 링크드 리스트
- std::vector는 배열형태
- 링크드 리스트는 메모리에서 연속되어 있지 않고 벡터는 연속되어있음
- 벡터의 삽입/삭제는 O(n)이고 list는 O(1)
- 벡터의 끝 삽입은 O(1), list는 삽입 자체는 O(1) (삽입위치 찾는 게 O(n)인거)

## 게임 루프에서 캐시미스 문제 & 해결 구조

- 캐시미스가 발생하면 램까지 다녀와야 해서 굉장히 성능저하가 일어난다.
- 발생하는 핵심 이유는 std::list처럼 메모리가 여기저기 흩어진 구조 때문
- 게임루프에서 캐시미스를 막는 가장 효과적인 방법은

1. 메모리를 연속적으로 배치 -> std::vector 나 배열사용
2. Hot/Cold 데이터 분리 -> 자주 접근하는 데이터끼리 모아두기
3. Data-Oriented Design(DOD) -> 언리얼에서도 권장하는 구조

실제로 같은 탐색 O(n)임에도 불구하고 std::vector나 std::list보다 2배이상 빠른이유가 캐시 지역성 때문

## CPU 캐시미스 반복시 성능 영향 설명

- CPU의 캐시데이터는 램보다 훨씬 빠르기때문에 최대한 캐시데이터에 남아있어야 빠른데
- 캐시 미스가 발생하면 더 느린 램까지 다녀와야 하고 버스를 통해서 데이터를 옮기는 자체도 시간이기 때문에 느림
- L1 -> L2 -> L3 -> RAM순서로 계층이 있고 캐시 미스 패널티는 L1히트 대비 RAM접근이 100배 넘게 느림, L2미스, L3미스 단계도 있음을 알아두자

- 컨테이너
  데이터를 저장, 관리하는 구조체(자료구조)들의 집합

- 반복자
  일종의 포인터처럼 컨테이너의 원소를 순회하는데 사용

- 알고리즘
  sort, find, count 등 여러가지 유용한 함수들이 제공
  이미 최적화된 코드라 직접 구현보다 유리한 경우가 ㅏ

## 힌트스테이지

```c++
// https://school.programmers.co.kr/learn/courses/30/lessons/468377

#include <string>
#include <vector>
#include <climits>
#include <algorithm>
#include <map>

using namespace std;

int answer = 0;
int N = 0;

void dfs(int depth, vector<vector<int>>& cost, vector<vector<int>>& hint, int total, map<int, int> bundle){
    if (depth == N) {
        answer = min(answer, total);
        return;
    }

    if (total >= answer) return;

    // bundle[depth]가 없더라도 0으로 생성되는데 그래도 상관없다고 생각
    int bundleCount = bundle[depth];
    if (bundleCount >= cost[depth].size()){
        bundleCount = cost[depth].size()-1;
    }
    int tmpCost = cost[depth][bundleCount];

    // 힌트 번들 구매 안할때
    dfs(depth+1, cost, hint, total + tmpCost, bundle);

    if (depth < N-1){
        // 구매하는 버전
        int hintCost = hint[depth][0];
        for (int i = 1; i < hint[depth].size(); ++i){
            // zero_index
            bundle[hint[depth][i]-1]++;
        }
        dfs(depth+1, cost, hint, total + tmpCost + hintCost, bundle);
    }
}

int solution(vector<vector<int>> cost, vector<vector<int>> hint) {
    answer = INT_MAX;
    N = cost.size();
    // cost에 힌트권 사용수에 따른 해결 비용
    // hint는 각 스테이지에서 구매 가능한 힌트 번들의 비용
    // dfs
    map<int, int> bundle;
    dfs(0, cost, hint, 0, bundle);

    return answer;
}
```

조금 짜증나는 문제였다. 공개된 테스트케이스에서 hint배열의 길이를 넘어가는 케이스가 있었는데 이를 확인하는데 오래걸려서 IDE를 써서 디버깅해버렸다. (물론 IDE로 어떤 성과가 나지는 않았음)
그냥 아래처럼 방어적으로 코드를 달아서 넘어가는경우 최대치의 할인율을 받도록 조정해서 해결하였다. 문제를 아주 상세하게 읽어야겠다. 그리고 테스크케이스도 분석하는 습관을 가져야 할 거 같다.
